# Аналитика: дыра в трекинге конверсий Яндекс.Метрики

**Дата**: 2026-04-29
**Автор**: Analytics Agent
**Приоритет**: HIGH
**Аудитория**: PO + CTO

---

## 1. Executive Summary

За 15 дней работы платформы (запуск 2026-04-14) Яндекс.Директ привёл 126 платных визитов на страницы аренды и PS Park. Метрика зафиксировала **1 конверсию** по цели `office_inquiry_submit` и **1** по `office_inquiry_success` — обе пришли не с рекламных кампаний, а с главной страницы через кэшированный трафик. По воронке «платный Директ → /rental → заявка» зафиксировано **0 конверсий**. При этом PO подтверждает: заявки в базе данных есть.

Это означает, что трекинг не работает, а не воронка. Мы тратим деньги на Директ и не можем посчитать CPL, оптимизировать кампании или принять решение об увеличении бюджета — работаем вслепую. Причина — клиентский `reachGoal()` блокируется защитами браузера/AdBlock до того, как завершился HTTP-запрос на сервер. Решение — server-side трекинг через Measurement Protocol Метрики.

## 3. Гипотезы причин

### Гипотеза 1 (основная): `reachGoal` блокируется до завершения fetch

**Файл**: `src/components/public/rental/inquiry-form.tsx`, строки 41 и 59

```
handleSubmit → reachGoal("office_inquiry_submit")   // строка 41 — вызов ПЕРЕД await fetch
             → await fetch("/api/rental/inquiries") // строка 43
             → if (data.success) reachGoal("office_inquiry_success")  // строка 59
```

Проблема двойная:
- `reachGoal("office_inquiry_submit")` вызывается на строке 41 **до** отправки запроса и **до** подтверждения успеха — если пользователь с AdBlock или мобильным браузером (iOS Safari, Samsung Browser) нажал кнопку, Метрика получает сигнал через `window.ym()`. AdBlock Plus, uBlock Origin, Brave и iOS ITP блокируют `mc.yandex.ru` — запрос к счётчику уходит в никуда.
- `reachGoal("office_inquiry_success")` на строке 59 вызывается после `await`, но в том же render-цикле, без `sendBeacon`. Если страница начала выгрузку или браузер ограничивает исходящие запросы — событие теряется.

Доля платного трафика (Директ) выше среди технически грамотной аудитории и мобильных пользователей — именно эта аудитория чаще использует блокировщики.

### Гипотеза 2: Конверсии идут в обход формы

**Подтверждено данными Метрики**: за 30 дней зафиксировано:
- 6 кликов по номеру телефона
- 7 переходов в мессенджер (WhatsApp/Telegram с Авито)

Эти лиды не попадают ни в `reachGoal`, ни в API — они звонят или пишут напрямую. На `/rental` нет обработчика `reachGoal('phone_click')` и `reachGoal('telegram_click')`, хотя цели объявлены в `src/lib/metrika.ts`.

**Файл**: `src/components/public/rental/rental-page-content.tsx` — в компоненте нет ни одного вызова `reachGoal`. Ссылки на телефон и мессенджеры, если они есть на странице, не трекируются.

### Гипотеза 3: Отсутствие server-side трекинга

**Файл**: `src/app/api/rental/inquiries/route.ts`

Функция `POST` создаёт запись в БД (`createInquiry`) и возвращает ответ — без каких-либо вызовов Measurement Protocol Метрики. Если клиент не отправил `reachGoal` (см. гипотезу 1), событие теряется полностью. Server-side трекинг — единственный способ гарантированно зафиксировать конверсию независимо от состояния браузера.

Та же проблема структурно присутствует в:
- `src/app/api/gazebos/bookings/route.ts` (цели `gazebo_booking_submit/success`)
- `src/app/api/ps-park/bookings/route.ts` (цели `pspark_booking_submit/success`)
- `src/app/api/cafe/orders/route.ts` (цель `cafe_order_submit`)

### Гипотеза 4: Мёртвая цель `office_inquiry_start`

**Файл**: `src/lib/metrika.ts`, строка 13

Цель `office_inquiry_start` объявлена в типе `YmGoal` и зарегистрирована в счётчике Метрики (следует проверить), но нигде в коде не вызывается — `grep` по всей `src/` даёт 0 результатов. Это:
- засоряет счётчик в интерфейсе Метрики
- создаёт ложное впечатление, что верхняя часть воронки трекируется

## 4. Влияние на бизнес

| Последствие | Описание |
|-------------|----------|
| CPL неизмерим | Не знаем, сколько стоит один лид из Директа. Нельзя сравнить кампании между собой |
| Оптимизация невозможна | Яндекс не получает сигнал о конверсиях → автостратегии («оплата за конверсии», «максимум конверсий») работают вслепую или вообще не работают |
| Решения об увеличении бюджета не подкреплены данными | Увеличить/снизить бюджет нельзя обоснованно — неизвестно, окупается ли текущий |
| Рекламные объявления не оптимизируются | A/B-тест объявлений требует данных о конверсиях, которых нет |
| Масштаб проблемы нарастает | Доля Директа выросла с 29% до 42% за последнюю неделю — дыра становится дороже |

**Приоритет**: HIGH. Каждый день без фикса — деньги на рекламу тратятся без возможности измерить ROI.

## 5. Техническое задание для CTO

### 5.1 Server-side трекинг через Measurement Protocol (приоритет 1)

**Что делать**: после успешного `createInquiry()` в `POST /api/rental/inquiries` отправлять событие в Метрику через Measurement Protocol — асинхронно, без блокировки ответа клиенту.

**Как получить `client_id`**: Метрика хранит идентификатор браузера в cookie `_ym_uid`. Клиент должен передавать его в теле запроса (новое опциональное поле `ymClientId`). Форма читает cookie и добавляет в JSON при отправке.

**Endpoint Метрики**: `https://mc.yandex.ru/watch/{COUNTER_ID}` с параметрами:
- `wmode=2` — режим отправки события
- `page-url` — URL страницы `/rental`
- `browser-info=ar%3A1%3Aen%3A1%3Ags%3A{goal_id}` — достижение цели (формат зависит от версии API, уточнить в документации Метрики Measurement Protocol)

Альтернатива: использовать официальный [Measurement Protocol Метрики](https://yandex.ru/support/metrica/objects/send-data.html) (POST-запрос на `https://mc.yandex.ru/watch/{counter_id}`).

**Файлы к изменению**:
- `src/app/api/rental/inquiries/route.ts` — добавить `ymClientId` из тела запроса, вызвать server-side hit после `createInquiry`
- `src/components/public/rental/inquiry-form.tsx` — добавить чтение cookie `_ym_uid` и передачу в тело запроса
- `src/lib/metrika.ts` — добавить server-side функцию `reachGoalServer(goal, clientId)` с `fetch` на Метрику

**То же самое** применить к (после `/rental` как пилота):
- `src/app/api/gazebos/bookings/route.ts`
- `src/app/api/ps-park/bookings/route.ts`
- `src/app/api/cafe/orders/route.ts`

### 5.2 Исправить race condition в форме (приоритет 2)

**Текущий код** (`inquiry-form.tsx:41`):
```typescript
reachGoal("office_inquiry_submit");  // строка 41 — до fetch, без гарантии
try {
  const res = await fetch("/api/rental/inquiries", { ... });
  if (data.success) {
    reachGoal("office_inquiry_success");  // строка 59 — может не успеть
```

**Что исправить**:

1. Убрать `reachGoal("office_inquiry_submit")` со строки 41 — этот вызов происходит до подтверждения сервером и дублирует server-side событие.
2. Если клиентский `reachGoal("office_inquiry_success")` оставить как резервный, завернуть в `navigator.sendBeacon` для гарантированной отправки при выгрузке страницы:

```typescript
// Резервный клиентский трекинг — только если server-side не настроен
if (typeof navigator !== "undefined" && navigator.sendBeacon) {
  // sendBeacon гарантирует отправку даже при закрытии страницы
}
reachGoal("office_inquiry_success");
```

После внедрения server-side трекинга (п. 5.1) клиентские `reachGoal` для submit/success можно полностью убрать — они станут дублями.

### 5.3 Добавить трекинг phone_click и telegram_click на /rental (приоритет 2)

**Проблема**: `src/components/public/rental/rental-page-content.tsx` не содержит ни одного вызова `reachGoal`. Цели `phone_click` и `telegram_click` объявлены в `src/lib/metrika.ts`, но на странице аренды не подключены.

**Что сделать**: найти в `rental-page-content.tsx` (или в родительском `page.tsx`) элементы с номером телефона и ссылками на мессенджеры, добавить обработчики:

Это позволит учитывать лиды, которые не заполняют форму, а сразу звонят или пишут.

### 5.5 Acceptance criteria

После деплоя изменений из п. 5.1–5.3 провести проверку через 7 дней:

- Число `office_inquiry_success` в Метрике (за 7 дней после деплоя) совпадает с числом записей в таблице `RentalInquiry` за тот же период, отклонение не более ±5%.

## Приложение: SQL для проверки после фикса

Запрос для сверки числа заявок в БД за период (запускать в admin-режиме, без PII):

```sql
-- Число заявок на аренду за период (без персональных данных)
SELECT
  DATE_TRUNC('day', created_at) AS day,
  COUNT(*) AS inquiries_count,
  COUNT(DISTINCT id) AS unique_inquiries
FROM rental_inquiry
WHERE created_at >= NOW() - INTERVAL '7 days'
GROUP BY 1
ORDER BY 1;
```

Результат этого запроса сравнивать с отчётом Метрики по цели `office_inquiry_success` за тот же период.
