# ADR: Пасхалки и мемы — тональный слой платформы

**Дата:** 2026-04-12

## Статус
Предложено

---

## Контекст

Платформа функционально завершена по Phase 0–4. Все модули работают, API покрыт, бизнес-логика реализована. При этом пользовательский опыт остаётся нейтрально-корпоративным: пустые состояния — заглушки вида «Нет данных», страницы ошибок — технические, тосты — шаблонные.

Аудитория PS Park и беседок — это геймеры и офисные сотрудники 18–35 лет, которые привыкли к Discord, Telegram, мемной культуре. Менеджеры видят интерфейс ежедневно. Суперадмин является техническим архитектором.

Задача: добавить эмоциональный слой, не затрагивая функционал и не нагружая кодовую базу новой инфраструктурой.

**Ключевое ограничение:** кафе, аренда офисов и финансовые разделы остаются деловыми. Пасхалки ограничены B2C-модулями (PS Park, беседки), страницами ошибок и элементами интерфейса в adminке, которые находятся вне финансового/CRM контекста.

---

## Варианты

### Вариант A: CMS для мемов (Prisma + Admin UI)
Хранить тексты пасхалок в БД, редактировать через интерфейс.

Минусы: избыточная инфраструктура для текстовых строк, которые не требуют редактирования в рантайме. Нарушает принцип «Config as Code» для контента с нулевой бизнес-ценностью в виде записей БД.

### Вариант B: Захардкодить строки прямо в каждом компоненте
Вставить тексты inline без централизованного файла.

Минусы: дублирование, сложно менять тон, нет единой точки контроля.

### Вариант C: Централизованный файл `src/lib/easter-eggs.ts` + inline-правки компонентов (выбрано)
Создать один файл с пулами фраз, типами и хелпером выбора случайной фразы. Компоненты импортируют нужный пул. Пустые состояния и тосты меняются минимальными правками.

Плюсы:
- Единая точка для правки тональности
- Нет новой инфраструктуры, нет изменений в БД
- Легко тестировать (статические массивы)
- Легко откатить: удалить файл, вернуть исходные строки

---

## Решение

Реализовать **Вариант C**.

### Архитектура

#### 1. Новый файл: `src/lib/easter-eggs.ts`

Содержит:
- Типизированные пулы строк (`TOAST_SUCCESS_MESSAGES`, `TOAST_CANCEL_MESSAGES`, `EMPTY_STATE_MESSAGES`, `SIDEBAR_TOOLTIP`)
- Pure-функцию `pickRandom<T>(arr: T[]): T` — берёт случайный элемент из массива
- Никаких импортов из Prisma, Redis, Next.js — чистый утилита-файл

Пример структуры:
```typescript
export const TOAST_SUCCESS_BOOKING: string[] = [
  "Готово! Стол забронирован. Не опаздывай.",
  "Есть! Бронь принята. Удачи в FIFA.",
  "Место за тобой. Время отрываться.",
  "Бронь оформлена. Контроллер ждёт.",
];

export const EMPTY_GAZEBOS_ADMIN: string = "Сегодня никто не бронировал. Может, все на шашлыках без нас?";
```

Вариативные тосты (4–8 вариантов) используют `pickRandom()` в момент вызова `showToast(...)`. Клиентский компонент вызывает функцию один раз при событии — случайность возникает в рантайме браузера, что корректно для `"use client"` компонентов. Серверные компоненты (пустые состояния в `page.tsx`) используют статические строки напрямую без randomness, чтобы не нарушать Server Component ограничения.

#### 2. Пасхалка в сайдбаре (tooltip при hover)

Сайдбар — `"use client"` компонент (`src/components/admin/sidebar.tsx`). Блок логотипа «Деловой Парк» в нижней части (`<span className="text-xs text-zinc-400 font-medium">Деловой Парк</span>`) оборачивается в `<div title="...">` или реализуется через CSS `group/tooltip` паттерн с Tailwind:

```tsx
<div className="relative group/logo flex items-center gap-2 px-1">
  <video ... />
  <span className="text-xs text-zinc-400 font-medium">Деловой Парк</span>
  <span className="absolute bottom-full left-0 mb-1 whitespace-nowrap rounded-lg bg-zinc-900 border border-zinc-700 px-3 py-1.5 text-xs text-zinc-300 opacity-0 group-hover/logo:opacity-100 transition-opacity pointer-events-none">
    Работает на кофе и дедлайнах ☕
  </span>
</div>
```

Никакого JS-состояния не требуется — чистый CSS hover через Tailwind `group`.

#### 3. Вариативность тостов в booking-flow

`booking-flow.tsx` (беседки) уже использует `showToast(message, type)`. Единственное изменение: вместо `showToast("Бронирование создано!", "success")` вызвать `showToast(pickRandom(TOAST_SUCCESS_BOOKING), "success")`.

Аналогично для cancel-сценариев в компонентах `booking-actions.tsx` (PS Park, беседки) — при успешной отмене показывать `pickRandom(TOAST_CANCEL_BOOKING)`.

`dark-availability-grid.tsx` (PS Park) не использует `Toast` компонент — там `bookingSuccess` state и inline-сообщение. Текст этого inline-сообщения меняется на живой вариант из пула (серверный рендер не задействован, это client-state).

#### 4. Страницы ошибок

- `src/app/not-found.tsx` — Server Component, статическая замена текста
- `src/app/error.tsx` — Client Component (`"use client"`), статическая замена текста
- `src/app/admin/forbidden/page.tsx` — Server Component, статическая замена текста

#### 5. Пустые состояния

Все целевые файлы — Server Components с inline-текстами. Правка сводится к замене строки:
- Беседки (публичный список): `gazebo-list.tsx` — компонент, вызывается с `resources` пропсом
- PS Park (публичный список столов): `page.tsx` — inline в JSX
- Беседки (брони в adminке): `admin/gazebos/page.tsx`
- PS Park (история в adminке): `admin/ps-park/page.tsx`
- Кафе (заказы в adminке): `admin/cafe/page.tsx`
- Мониторинг (события): `admin/monitoring/page.tsx`
- Логи/аудит: компоненты `EventsTable.tsx` / `AuditTable.tsx` или напрямую в `admin/architect/logs/page.tsx`
- Карта системы (0 модулей): `admin/architect/page.tsx`

#### 6. Нужны ли изменения в Prisma schema?
Нет. Все изменения — строки в компонентах и один новый утилита-файл.

## Последствия

### Положительные
- Нет изменений в схеме БД, API, бизнес-логике
- Нет новых зависимостей
- `src/lib/easter-eggs.ts` — единственный файл, который нужно поддерживать для всей тональности
- Откат прост: удалить файл, восстановить строки
- Тесты для `pickRandom()` тривиальны (статический массив)

### Отрицательные / риски
- Субъективность тональности: некоторые фразы могут не понравиться части аудитории. Митигация: выбирать иронию без агрессии, не затрагивать финансовые/B2B разделы
- Серверные компоненты не могут использовать `Math.random()` стабильно при SSR (гидратация). Митигация: статические строки в server components, `pickRandom()` только в client components
- Менеджер, видящий интерфейс ежедневно, может устать от одних и тех же фраз. Митигация: пул 4–8 вариантов для тостов, ротация происходит на каждом событии

### Вне скопа (подтверждено)
- GIF, анимированные мемы — нет
- CMS/база данных для контента — нет
- Кафе, аренда офисов, финансы, CRM арендаторов — остаются нейтральными
