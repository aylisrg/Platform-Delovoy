# ADR: Управленка -- система управленческого учёта расходов

## Статус
Предложено

## Контекст

Бизнес-парк "Деловой" запущен 17 апреля 2026. С первого дня идут расходы: хостинг Timeweb, домен, GitHub, реклама в Яндекс.Директ, телефония. Владелец ведёт учёт вручную (или не ведёт). Нужна единая точка учёта расходов в админке, с автоматическим созданием записей по подпискам.

Полные бизнес-требования -- в PRD `docs/requirements/2026-04-18-cost-tracker-management-accounting-prd.md`.

Ключевые требования PO:
1. Два типа расходов: recurring-шаблоны (подписки) и разовые ручные записи.
2. Фоновый процесс ежедневно создаёт записи из recurring-шаблонов.
3. Фиксированные 4 категории расходов (enum).
4. Только SUPERADMIN имеет доступ.
5. Блок "Ожидается до конца месяца" на дашборде.
6. Auto-generated записи immutable (нельзя PATCH/DELETE).

Затрагиваемые модули: новый модуль `management`. Не затрагивает cafe, ps-park, gazebos, rental, inventory.

---

## Варианты

### Вариант A: Одна таблица Expense с полем isRecurring

Хранить и шаблоны подписок, и фактические записи расходов в одной таблице `Expense`. Поле `isRecurring: Boolean` различает шаблон от записи. Поле `parentId` связывает auto-generated запись с шаблоном.

- Плюсы: одна таблица, проще миграция, один CRUD.
- Минусы: нарушает разделение ответственности -- шаблон подписки и фактическая запись расхода имеют разные жизненные циклы и разные наборы полей (у шаблона есть `frequency`, `nextBillingDate`; у записи -- `date`, `isAutoGenerated`). Запросы усложняются фильтрами `WHERE isRecurring = false` повсюду. Soft delete шаблона не должен удалять его записи, но с одной таблицей это путаница. PO явно описал два отдельных объекта.

### Вариант B: Две таблицы -- RecurringExpense (шаблон) и Expense (запись) (выбран)

`RecurringExpense` -- шаблон подписки с полями `amount`, `frequency`, `nextBillingDate`, `startDate`.
`Expense` -- фактическая запись расхода с полями `amount`, `date`, `isAutoGenerated`, `recurringExpenseId` (nullable FK).

- Плюсы: чёткое разделение ответственности; шаблоны и записи имеют свои CRUD; UI делится на две вкладки; auto-generated записи связаны с шаблоном через FK; соответствует решениям PO.
- Минусы: две таблицы вместо одной, две миграции, два набора CRUD endpoints. Это приемлемый трейдофф -- сложность минимальна, а выгода от разделения значительна.

## Решение

Выбран **Вариант B**. Обоснование: PO явно описал два отдельных объекта с разными жизненными циклами. Шаблон -- это "что будет тратиться", запись -- "что уже потрачено". Смешение в одной таблице создаст путаницу на уровне запросов и UI.

## Схема данных

### Новые Prisma enums

```prisma
enum ExpenseCategory {
  IT_INFRASTRUCTURE   // IT-инфраструктура (хостинг, домен, SaaS)
  ADVERTISING         // Реклама (Яндекс.Директ, таргет)
  TELEPHONY           // Телефония и SMS
  OPERATIONS          // Операционные расходы
}

enum ExpenseFrequency {
  MONTHLY             // Ежемесячно
  QUARTERLY           // Ежеквартально
  YEARLY              // Ежегодно
}
```

### Модель RecurringExpense

```prisma
model RecurringExpense {
  id              String           @id @default(cuid())
  name            String                        // "Timeweb VPS", "Домен delovoy-park.ru"
  description     String?
  category        ExpenseCategory
  frequency       ExpenseFrequency
  amount          Decimal                       // 3500.00
  currency        String           @default("RUB")
  startDate       DateTime                      // Дата начала подписки
  nextBillingDate DateTime                      // Следующая дата списания (обновляется cron-процессом)
  isActive        Boolean          @default(true)
  deletedAt       DateTime?                     // Soft delete
  createdById     String                        // userId (SUPERADMIN) кто создал
  createdAt       DateTime         @default(now())
  updatedAt       DateTime         @updatedAt

  expenses        Expense[]

  @@index([isActive, nextBillingDate])
  @@index([category])
  @@index([deletedAt])
}
```

**Пояснение полей:**
- `nextBillingDate` -- ключевое поле для cron-процесса. Cron ежедневно проверяет: `WHERE isActive = true AND deletedAt IS NULL AND nextBillingDate <= NOW()`. После создания записи `Expense` поле `nextBillingDate` сдвигается на следующий период.
- `startDate` -- дата, с которой начинается подписка. Используется для первоначального расчёта `nextBillingDate`.
- `createdById` -- для аудита, кто создал шаблон. Не FK к User, чтобы не создавать circular dependency (только SUPERADMIN может создавать).
- `deletedAt` -- soft delete. Деактивированный шаблон перестаёт генерировать записи, но история остаётся.

### Модель Expense

```prisma
model Expense {
  id                  String          @id @default(cuid())
  name                String                    // "Timeweb VPS -- апрель 2026" или "Настройка рекламы Авито"
  description         String?
  category            ExpenseCategory
  amount              Decimal                   // Фактическая сумма расхода
  currency            String          @default("RUB")
  date                DateTime                  // Фактическая дата расхода (день списания)
  isAutoGenerated     Boolean         @default(false)
  recurringExpenseId  String?                   // FK на RecurringExpense (null = ручной расход)
  recurringExpense    RecurringExpense? @relation(fields: [recurringExpenseId], references: [id])
  deletedAt           DateTime?                 // Soft delete (только для ручных)
  createdById         String                    // userId или "system" для auto-generated
  createdAt           DateTime        @default(now())
  updatedAt           DateTime        @updatedAt

  @@index([category, date])
  @@index([recurringExpenseId])
  @@index([date])
  @@index([deletedAt])
  @@index([isAutoGenerated])
}
```

**Пояснение полей:**
- `isAutoGenerated = true` + `recurringExpenseId != null` -- запись создана cron-процессом из шаблона. Такие записи immutable: PATCH и DELETE возвращают 422.
- `isAutoGenerated = false` + `recurringExpenseId = null` -- ручной расход, полный CRUD.
- `date` -- фактическая дата расхода, не `createdAt`. Позволяет ввести расход за прошлый период.
- `createdById` -- для auto-generated это `"system"`, для ручных -- `session.user.id`.
- `deletedAt` -- soft delete. Для auto-generated записей `DELETE` запрещён (422). Для ручных -- проставляет `deletedAt`.
- `name` -- для auto-generated формируется автоматически: `"{recurringExpense.name} -- {monthName} {year}"`.

### Связь с User (отсутствие FK)

`createdById` в обеих моделях -- это `String`, а не FK к `User`. Причина: для auto-generated записей значение `"system"` -- это не userId. Добавление FK потребовало бы либо системного пользователя в таблице User, либо nullable FK. Проще хранить как строку и разрешать в UI при необходимости. AuditLog фиксирует все мутации с полноценным FK.

## API-контракт

### Общие правила

- Все endpoints под prefix `/api/management/`.
- Все endpoints: только `role === "SUPERADMIN"` (жёсткая проверка, не `hasModuleAccess`).
- Все ответы через `apiResponse()` / `apiError()`.
- Все входные данные через Zod-валидацию.
- Все мутации логируются в `AuditLog`.

### 11. GET /api/cron/process-recurring

Фоновый процесс: создание записей Expense из recurring-шаблонов.

**Аутентификация:** Bearer token (`process.env.CRON_SECRET`), как в существующих cron-endpoints (`/api/cron/no-show`, `/api/cron/inventory`).

**Не требует** сессии пользователя -- это системный процесс.

**Расписание:** Ежедневно, один раз в день (рекомендуется 00:05 по серверному времени). Настраивается в системном crontab или GitHub Actions.

**Логика:**
```
1. Проверить Bearer token === CRON_SECRET
2. SELECT * FROM RecurringExpense WHERE isActive = true AND deletedAt IS NULL AND nextBillingDate <= NOW()
3. Для каждого шаблона:
   a. Создать Expense:
      - name: "{recurringExpense.name} -- {monthName(nextBillingDate)} {year(nextBillingDate)}"
      - category: recurringExpense.category
      - amount: recurringExpense.amount
      - currency: recurringExpense.currency
      - date: recurringExpense.nextBillingDate
      - isAutoGenerated: true
      - recurringExpenseId: recurringExpense.id
      - createdById: "system"
   b. Обновить RecurringExpense.nextBillingDate на следующий период:
      - MONTHLY: +1 month
      - QUARTERLY: +3 months
      - YEARLY: +1 year
   c. Обернуть (a) и (b) в prisma.$transaction для атомарности
4. Залогировать SystemEvent (INFO) с результатами
```

**Response (200):**
```json
{
  "success": true,
  "data": {
    "processedAt": "2026-04-18T00:05:00.000Z",
    "created": 2,
    "skipped": 0,
    "errors": [],
    "details": [
      { "recurringId": "cl...", "expenseId": "cl...", "name": "Timeweb VPS -- апрель 2026" },
      { "recurringId": "cl...", "expenseId": "cl...", "name": "GitHub Team -- апрель 2026" }
    ]
  }
}
```

**Idempotency:** Если cron вызван дважды в один день, на второй вызов шаблон уже имеет `nextBillingDate` в будущем -- ничего не создастся. Это обеспечивает идемпотентность без дополнительного лока.

**Обработка ошибок:** Если создание записи для одного шаблона падает, остальные продолжают обрабатываться. Ошибки собираются в массив `errors` и логируются в SystemEvent (WARNING).

| `/api/cron/process-recurring` | GET | System | `Bearer CRON_SECRET` |

### Проверка доступа в route handler (паттерн)

```typescript
const session = await auth();
if (!session?.user?.id) return apiUnauthorized();
if (session.user.role !== "SUPERADMIN") return apiForbidden("Доступ только для администратора");
```

Не используется `hasModuleAccess()` и `requireAdminSection()` -- это жёсткое ограничение на роль, не на секцию. Причина: финансовые данные всего бизнеса должны быть доступны только владельцу. Даже MANAGER с AdminPermission на секцию "management" не получит доступ к API.

**Уточнение по UI:** Для отображения пункта в sidebar используется `AdminPermission.section = "management"`. Но это только UI-уровень -- на сервере проверяется `role === "SUPERADMIN"` напрямую. SUPERADMIN по определению имеет доступ ко всем секциям (hardcoded в `hasAdminSectionAccess`).

### Rate limiting

SUPERADMIN exempt от rate limiting (текущее поведение в проекте). Новых ограничений не требуется.

### Injection-защита

- `name`, `description` -- свободный текст, ограниченный Zod (max length). Хранится в Prisma (параметризованные запросы). React экранирует при отображении.
- `amount` -- числовой, валидируется Zod как `z.number().positive()`.
- `date`, `startDate`, `nextBillingDate` -- regex-валидация `YYYY-MM-DD`.
- Нет URL-полей, нет file paths -- SSRF невозможен.

## Фоновый процесс: Process Recurring

### Вариант реализации

Next.js API Route `/api/cron/process-recurring`, вызываемый системным crontab. Это соответствует существующему паттерну проекта (`/api/cron/inventory`, `/api/cron/no-show`, `/api/cron/notifications`).

### Crontab entry

```cron
5 0 * * * curl -s -H "Authorization: Bearer $CRON_SECRET" https://delovoy-park.ru/api/cron/process-recurring
```

### Расчёт nextBillingDate

```typescript
function advanceNextBillingDate(current: Date, frequency: ExpenseFrequency): Date {
  const next = new Date(current);
  switch (frequency) {
    case "MONTHLY":
      next.setMonth(next.getMonth() + 1);
      break;
    case "QUARTERLY":
      next.setMonth(next.getMonth() + 3);
      break;
    case "YEARLY":
      next.setFullYear(next.getFullYear() + 1);
      break;
  }
  return next;
}
```

### Формирование имени auto-generated Expense

```typescript
function formatAutoExpenseName(templateName: string, billingDate: Date): string {
  const monthNames = [
    "январь", "февраль", "март", "апрель", "май", "июнь",
    "июль", "август", "сентябрь", "октябрь", "ноябрь", "декабрь"
  ];
  const month = monthNames[billingDate.getMonth()];
  const year = billingDate.getFullYear();
  return `${templateName} -- ${month} ${year}`;
}
```

### Edge cases

1. **Cron пропущен на несколько дней.** Если `nextBillingDate` отстал на 3 дня, при следующем запуске cron создаст запись и сдвинет дату. Одна запись на цикл, не три -- потому что проверка `nextBillingDate <= NOW()` отрабатывает один раз, после чего дата уходит в будущее.

2. **Cron вызван дважды в минуту.** Транзакция `prisma.$transaction` гарантирует, что если запись уже создана и `nextBillingDate` сдвинут, второй вызов не найдёт этот шаблон.

3. **Шаблон деактивирован (`isActive = false`) или soft-deleted.** `WHERE isActive = true AND deletedAt IS NULL` -- такие шаблоны не обрабатываются.

4. **Сумма 0 рублей (seed).** Допускается -- seed создаёт шаблоны с `amount: 0` и предупреждением в описании. Cron создаст записи с суммой 0. Владелец обновит суммы, когда узнает реальные.

### Phase 5.3 (Дашборд владельца) -- совместимость

Модуль "Управленка" создаёт данные, которые Phase 5.3 (P&L дашборд) сможет агрегировать:
- Расходы по категориям: `SELECT SUM(amount) FROM Expense WHERE ... GROUP BY category`.
- Доходы: из `FinancialTransaction` (ps-park), `Booking.metadata.totalPrice` (gazebos), `RentalContract.monthlyRate` (rental).
- P&L = Доходы - Расходы.

Модель Expense спроектирована с учётом этой будущей агрегации: поле `date` (не `createdAt`) позволяет точную привязку расхода к периоду.
