# Context Log: User Management Unified

**RUN_ID**: `2026-04-17-user-management-unified`
**Дата**: 2026-04-17
**Задача**: Рефакторинг системы управления пользователями

## Описание от заказчика

1. **Ревью** -- проанализировать все разделы управления пользователями в админке (Пользователи, Клиенты, Telegram). Реестр юзеров разбросан.
2. **Улучшение логики входа** -- Telegram login не должен просить email. Оставить Telegram, Email, Яндекс. Добавить WhatsApp. Убрать VK и Google.
3. **Личный кабинет пользователя** -- в админке раздел настроек: все подключённые аккаунты (Telegram, Email, Яндекс), настройки уведомлений, возможность привязать/отвязать каналы.
4. **Единый раздел управления пользователями** -- всё про всех в одном окне. Функция мержа пользователей (для суперадминов) -- слить несколько аккаунтов в один.

## PO -- Ключевые решения

### 1. Структура единого реестра: вкладки, не новая страница

**Решение**: единый раздел `/admin/users` с тремя вкладками -- "Все", "Команда" (SUPERADMIN + MANAGER), "Клиенты" (USER с активностью).

**Почему**: оба существующих API (`/api/users` и `/api/admin/clients`) уже возвращают нужные ��анные. Создавать новую структуру с нуля не требуется -- нужно объединить два UI-компонента под одной навигационной точкой. Вкладки дают прозрачность без потери ни одного существующего действия.

**Альтернатива, отвергнутая**: одна таблица для всех с фильтром по роли. Отвергнута, потому что для команды и для клиентов нужны принципиально разные колонки (для команды -- роль, права, последний вход; для клиентов -- траты, активность по модулям). Одна таблица становится либо перегруженной, либо показывает ненужные колонки.

### 2. Раздел "Telegram" переезжает в "Мониторинг", а не исчезает

**Решение**: настройки Telegram-бота (chat ID, токен, карта уведомлений) переезжают во вкладку "Система" в разделе "Мониторинг". Раздел "Telegram" из сайдбара удаляется.

**Почему**: настройки бота -- это инфраструктура (токены, каналы алертов), а не управление людьми. Логически они относятся к мониторингу системы, а не к CRM пользователей. Суперадмин найдёт их там же, где health-check и системные события.

**Риск**: менеджеры, знающие путь `/admin/telegram`, не найдут раздел. Митигируется редиректом со старого URL.

### 3. Google и VK: убираем из auth.ts, но сначала проверяем данные

**Решение**: перед удалением провайдеров из кода Architect должен запросить у БД `SELECT COUNT(*) FROM "Account" WHERE provider IN ('google', 'vk')`. Если есть активные аккаунты -- уведомить владельцев (через email или вручную) и предложить привязать альтернативный канал до удаления.

**Почему жёсткое удаление без проверки -- не вариант**: поле `vkId` присутствует в модели User, VK отображается в ContactsCard как канал связи. Это значит, что система уже могла создать пользователей через VK. Тихое удаление заблокирует вход этим людям.

**Что убираем точно и без проверки**: кнопки на странице входа. UI-изменение безопасно -- провайдер в auth.ts продолжает принимать существующие сессии даже без кнопки на фронте.

### 4. Отвязка каналов: последний канал нельзя удалить

**Решение**: API `/api/profile/contacts/{channel}/detach` проверяет количество активных провайдеров пользователя перед выполнением операции. Если остаётся один -- возвращает ошибку с сообщением для пользователя.

**Почему**: без этой защиты пользователь может заблокировать себе вход, удалив единственный способ аутентификации. Особенно опасно для Telegram-пользователей без email -- после отвязки Telegram аккаунт станет недоступен навсегда.

**Ограничение**: "минимум один канал" -- это минимальная защита. В будущем можно ужесточить до "минимум один верифицированный email" для восстановления доступа, но это выходит за рамки текущей итерации.

### 5. Личный кабинет -- не "Мой профиль в админке", а улучшение существующего /dashboard

**Решение**: функционал профиля остаётся в `/dashboard` (публичный ЛК), а не переезжает в отдельный раздел в `/admin`. Раздел "Мой профиль" в админке -- только редирект или ссылка на `/dashboard`, но не отдельная страница с дублированием.

**Почему**: пользователи (роль USER) не имеют доступа к `/admin`. Если перенести профиль в админку, обычные клиенты потеряют к нему доступ. `/dashboard` -- единственная точка, доступная всем ролям.

**Что это означает для SUPERADMIN/MANAGER**: они могут редактировать свой профиль через `/dashboard` так же, как обычные клиенты. Это осознанное решение -- профиль у всех работает одинаково.

### 6. Функция слияния -- только для SUPERADMIN, всегда с preview

**Решение**: кнопка "Объединить" видна только для SUPERADMIN. Preview обязателен перед подтверждением -- нельзя запустить merge без явного показа конфликтов. Подтверждение через текст "ОБЪЕДИНИТЬ" сохраняется.

**Почему**: слияние аккаунтов необратимо -- удаляется secondary-аккаунт со всеми сессиями. Право на это действие должно быть у минимально возможного круга лиц. Preview уже реализован в `/api/admin/clients/merge/preview` -- использование его обязательно, не опционально.

### 7. Telegram-виджет: убираем data-request-access=write

**Решение**: параметр `data-request-access=write` убирается из Telegram Login Widget. Вход через Telegram работает без запроса телефонного номера.

**Почему заказчик прав**: `write` требует номер телефона у Telegram-пользователя. Это создаёт дополнительный барьер при входе -- пользователь видит запрос "Разрешить @Bot доступ к вашему номеру?" и часто отказывается. Для аутентификации на сайте телефон не нужен -- достаточно Telegram ID, имени и фото.

**Потенциальная потеря**: бот не сможет автоматически получить номер телефона при первом входе нового пользователя. Это приемлемо -- пользователь может добавить телефон вручную через "Мои контакты".

## Architect -- Ключевые решения

ADR: `docs/architecture/2026-04-17-user-management-unified-adr.md`

### 1. Нет изменений в Prisma-схеме

Все необходимые модели и поля уже существуют: `User.telegramId`, `User.vkId`, `Account` (для OAuth провайдеров), `TelegramLinkToken`. Поле `vkId` и enum-значение `VK` в `NotificationChannel` остаются в схеме для backward compatibility -- удаление в следующей итерации (add/hide now, remove later).

### 2. Один новый API endpoint -- отвязка каналов

`DELETE /api/profile/contacts/{channel}/detach` -- единственный новый бэкенд-endpoint. Защита от удаления последнего канала: сервис считает все активные методы входа (telegramId, email, phone, Account записи, passwordHash) и блокирует при count <= 1. AuditLog обязателен. Rate limit: 10 req/min на userId.

### 3. Порядок реализации: US-3 -> US-4 -> US-1 -> US-2

- US-3 (страница входа) -- zero-dependency, критический фикс для production-запуска, минимальный объем
- US-4 (профиль ЛК) -- создает новый endpoint, независим от админки
- US-1 (единый реестр) -- основной рефакторинг, трогает sidebar, permissions, 3 страницы
- US-2 (merge из реестра) -- зависит от US-1, но переиспользует существующий MergeDialog

### 4. Провайдеры Google/VK остаются в auth.ts

Кнопки убираются только из UI страницы входа. Провайдеры остаются в `auth.ts` для обратной совместимости -- существующие сессии и Account записи продолжают работать. Перед деплоем обязательна SQL-проверка: `SELECT COUNT(*) FROM "Account" WHERE provider IN ('google', 'vk')`.

### 5. Единый реестр = композиция существующих API, не новый API

Вкладка "Команда" использует `/api/users` (с добавлением пагинации), вкладка "Клиенты" использует `/api/admin/clients`. Вкладка "Все" использует расширенный `/api/users` (с authProviders). Поиск работает через API каждой вкладки. Никаких новых "unified" API-эндпоинтов.

### 6. Секции clients и telegram удаляются из ADMIN_SECTIONS

Необходима миграция прав:
```sql
UPDATE "AdminPermission" SET section = 'users' WHERE section = 'clients';
UPDATE "AdminPermission" SET section = 'monitoring' WHERE section = 'telegram';
```
Выполнить перед деплоем.

### 7. Yandex в профиле -- через таблицу Account

Наличие записи `Account` с `provider = "yandex"` означает привязанный Yandex. В `getProfile()` добавляется join на `accounts`. Привязка Yandex -- через стандартный OAuth flow NextAuth (`signIn("yandex", { callbackUrl: "/dashboard" })`). Отвязка -- удаление записи из `Account`.

### 8. Привязка Telegram -- существующий механизм TelegramLinkToken

Уже реализован: `POST /api/profile/telegram/generate-link` создает `TelegramLinkToken` с TTL. В ContactsCard нужно добавить UI: кнопка "Привязать Telegram" -> показать deep-link `t.me/BotName?start=link_TOKEN`.
