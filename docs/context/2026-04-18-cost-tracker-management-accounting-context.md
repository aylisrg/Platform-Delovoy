# Pipeline Context: Cost Tracker / Управленка

**RUN_ID:** `2026-04-18-cost-tracker-management-accounting`
**Дата:** 2026-04-18
**Задача:** Система учёта расходов (управленческий учёт) с автоматическим трекингом

## Описание от пользователя

Нужен "кост трекер" — автоматическая система учёта расходов. Новый пункт меню "Управленка" в админке. Сюда входят все доходы и расходы, но сейчас делаем часть с расходами.

Категории расходов:
- Реклама (Яндекс.Директ, таргет и т.д.)
- IT-сервисы (хостинг, домен, API-сервисы)
- Телефония (звонилки)
- Прочие операционные расходы

Ключевое требование: автоматическая запись трат — когда тратятся деньги, расход сразу попадает в управленку.

IT-расходы (известные):
- Timeweb VPS хостинг
- Домен delovoy-park.ru
- GitHub (если платный)
- Telegram Bot API (бесплатный)
- Прочие SaaS

## PO — Ключевые решения

### 1. "Автоматическая запись" = recurring-шаблоны, не интеграции с API внешних сервисов

**Решение:** "Автоматически" в контексте задачи означает: настраиваешь recurring-расход один раз (Timeweb 3 500 ₽/месяц) — система сама создаёт запись расхода в нужную дату. Никаких интеграций с API Timeweb, Яндекс, GitHub нет.

**Почему:** Реализация интеграций с тремя и более внешними API (каждый со своей авторизацией, форматом, rate limiting) — это месяц разработки ради данных, которые владелец и так знает. Recurring-шаблон закрывает 90% боли: "забыл внести — пропущено". Разовые расходы вносятся вручную — их по определению немного.

**Альтернативы, отклонённые:** Импорт банковской выписки — нет банковского API, сложно. Скрапинг кабинетов сервисов — нестабильно, нарушает ToS.

---

### 2. Два вида расходов: Expense (запись) и RecurringExpense (шаблон)

**Решение:** Два отдельных сущности. `RecurringExpense` — шаблон подписки. `Expense` — фактическая запись в журнале (ручная или auto-generated из шаблона). Связь: `Expense.recurringExpenseId` (nullable).

**Почему:** Если хранить всё в одной таблице с флагом `isRecurring`, нельзя отделить "шаблон на следующий месяц" от "запись за прошлый месяц". Нужны две разные операции: управление шаблонами (CRUD на RecurringExpense) и просмотр истории (READ на Expense + агрегаты). Это две разные вкладки в UI.

**Граница:** `Expense.isAutoGenerated = true` — запись создана системой по шаблону. Такую запись нельзя редактировать и удалять. Хочешь исправить — правь шаблон, ошибочная запись помечается (но не удаляется для целостности истории).

---

### 3. Фиксированные 4 категории, без пользовательских

**Решение:** IT-инфраструктура, Реклама, Телефония/SMS, Операционные расходы — жёсткий список в MVP.

**Почему:** На старте парка расходы хорошо укладываются в эти 4 ведра. Пользовательские категории — UI управления, валидация уникальности, миграции справочника, риск "300 категорий у одного пользователя". Это работа ради 5% кейсов. Если после запуска владелец скажет "мне нужна категория Юридические расходы" — добавим в следующей итерации.

---

### 4. Только SUPERADMIN — никакого MANAGER-доступа

**Решение:** Весь раздел `/admin/management` и все эндпоинты `/api/management/*` закрыты жёсткой проверкой `role === 'SUPERADMIN'`.

**Почему:** Управленка — это финансовые данные всего бизнеса: сколько тратит парк на рекламу, сколько стоит хостинг. Менеджер кафе или беседок не должен это видеть — ни из соображений конфиденциальности, ни из практических (им это не нужно для работы). Нет исключений: даже MANAGER, назначенный на модуль "management", доступ не получит — потому что такого назначения не будет в seed.

---

### 5. Расходы склада кафе (StockReceipt) в Управленку не переносятся

**Решение:** `StockReceipt` и `StockReceiptItem` остаются в модуле инвентаризации. В "Управленку" не интегрируются.

**Почему:** Закупки кафе — это операционный учёт склада с отдельной логикой (приёмка товара, партии, бракованное). Тащить их в управленческий учёт создаст две точки правды и запутает агрегаты. Если нужен общий P&L — Phase 5.3 (Дашборд владельца) агрегирует данные из обоих источников отдельно.

---

### 6. Приоритет: Should have (не Must have)

**Решение:** Модуль не блокирует работу бизнеса — бронирования идут, клиенты обслуживаются. "Must have" — слишком сильно для инструмента, без которого парк работает прямо сейчас. "Should have" — правильная категория: значительная ценность, делаем в ближайшей итерации, но не на день запуска.

**Почему не "Could have":** Расходы уже идут с 17 апреля 2026. Каждый месяц без трекера — это данные, которые придётся восстанавливать из почты. Чем раньше запустить — тем полнее история.

---

### 7. Ожидаемые расходы (прогноз до конца месяца) — в MVP

**Решение:** Блок "Ожидается до конца месяца" на дашборде — включаем в MVP, не в "Could have".

**Почему:** Это самый ценный инсайт из всего раздела. Знать, что ты уже потратил — хорошо. Знать, что ещё спишется в этом месяце — необходимо для планирования. Реализация тривиальная: выборка активных RecurringExpense с `nextBillingDate` между сегодня и концом месяца.

## Architect — Ключевые решения

### 1. Две таблицы: RecurringExpense и Expense

**Решение:** Вариант B из ADR -- отдельные таблицы для шаблонов подписок и записей расходов.

**Prisma-модели:**
- `RecurringExpense`: `id`, `name`, `description?`, `category` (enum), `frequency` (enum), `amount` (Decimal), `currency`, `startDate`, `nextBillingDate`, `isActive`, `deletedAt?`, `createdById`, timestamps.
- `Expense`: `id`, `name`, `description?`, `category` (enum), `amount` (Decimal), `currency`, `date`, `isAutoGenerated`, `recurringExpenseId?` (FK), `deletedAt?`, `createdById`, timestamps.

**Enums:** `ExpenseCategory` (IT_INFRASTRUCTURE, ADVERTISING, TELEPHONY, OPERATIONS), `ExpenseFrequency` (MONTHLY, QUARTERLY, YEARLY).

**Связь:** `Expense.recurringExpenseId` -> `RecurringExpense.id` (nullable). Auto-generated записи имеют `isAutoGenerated = true` и заполненный `recurringExpenseId`.

**Почему не FK на User:** `createdById` -- строка, не FK. Для auto-generated записей значение `"system"`. Создание системного пользователя в таблице User было бы overengineering для одного поля. AuditLog покрывает полный аудит с FK.

### 2. Фоновый процесс -- API route с Bearer token

**Решение:** `GET /api/cron/process-recurring` с аутентификацией через `Authorization: Bearer $CRON_SECRET`. Соответствует паттерну `/api/cron/no-show` и `/api/cron/inventory`.

**Расписание:** Ежедневно в 00:05 через системный crontab. Не Next.js middleware, не отдельный воркер -- API route, вызываемый curl.

**Idempotency:** После создания Expense из шаблона, `nextBillingDate` сдвигается в будущее. Повторный вызов не найдёт этот шаблон (WHERE nextBillingDate <= NOW()). Без дополнительных локов.

**Error isolation:** Каждый шаблон обрабатывается независимо. Ошибка в одном не блокирует остальные.

### 3. SUPERADMIN-only без hasModuleAccess

**Решение:** Жёсткая проверка `session.user.role === "SUPERADMIN"` в каждом route handler. Не используется `requireAdminSection()` и `hasModuleAccess()`.

**Почему:** Финансовые данные всего бизнеса. MANAGER с AdminPermission на "management" не получит API-доступ. UI-секция "management" в sidebar используется только для навигации -- SUPERADMIN получает её автоматически через `hasAdminSectionAccess()` (hardcoded return true для SUPERADMIN).

### 4. Immutable auto-generated записи

**Решение:** PATCH и DELETE на `Expense` с `isAutoGenerated === true` возвращают HTTP 422 с кодом `IMMUTABLE_AUTO_EXPENSE`. Проверка в service-слое, не в route handler.

**Почему:** Целостность журнала. Если шаблон ошибочный -- правь шаблон, а не конкретную запись. История расходов должна быть неизменяемой для аудита.

### 5. Summary endpoint с прогнозом

**Решение:** `GET /api/management/summary` возвращает одним запросом: totalSpent, byCategory, upcoming (recurring с nextBillingDate в остатке периода), projectedTotal.

**Формула:** `projectedTotal = totalSpent + SUM(upcoming recurring amounts where nextBillingDate between today and end of period)`.

**Почему один endpoint:** UI дашборда делает один fetch. Нет необходимости в отдельных endpoints для каждой метрики -- объём данных минимален (4 категории, ~10 recurring шаблонов).

### 6. Seed с нулевыми суммами

**Решение:** Seed создаёт 3 RecurringExpense (Timeweb, домен, GitHub) с `amount: 0` и предупреждением в description.

**Почему:** Точные суммы неизвестны на момент seed. Шаблоны с нулевой суммой -- это placeholder: система начинает работать, а владелец обновляет суммы когда узнает реальные. Cron создаст записи с amount=0, что лучше, чем пропущенные записи.

### 7. Миграция additive-only

**Решение:** Prisma migration добавляет 2 enum и 2 таблицы. Не меняет и не удаляет существующие объекты. Полная backward compatibility.
