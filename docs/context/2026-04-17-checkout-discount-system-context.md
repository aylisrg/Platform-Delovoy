# Context Log: Checkout Discount System

**RUN_ID:** `2026-04-17-checkout-discount-system`
**Задача:** Система скидок при чекауте беседок и столов PlayStation. Менеджер может применить скидку, все скидки логируются.

---

## Требования от пользователя

- Менеджер при чекауте беседки или стола PlayStation может применить скидку
- Система должна быть удобной и безопасной
- Скидка должна быть точно залогирована (аудит)
- Модули: gazebos, ps-park

---

## PO -- Ключевые решения

### 1. Скидка -- только в момент чекаута (COMPLETED), не раньше

**Решение:** Скидка применяется исключительно при переводе бронирования в статус `COMPLETED`. Ни PENDING, ни CONFIRMED не затрагиваются.

**Почему:** Это единственная точка, где известна итоговая сумма (для PS Park -- фактические часы + товары). Изменять сумму "заранее" создаёт операционный хаос: клиент ещё не пришёл, менеджер поменялся, цена пересчиталась при продлении.

**Альтернативы, отклонённые:** Скидка при создании брони -- отклонена, так как итоговая сумма ещё не финальна (можно продлить, добавить товары). Скидка при CONFIRMED -- отклонена по той же причине.

---

### 2. Хранение скидки в JSONB, без новой таблицы

**Решение:** Поле `discount` добавляется в существующий `Booking.metadata` (JSONB). Новая миграция БД не нужна.

**Почему:** Скидка -- это атрибут одного конкретного события (чекаута), а не самостоятельная сущность, требующая запросов. Вся аналитика по скидкам будет идти через `AuditLog`, а не через JOIN к отдельной таблице.

**Риск:** При переходе к программе лояльности (Phase 5.1) нужно убедиться, что чтение `finalAmount` идёт из правильного поля. Зафиксировано в разделе рисков PRD.

---

### 3. Причина скидки -- обязательный выбор из справочника + free text для "Другое"

**Решение:** 5 фиксированных причин + "Другое" с обязательным текстом (минимум 5 символов). Причина обязательна при скидке > 0.

**Почему:** Свободный текст без структуры невозможно агрегировать. Выбор из списка даёт суперадмину группировку по типу скидок. "Другое" с пояснением закрывает исключения без потери данных.

**Альтернативы, отклонённые:** Только свободный текст -- отклонён (невозможно агрегировать). Причина необязательна -- отклонена (теряет смысл аудита).

---

### 4. Максимальный процент скидки -- в конфиге модуля, не в коде

**Решение:** `Module.config.maxDiscountPercent` (JSONB), по умолчанию 30%. Меняется суперадмином через Config GUI без деплоя.

**Почему:** Разные модули могут иметь разные лимиты. Беседки могут давать до 20%, а PS Park -- до 40% в рамках акции. Жёсткий лимит в коде сделал бы каждое изменение деплоем.

**Граница по умолчанию 30%:** Больше -- риск злоупотреблений (скидка 50% на беседку это -3000 руб. при среднем чеке 6000). Меньше -- менеджер не покрывает реальные ситуации (компенсация за серьёзные неудобства).

---

### 5. Скидка передаётся в существующий PATCH, без нового эндпоинта

**Решение:** Поля `discountPercent`, `discountReason`, `discountNote` добавляются в тело существующего `PATCH /api/{module}/bookings/:id`.

**Почему:** Скидка -- это атрибут операции "завершить бронирование", а не самостоятельная операция. Отдельный эндпоинт `POST /api/{module}/bookings/:id/discount` создал бы race condition (применить скидку до чекаута? после? в каком порядке?).

---

### 6. Аудит-лог -- единственный источник правды для отчётности по скидкам

**Решение:** Новый экран "Отчёт по скидкам" не создаётся. Суперадмин использует существующий журнал аудита с фильтром `action = "booking.discount_applied"`.

**Почему:** Запуск через 0 дней (17 апреля). Создавать новый дашборд некогда. Фильтр по action в уже готовом журнале аудита закрывает 80% потребности. Полноценный отчёт -- в Phase 5.3 (Дашборд владельца).

## Architect -- Ключевые решения

### 1. Вариант хранения: JSONB без миграции (Вариант B)

**Решение:** `Booking.metadata.discount` (JSONB) + `AuditLog` с `action = "booking.discount_applied"`. Новых таблиц и миграций нет.

**Почему:** PO явно выбрал этот вариант. Скидка -- атрибут чекаута, не самостоятельная сущность. Аналитика покрывается `AuditLog`. Ноль миграций = минимальный risk на day-1.

**Альтернатива отклонена:** Отдельная таблица `Discount` с FK -- оверхед для текущих потребностей.

---

### 2. Аудит-лог скидки внутри транзакции

**Решение:** `tx.auditLog.create()` вызывается внутри `prisma.$transaction` вместе с `booking.update` (и `financialTransaction.create` для PS Park).

**Почему:** Существующий хелпер `logAudit()` из `src/lib/logger.ts` не принимает транзакционный клиент (`tx`). Если создавать аудит-лог вне транзакции, при сбое между booking.update и auditLog.create данные рассинхронизируются: скидка применена, но не залогирована. Это именно то, что мы боремся.

**Компромисс:** `logAudit()` в route handler для `booking.status_change` остаётся вне транзакции (текущее поведение, не меняем). `booking.discount_applied` -- новый action, создаётся внутри транзакции.

---

### 3. Обновление `metadata.totalPrice` для gazebos

**Решение:** При применении скидки `metadata.totalPrice` перезаписывается на `finalAmount`.

**Почему:** Аналитика gazebos (`getAnalytics()`, строка 799) читает `metadata.totalPrice` для подсчёта выручки. Если оставить оригинальную сумму, выручка в отчётах будет завышена. Оригинал сохраняется в `metadata.discount.originalAmount`.

---

### 4. PS Park: `FinancialTransaction.totalAmount` = `finalAmount`

**Решение:** `FinancialTransaction` создаётся с `totalAmount = finalAmount` (после скидки). Оригинал сохраняется в `metadata` транзакции (`originalAmount`, `discountPercent`).

**Почему:** PS Park аналитика считает выручку из `FinancialTransaction.totalAmount`. Если записать оригинальную сумму, отчёты будут некорректны. `cashAmount + cardAmount` должны балансироваться к `finalAmount`.

---

### 5. Zod-схема -- отдельный файл `src/modules/booking/validation.ts`

**Решение:** Общая схема `checkoutDiscountSchema` в `src/modules/booking/validation.ts`, используется обоими модулями.

**Почему:** Скидка -- cross-module функциональность (gazebos + ps-park). Дублировать Zod-схему в обоих модулях -- нарушение DRY. Файл `src/modules/booking/` уже содержит общие типы (`types.ts`) и утилиты (state-machine, pricing, cancellation).

---

### 6. Справочник причин -- TypeScript const + labels

**Решение:** `DISCOUNT_REASONS` как `const` массив в `src/modules/booking/discount.ts`, `DISCOUNT_REASON_LABELS` для UI.

**Почему:** Энум Prisma потребовал бы миграции БД. TypeScript union type + Zod enum -- достаточно. Если потребуется расширить справочник -- это изменение кода, не миграция.

---

### 7. `getMaxDiscountPercent()` -- один запрос к БД

**Решение:** Отдельная функция `getMaxDiscountPercent(moduleSlug)` в `src/modules/booking/discount.ts`. Читает `Module.config.maxDiscountPercent`, fallback на `DEFAULT_MAX_DISCOUNT_PERCENT = 30`.

**Почему:** Лимит проверяется в сервисе, не в route handler. Если лимит поменялся -- следующий запрос читает актуальное значение. Кэширование не нужно (менеджеры делают чекаут редко, 1 дополнительный запрос несущественен).

---

### 8. Формула: `Math.round()` вместо `Math.floor()`

**Решение:** `discountAmount = Math.round(originalAmount * discountPercent / 100)`.

**Почему:** PRD указывает "округление до рубля". `Math.round()` ближе к стандартному бухгалтерскому округлению. `Math.floor()` всегда в пользу клиента (меньше скидка). `Math.ceil()` всегда в пользу клиента (больше скидка). `Math.round()` -- нейтрален.

**`finalAmount = originalAmount - discountAmount`** (не наоборот) -- чтобы `discountAmount + finalAmount === originalAmount` всегда, без ошибок округления.
