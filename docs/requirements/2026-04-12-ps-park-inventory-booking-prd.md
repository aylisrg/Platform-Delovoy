# PRD: PS Park — бронирование столов, инвентарь и продажи при бронировании

**Дата:** 2026-04-12
**Автор:** Product Owner, Platform Delovoy
**Статус:** Draft
**Версия:** 1.0

---

## Проблема

### Текущее состояние

PS Park в системе существует как модуль с API для бронирования, но реализован неполностью:

1. **Бронирование столов** работает технически, однако цена стола может быть изменена любым менеджером — нет ограничения только для Superadmin. При этом беседки работают по той же модели (Resource + Booking), что создаёт возможность переиспользования паттерна.

2. **Инвентарь отсутствует как концепция.** Менеджеры учитывают приход товара (снеки, напитки) в Excel или на бумаге. Нет ни контроля остатков, ни истории движения товара.

3. **Продажи товара при бронировании не предусмотрены.** Клиент, бронирующий стол в PS Park или беседку, не может заказать напитки и снеки заранее — это делается устно при встрече, и доход от таких продаж нигде не фиксируется.

### Последствия для бизнеса

- Потеря выручки от сопутствующих продаж (напитки, снеки при бронированиях)
- Некорректные данные об остатках — перезаказ или дефицит товара
- Отсутствие аудита изменений цен — менеджер может случайно изменить стоимость стола
- Разрыв между тем, что клиент заказал, и тем, что менеджер видит в системе

---

## Решение

### Блок 1 — Защита цены ресурса (PS Park)

Ограничить изменение поля `pricePerHour` у `Resource` с `moduleSlug = "ps-park"` только для роли `SUPERADMIN`. Менеджеры видят цену, но не могут её менять.

Начальная цена: 300 руб/час за стол. Стартовое количество столов: 5.

### Блок 2 — Модуль инвентаря (новый)

Новый доменный модуль `inventory` со своими моделями БД, сервисом, типами и валидацией. Инвентарь — общий для всего парка (не привязан к конкретному модулю бронирования). Товары из инвентаря доступны для добавления к любому бронированию.

Роли:
- **MANAGER** — только добавляет приход товара (новая партия пришла)
- **SUPERADMIN** — полное управление: удаление, корректировка остатков, добавление/архивирование SKU

### Блок 3 — Продажи при бронировании

Расширение flow бронирования для PS Park и Газебо: на шаге подтверждения бронирования клиент (или менеджер от имени клиента) может добавить товары из инвентаря к заказу. Финальная сумма = стоимость бронирования + стоимость товаров.

Товар списывается с остатков при переходе бронирования в статус `CONFIRMED`.

## Целевая аудитория

| Роль | Задача в рамках этих фич |
|------|--------------------------|
| USER (клиент) | Добавить напитки/снеки при бронировании стола или беседки |
| MANAGER (менеджер PS Park / беседок) | Видеть заказанные товары по каждому бронированию; фиксировать приход нового товара |
| SUPERADMIN | Управлять SKU, остатками, ценами ресурсов; видеть аналитику по продажам |

## User Stories

### US-1: Защита цены стола в PS Park

- **Как** Superadmin
- **Я хочу** быть единственным, кто может изменять цену аренды стола в PS Park
- **Чтобы** менеджеры не меняли цены случайно или без согласования

**Acceptance Criteria:**
- [ ] AC-1.1: `PATCH /api/ps-park/:id` с полем `pricePerHour` возвращает `403 Forbidden` для роли `MANAGER`
- [ ] AC-1.2: `PATCH /api/ps-park/:id` с полем `pricePerHour` выполняется успешно для роли `SUPERADMIN`
- [ ] AC-1.3: Менеджер может обновлять другие поля стола (`name`, `description`, `isActive`) без ошибки
- [ ] AC-1.4: Изменение цены логируется в `AuditLog` с полями `before` и `after`
- [ ] AC-1.5: В UI (панель менеджера) поле цены отображается как read-only, без кнопки редактирования
- [ ] AC-1.6: В UI (панель Superadmin) поле цены редактируемо

---

### US-2: Просмотр каталога товаров инвентаря

- **Как** клиент, менеджер или Superadmin
- **Я хочу** видеть список доступных товаров с ценами и остатками
- **Чтобы** знать, что можно добавить к бронированию

**Acceptance Criteria:**
- [ ] AC-2.1: `GET /api/inventory` возвращает список активных SKU с полями: `id`, `name`, `category`, `price`, `unit`, `stockQuantity`
- [ ] AC-2.2: SKU с `isActive = false` не отображаются в публичном каталоге
- [ ] AC-2.3: SKU с `stockQuantity = 0` отображаются с флагом `outOfStock: true` (не скрываются)
- [ ] AC-2.4: Каталог доступен без авторизации (публичный endpoint)
- [ ] AC-2.5: Ответ кэшируется в Redis на 60 секунд

---

### US-3: Приход нового товара (менеджер)

- **Как** менеджер
- **Я хочу** зафиксировать поступление новой партии товара нажатием кнопки "Приехал новый товар"
- **Чтобы** остатки в системе соответствовали реальному наличию

**Acceptance Criteria:**
- [ ] AC-3.1: `POST /api/inventory/receive` принимает массив `[{ skuId, quantity, note? }]` и создаёт записи `InventoryTransaction` с типом `RECEIPT`
- [ ] AC-3.2: После успешного запроса `stockQuantity` каждого SKU увеличивается на указанное количество
- [ ] AC-3.3: Запрос доступен только для ролей `MANAGER` и `SUPERADMIN`
- [ ] AC-3.4: `quantity` должен быть положительным целым числом (≥ 1). Нулевое или отрицательное значение возвращает `400 Bad Request`
- [ ] AC-3.5: Каждая транзакция сохраняет `performedById` (кто зафиксировал приход)
- [ ] AC-3.6: В UI менеджера есть кнопка "Приехал новый товар", открывающая форму с выбором SKU и количества
- [ ] AC-3.7: После отправки формы менеджер видит подтверждение с обновлёнными остатками

---

### US-4: Корректировка и удаление (только Superadmin)

- **Как** Superadmin
- **Я хочу** иметь возможность скорректировать остаток или удалить ошибочную транзакцию прихода
- **Чтобы** исправить ошибки менеджера без потери истории

**Acceptance Criteria:**
- [ ] AC-4.1: `POST /api/inventory/adjust` принимает `{ skuId, quantity, reason }` и создаёт транзакцию типа `ADJUSTMENT` (quantity может быть отрицательным)
- [ ] AC-4.2: `DELETE /api/inventory/transactions/:id` — мягкое удаление транзакции (`isVoided = true`), остаток пересчитывается
- [ ] AC-4.3: Оба endpoint возвращают `403 Forbidden` для роли `MANAGER`
- [ ] AC-4.4: Каждая корректировка требует обязательного поля `reason` (min 10 символов)
- [ ] AC-4.5: Все корректировки и удаления логируются в `AuditLog`
- [ ] AC-4.6: История всех транзакций доступна через `GET /api/inventory/transactions?skuId=...` с пагинацией

---

### US-5: Управление SKU (каталогом товаров)

- **Как** Superadmin
- **Я хочу** добавлять, редактировать и архивировать позиции товаров
- **Чтобы** поддерживать актуальный каталог для продаж при бронировании

**Acceptance Criteria:**
- [ ] AC-5.1: `POST /api/inventory/sku` создаёт новую позицию с полями: `name`, `category`, `price`, `unit` (штука/упаковка/литр), `startingStock?`
- [ ] AC-5.2: `PATCH /api/inventory/sku/:id` обновляет поля SKU. Изменение `price` логируется в `AuditLog` с `before`/`after`
- [ ] AC-5.3: `DELETE /api/inventory/sku/:id` — мягкое удаление (`isActive = false`). SKU с привязанными транзакциями нельзя удалить полностью, только архивировать
- [ ] AC-5.4: Все три endpoint доступны только для `SUPERADMIN`
- [ ] AC-5.5: При создании SKU c `startingStock > 0` автоматически создаётся транзакция `INITIAL` с этим количеством

### US-6: Добавление товаров при бронировании стола PS Park

- **Как** клиент
- **Я хочу** добавить напитки и снеки к своему бронированию стола
- **Чтобы** всё было готово к моему приходу и я заплатил за всё сразу

**Acceptance Criteria:**
- [ ] AC-6.1: `POST /api/ps-park/book` принимает необязательное поле `items: [{ skuId, quantity }]`
- [ ] AC-6.2: При передаче `items` система проверяет наличие достаточного количества каждого SKU. При нехватке возвращает `400` с кодом `INVENTORY_INSUFFICIENT` и перечислением позиций
- [ ] AC-6.3: Итоговая сумма бронирования включает стоимость товаров: `totalAmount = hoursBooked * pricePerHour + sum(item.quantity * sku.price)`
- [ ] AC-6.4: `totalAmount` сохраняется в `metadata.totalAmount` бронирования
- [ ] AC-6.5: Товары НЕ списываются в момент создания бронирования (`PENDING`) — только при переходе в `CONFIRMED`
- [ ] AC-6.6: При переходе бронирования в `CONFIRMED` создаются транзакции `SALE` для каждого товара, `stockQuantity` уменьшается
- [ ] AC-6.7: При отмене бронирования (`CANCELLED`) в статусе `CONFIRMED` товары возвращаются на склад — создаются транзакции `RETURN`
- [ ] AC-6.8: Список заказанных товаров виден менеджеру в деталях бронирования
- [ ] AC-6.9: Список заказанных товаров виден клиенту в личном кабинете

---

### US-7: Добавление товаров при бронировании беседки

- **Как** клиент
- **Я хочу** добавить напитки и снеки к своему бронированию беседки
- **Чтобы** организовать отдых без лишних походов за покупками

**Acceptance Criteria:**
- [ ] AC-7.1: `POST /api/gazebos/book` принимает необязательное поле `items: [{ skuId, quantity }]` (тот же формат, что и в PS Park)
- [ ] AC-7.2: Логика проверки наличия, расчёта суммы и списания/возврата товаров идентична US-6 (AC-6.2 — AC-6.7)
- [ ] AC-7.3: `POST /api/gazebos/admin-book` (бронирование менеджером) также поддерживает `items`
- [ ] AC-7.4: Список заказанных товаров виден менеджеру беседок в деталях бронирования

---

### US-8: Просмотр аналитики инвентаря (Superadmin)

- **Как** Superadmin
- **Я хочу** видеть сводный отчёт по движению товаров и выручке от продаж
- **Чтобы** принимать решения о закупках и ценообразовании

**Acceptance Criteria:**
- [ ] AC-8.1: `GET /api/inventory/analytics?from=...&to=...` возвращает по каждому SKU: `totalSold`, `totalReturned`, `totalReceived`, `currentStock`, `revenue`
- [ ] AC-8.2: Общая выручка от продаж товаров при бронировании доступна в разбивке по модулям (`ps-park` / `gazebos`)
- [ ] AC-8.3: Позиции с `currentStock < lowStockThreshold` помечены флагом `lowStock: true`
- [ ] AC-8.4: `lowStockThreshold` настраивается через `PATCH /api/inventory/sku/:id` (поле `lowStockThreshold`, default: 5)
- [ ] AC-8.5: Endpoint доступен только для `SUPERADMIN`

## Схема данных (новые модели)

### InventorySku

```prisma
model InventorySku {
  id                String   @id @default(cuid())
  name              String
  category          String             // "Напитки", "Снеки", "Прочее"
  unit              String   @default("шт")
  price             Decimal
  stockQuantity     Int      @default(0)
  lowStockThreshold Int      @default(5)
  isActive          Boolean  @default(true)
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt

  transactions      InventoryTransaction[]
}
```

### InventoryTransaction

```prisma
model InventoryTransaction {
  id          String                  @id @default(cuid())
  skuId       String
  sku         InventorySku            @relation(fields: [skuId], references: [id])
  type        InventoryTransactionType
  quantity    Int                     // Положительное для RECEIPT/INITIAL/RETURN, отрицательное для SALE/ADJUSTMENT
  bookingId   String?                 // Связь с бронированием (для SALE / RETURN)
  moduleSlug  String?                 // "ps-park" / "gazebos" (для SALE / RETURN)
  performedById String
  note        String?
  isVoided    Boolean  @default(false)
  createdAt   DateTime @default(now())
}

enum InventoryTransactionType {
  INITIAL     // Начальный остаток при создании SKU
  RECEIPT     // Приход от менеджера ("Приехал новый товар")
  SALE        // Списание при подтверждении бронирования
  RETURN      // Возврат при отмене подтверждённого бронирования
  ADJUSTMENT  // Корректировка Superadmin (quantity может быть отрицательным)
}
```

## Зависимости и риски

### Зависимости

- Модуль `inventory` должен быть зарегистрирован в таблице `Module` со slug `"inventory"` до деплоя
- Prisma-миграция для новых моделей (`InventorySku`, `InventoryTransaction`) должна быть применена до деплоя
- Seed-скрипт должен добавить начальные SKU (минимум: Cola, Pepsi, вода, снек-набор)
- Транзакция списания при `CONFIRMED` должна выполняться атомарно с обновлением статуса бронирования (Prisma `$transaction`)

### Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Гонка остатков при одновременных бронированиях | Средняя | Высокое | Pessimistic lock или проверка остатков внутри `prisma.$transaction` |
| Менеджер не фиксирует приход вовремя — остатки занижены | Высокая | Среднее | UX кнопки "Приехал товар" вынесена на главный экран панели менеджера |
| Клиент добавляет товары, бронирование отклоняется — товары не списались, всё ок | Низкая | Низкое | Списание только при `CONFIRMED` — поведение уже задано в AC-6.5 |

## Вне скоупа

- Онлайн-оплата товаров (оплата наличными/картой на месте)
- Доставка товаров в офисы (это область модуля Кафе)
- Учёт алкоголя (требует отдельной лицензии и регуляции)
- Мобильное приложение для инвентаризации
- Интеграция с кассовым аппаратом / ФЗ-54
- Возможность менеджеру самостоятельно добавлять новые SKU (только Superadmin)
- Скидки и промокоды на товары при бронировании
