# PRD: Мобильная переработка админ-панели

**Дата**: 2026-04-16
**Автор**: PO (pipeline `2026-04-16-mobile-admin-redesign`)
**Модуль**: admin (cross-module UI — ps-park, gazebos, cafe, rental, users, monitoring и т.д.)

---

## Проблема

Админка (`/admin/*`) спроектирована только под десктоп. Боль — менеджер приходит к клиенту со смартфоном в руках и не может оперативно закрыть базовые сценарии (принять бронь Плей Парка, подтвердить ожидающую бронь, посмотреть активные сессии).

### Факты по текущему коду (ревью `src/app/admin/*` и `src/components/admin/*`)

**Вывод**: текущая админка **не юзабельна** на мобильном. Для сценария "быстро принять бронь у стойки" она хуже, чем позвонить и записать в блокнот.

---

## Решение

Mobile-first переработка админ-интерфейса для экранов < `lg` (1024px), **без изменения UX на `lg+`**. Ключевые идеи:

1. **Навигация** — сайдбар скрывается в выдвижной drawer (hamburger), плюс нижний bottom-tab с 4-5 самыми частыми разделами.
2. **Списки вместо таблиц** — все `<table>` в админке на мобильном превращаются в карточки.
3. **Кнопки действий ≥ 44×44px**, важные CTA занимают всю ширину.
4. **PS Park бронирование с телефона в 3 касания** — плитка ресурсов → слот времени → подтверждение. Модалка бронирования превращается в bottom sheet, полностью экранная, с "липкой" кнопкой снизу.
5. **Консистентные отступы** — `px-4` на мобильном, `px-8` на десктопе.

---

## Целевая аудитория

- **MANAGER** (основная) — менеджер Плей Парка, беседок, кафе. Живёт на смене со смартфоном.
- **SUPERADMIN** (вторичная) — владелец парка, проверяющий метрики/уведомления с телефона.

USER не затрагивается (публичная часть отдельно, в этом PRD не трогаем).

---

## User Stories

### US-1: Мгновенное бронирование стола Плей Парка с телефона (критический сценарий)

- **Как** MANAGER Плей Парка
- **Я хочу** с телефона создать бронь стола на ближайший час за 3 касания
- **Чтобы** не заставлять клиента ждать у стойки и не переключаться на ноутбук

**Acceptance Criteria:**
- [ ] AC-1.1: На `/admin/ps-park` с экрана 375×667 (iPhone SE) видны без горизонтальной прокрутки: шапка, активные сессии (если есть), расписание (таймлайн), свёрнутая история
- [ ] AC-1.2: Таймлайн на мобильном показывает ресурсы вертикальным списком (плитки-строки), клик по строке открывает пошаговую форму: ресурс → время → клиент → подтверждение
- [ ] AC-1.3: Каждый интерактивный элемент в таймлайне имеет размер ≥ 44×44px
- [ ] AC-1.4: Форма создания брони открывается как bottom sheet (снизу), содержит: выбор длительности чипами (30 мин / 1 ч / 1.5 ч / 2 ч), поле имени, телефон (опционально). Сумма и итоговое время пересчитываются сразу
- [ ] AC-1.5: Кнопка "Забронировать" всегда видна (sticky bottom), не перекрывается экранной клавиатурой
- [ ] AC-1.6: После создания брони bottom sheet закрывается, таймлайн обновляется, на месте нового брони появляется блок
- [ ] AC-1.7: Полный цикл "открыть страницу → создать бронь на 1 час → увидеть подтверждение" укладывается в ≤ 4 касания (ресурс/слот → чип длительности → ввод имени → "Забронировать")
- [ ] AC-1.8: Сценарий работает на iOS Safari 15+ и Chrome Android 100+

### US-2: Мобильная навигация между модулями

- **Как** MANAGER (или SUPERADMIN)
- **Я хочу** быстро переключаться между разделами с телефона
- **Чтобы** не терять время на drag/drop сайдбара

**Acceptance Criteria:**
- [ ] AC-2.1: На ширине < 1024px сайдбар скрыт по умолчанию
- [ ] AC-2.2: В верхнем header слева — hamburger-кнопка (≥ 44×44px), открывающая drawer с полным списком разделов, overlay затемняет основной контент
- [ ] AC-2.3: Drawer закрывается по тапу на overlay, по свайпу влево, или по кнопке закрытия внутри drawer
- [ ] AC-2.4: Drawer содержит те же разделы, что и десктопный сайдбар, уважает RBAC (только доступные), но **без drag/drop режима редактирования** (на мобильном не нужен)
- [ ] AC-2.5: Badge-счётчики (pending-брони и т.д.) отображаются рядом с пунктами
- [ ] AC-2.6: На десктопе (≥ 1024px) поведение не меняется — сайдбар виден статически, редактирование доступно
- [ ] AC-2.7: Активный раздел подсвечен в drawer

### US-3: Просмотр и управление списком броней на мобильном

- **Как** MANAGER
- **Я хочу** видеть список броней (pending + history) в виде карточек с читаемой информацией и действиями
- **Чтобы** принимать/отменять/звонить клиенту одним тапом

**Acceptance Criteria:**
- [ ] AC-3.1: На `/admin/ps-park` на ширине < 768px блок "Ожидают подтверждения" и "История" отрисованы карточками, не таблицей
- [ ] AC-3.2: Каждая карточка содержит: статус (badge), клиент (имя + телефон), ресурс, дата/время, действия ("Подтвердить", "Отменить", "Позвонить" если телефон есть)
- [ ] AC-3.3: Кнопки действий имеют размер ≥ 44×44px, главное действие визуально выделено
- [ ] AC-3.4: На десктопе (≥ 1024px) — прежнее таблич­ное представление
- [ ] AC-3.5: Pending-карточка визуально отличается (amber-рамка или фон)

### US-4: Управление ресурсами с телефона

- **Как** MANAGER Плей Парка
- **Я хочу** отредактировать цену/название/активность стола с телефона
- **Чтобы** реагировать на акции и изменения без доступа к ноутбуку

**Acceptance Criteria:**
- [ ] AC-4.1: На `/admin/ps-park/resources` (и аналогичных) список ресурсов на мобильном — карточки, не таблица
- [ ] AC-4.2: Кнопка "Редактировать" открывает bottom sheet с полями, сохранение одним тапом
- [ ] AC-4.3: Тумблер "Активен/Отключён" работает как нативный switch, ≥ 44px touch area

### US-5: Быстрые действия с активными сессиями PS Park

- **Как** MANAGER на смене
- **Я хочу** с телефона завершить активную сессию и получить чек
- **Чтобы** принимать оплату у стойки сразу

**Acceptance Criteria:**
- [ ] AC-5.1: Панель активных сессий (`ActiveSessionsPanel`) на мобильном — вертикальный стек карточек, каждая карточка fit-to-width
- [ ] AC-5.2: В карточке сессии — имя клиента, стол, таймер до конца, кнопка "Завершить" (primary, ≥ 44px)
- [ ] AC-5.3: Алерт "осталось 10 мин" на мобильном виден сверху, не под сайдбаром
- [ ] AC-5.4: После "Завершить" открывается bill-modal (уже существующий `SessionBillModal`) — адаптирован под мобильный (bottom sheet)

### US-6: Адаптивный дашборд и общие страницы

- **Как** SUPERADMIN
- **Я хочу** увидеть ключевые метрики (бронирования сегодня, статус системы, pending-заявки) с телефона
- **Чтобы** держать руку на пульсе вне офиса

**Acceptance Criteria:**
- [ ] AC-6.1: `/admin/dashboard` на мобильном — карточки одной колонкой, каждая fit-to-width, кликабельна целиком (не только заголовок)
- [ ] AC-6.2: `AdminHeader` на мобильном скрывает подпись "Администратор", показывает hamburger слева, bell и критичный badge справа
- [ ] AC-6.3: Все страницы `src/app/admin/*` имеют `px-4 lg:px-8` для контейнеров (вместо хардкод `p-8`)

## Приоритет (MoSCoW)

- **Must have (в этой итерации)**:
  - US-1 (критический сценарий бронирования PS Park)
  - US-2 (мобильная навигация — без неё ничего не работает)
  - US-3 (списки броней, нужны для принятия решений)
  - US-6 (общие отступы/заголовки, без них остальные экраны ломаются)
- **Should have**: US-4 (ресурсы — реже используются), US-5 (активные сессии — важно, но работоспособно и без доработок)
- **Could have**: mobile-оптимизация аналитических экранов, tables в gazebos/cafe (можно в следующей итерации)
- **Won't have (вне скоупа)**: редактирование сайдбара на мобильном, PWA-установка, push-уведомления, нативное мобильное приложение

**Обоснование приоритета Must**: Без бронирования PS Park с телефона (US-1) основной сценарий не закрыт — это и есть цель задачи. Без навигации (US-2) менеджер даже не откроет нужный раздел.

## Вне скоупа

- **Публичные страницы** (`src/app/(public)/*`, `src/app/webapp/*`) — не трогаем, они уже отдельно адаптированы.
- **Десктопный UX** (`lg+`) — оставляем как есть, никакой регрессии быть не должно.
- **Новая функциональность** — не добавляем новых бизнес-правил (создание брони, управление ресурсами, статусы — всё как сейчас).
- **Новые API endpoints** — переиспользуем существующие (`/api/ps-park/admin-book`, `/api/ps-park/timeline` и др.). Возможны мелкие правки на стороне типов, но не контракта.
- **Полная переработка `Sidebar` drag/drop UX на мобильном** — просто скрываем режим редактирования при ширине < 1024px.
- **Gazebos, Cafe, Rental глубокая мобильная переработка** — в этой итерации делаем только обёртку (контейнеры, отступы, hamburger). Списки/формы внутри — следующая итерация (Could have).
- **A11y полный аудит** — фокус только на touch targets и видимость контента. WCAG AAA не ставим целью.
- **Интеграция с Telegram WebApp** — уже существует `/webapp`, это отдельный интерфейс.
