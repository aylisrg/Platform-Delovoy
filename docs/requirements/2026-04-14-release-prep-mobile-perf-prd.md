# PRD: Подготовка к production-релизу -- Mobile-first, производительность, баги

**Дата:** 2026-04-14
**Статус:** Драфт
**Автор:** Product Owner (Claude)

---

## Проблема

Платформа "Деловой Парк" функционально готова: работают все модули (Барбекю Парк, Плей Парк, Кафе, Парковка, Аренда офисов, Личный кабинет, Админ-панель). Однако перед выходом в production необходимо устранить ряд проблем, которые напрямую влияют на пользовательский опыт:

1. **Мобильные пользователи получают неудобный опыт.** По данным аналогичных бизнесов, 70-80% клиентов B2C-услуг (бронирование беседок, игровых столов, заказ еды) приходят со смартфонов. Несколько страниц содержат таблицы и элементы, не адаптированные для узких экранов.

2. **Производительность страдает из-за отсутствия оптимизации.** Все публичные страницы с данными из БД помечены `force-dynamic`, что полностью отключает серверное кэширование. Видео на главной загружается без ленивой загрузки. Шрифты подключены грамотно (`display: swap`), но другие ресурсы не оптимизированы.

3. **Обнаружены конкретные баги в UI**, которые ухудшают впечатление и могут помешать конверсии.

**Последствия бездействия:**
- Потеря мобильных клиентов на этапе бронирования / заказа (отказы из-за неудобного интерфейса)
- Низкие позиции в мобильном поиске (Google и Яндекс используют Core Web Vitals)
- Плохое первое впечатление от бренда "Деловой Парк"

---

## Решение

Провести целенаправленную доработку фронтенда по трём направлениям: мобильная адаптация, производительность, исправление багов. Без добавления нового функционала -- только доведение существующего до production quality.

---

## Целевая аудитория

| Персона | Контекст использования | Что критично |
|---------|----------------------|--------------|
| USER (гость парка) | Бронирует беседку/стол с телефона, часто в дороге или на месте | Быстрая загрузка, удобные формы на маленьком экране |

## Найденные баги (Code Review)

### BUG-1: Заголовок "Барбекю Парк" нечитаем на видео-фоне (Критичный)

**Где:** Страница `/gazebos`, hero-секция
**Суть:** Заголовок `<h1>` имеет цвет `text-[#1d1d1f]` (почти чёрный), но расположен поверх тёмного видео с полупрозрачным оверлеем `from-black/70`. Тёмный текст на тёмном фоне -- заголовок практически не виден.
**Строка:** `src/app/(public)/gazebos/page.tsx`, строка 62

### BUG-2: Таблицы в Личном кабинете не скроллятся на мобильном (Критичный)

**Где:** Страница `/dashboard`
**Суть:** Таблицы бронирований и заказов используют `<table>` с 5 колонками без горизонтального скролла. На экране шириной <375px столбцы сжимаются до нечитаемого состояния, текст обрезается.
**Строка:** `src/app/(public)/dashboard/page.tsx`, строки 108-149, 162-199

### BUG-3: Корзина кафе скрыта за контентом на мобильном (Значительный)

**Где:** Страница `/cafe`, компонент `MenuList`
**Суть:** Используется layout `grid-cols-1 lg:grid-cols-3` -- на мобильном корзина оказывается ниже всего меню, и пользователь может не догадаться прокрутить вниз. Нет визуального индикатора корзины, пока пользователь не проскроллит до конца.
**Файл:** `src/components/public/cafe/menu-list.tsx`

### BUG-4: Контакты на лендинге содержат плейсхолдеры (Критичный)

**Где:** Компонент `ContactsSection`, лендинг
**Суть:** Телефон отображается как `+7 (XXX) XXX-XX-XX`, WhatsApp как `+7 (XXX) XXX-XX-XX`. Ссылки ведут на `tel:+7XXXXXXXXXX` и `https://wa.me/7XXXXXXXXXX`. Посетитель не может связаться с парком.
**Файл:** `landing-delovoy-park.ru/components/contacts-section.tsx`, строки 1-20

### BUG-5: Телефон в JSON-LD schema.org тоже плейсхолдер (Средний)

**Где:** Главная страница, structured data
**Суть:** `telephone: "+7-000-000-00-00"` в JSON-LD. Поисковые системы могут показать этот номер в результатах поиска.
**Файл:** `src/app/page.tsx`, строка 37

### BUG-6: Нет viewport meta-тега (Критичный)

**Где:** `src/app/layout.tsx`
**Суть:** В root layout отсутствует экспорт `viewport` из Next.js metadata API. Next.js 15 не добавляет viewport meta-тег автоматически через `metadata` -- для этого нужен отдельный экспорт `viewport`. Без `<meta name="viewport" content="width=device-width, initial-scale=1">` мобильные браузеры рендерят страницу в десктопном режиме -- все проблемы с мобильной адаптацией усугубляются.

### BUG-7: Страница Парковки не использует общий Navbar/Footer (Незначительный)

**Где:** Страница `/parking`
**Суть:** Страницы `/gazebos` и лендинг используют компонент `<Navbar />` и `<Footer />`. Страница парковки использует простой `<header>` с текстовой ссылкой "Главная". Несогласованный внешний вид между страницами.
**Файл:** `src/app/(public)/parking/page.tsx`

### BUG-8: Страница Аренды офисов не использует общий Navbar/Footer (Незначительный)

**Где:** Страница `/rental`
**Суть:** Аналогично парковке -- свой header, нет Navbar/Footer.
**Файл:** `src/app/(public)/rental/page.tsx`

### BUG-9: Страница Кафе не использует общий Navbar/Footer (Незначительный)

**Где:** Страница `/cafe`
**Суть:** Аналогично -- свой header, нет Navbar/Footer.
**Файл:** `src/app/(public)/cafe/page.tsx`

### BUG-10: Плей Парк не использует общий Navbar, свой footer (Незначительный)

**Где:** Страница `/ps-park`
**Суть:** Есть свой footer с простой ссылкой "На главную", но нет общего Navbar. Тёмная тема страницы может потребовать тёмного варианта Navbar.
**Файл:** `src/app/(public)/ps-park/page.tsx`

## User Stories

### US-1: Viewport и базовая мобильная совместимость

> Как USER, открывающий сайт со смартфона, я хочу видеть страницу, адаптированную под экран моего устройства, чтобы не было необходимости масштабировать и горизонтально скроллить.

**Приоритет:** Must have

**Acceptance Criteria:**
1. В root layout (`layout.tsx`) экспортирован `viewport` с `width: "device-width"` и `initialScale: 1`
2. На всех публичных страницах (`/`, `/gazebos`, `/ps-park`, `/cafe`, `/parking`, `/rental`, `/dashboard`) при ширине экрана 375px отсутствует горизонтальный скролл
3. Текст на всех страницах читаем без масштабирования при ширине 320px
4. Проверено в Chrome Android (последние 2 версии), Safari iOS (последние 2 версии), Samsung Internet (последняя версия)

---

### US-2: Мобильная адаптация таблиц в Личном кабинете

> Как USER на смартфоне, я хочу видеть свои бронирования и заказы в удобном формате, чтобы понимать статус каждой записи.

**Приоритет:** Must have

**Acceptance Criteria:**
1. На экранах < 640px таблицы бронирований и заказов отображаются в формате карточек (card layout) вместо табличного вида, либо таблица получает горизонтальный скролл с фиксированным первым столбцом
2. Каждый элемент (бронирование/заказ) содержит всю ключевую информацию: модуль, ресурс, дата, время, статус
3. Статус-бейджи читаемы и различимы по цвету на маленьком экране
4. Пользователь может просмотреть все свои записи без горизонтального скролла

---

### US-3: Мобильная корзина кафе

> Как USER, заказывающий еду с телефона, я хочу видеть свою корзину и её состояние, не прокручивая всё меню до конца, чтобы быстро оформить заказ.

**Приоритет:** Must have

**Acceptance Criteria:**
1. На мобильных экранах (< 1024px) при добавлении товара в корзину появляется фиксированная плашка внизу экрана с количеством товаров и суммой
2. По нажатию на плашку открывается корзина (bottom sheet или отдельный экран)
3. Корзина содержит: список товаров, количество, стоимость, кнопку "Оформить заказ"
4. Из корзины можно изменить количество товара или удалить позицию
5. Плашка корзины не перекрывает кнопки "В корзину" на карточках меню

---

### US-4: Исправление заголовка на странице Барбекю Парк

> Как USER, заходящий на страницу Барбекю Парк, я хочу видеть название раздела чётко и контрастно, чтобы сразу понять, где я нахожусь.

**Приоритет:** Must have

**Acceptance Criteria:**
1. Заголовок "Барбекю Парк" на hero-секции читается с первого взгляда на любом устройстве
2. Контрастность текста относительно фона соответствует WCAG AA (контрастность >= 4.5:1 для обычного текста, >= 3:1 для крупного)
3. Подзаголовок также контрастен и читаем

---

### US-5: Единообразная навигация на всех публичных страницах

> Как USER, переходящий между разными страницами парка, я хочу видеть единообразную навигацию (header/footer), чтобы легко ориентироваться и переходить между разделами.

**Приоритет:** Should have

**Acceptance Criteria:**
1. Все публичные страницы (`/cafe`, `/parking`, `/rental`, `/ps-park`, `/gazebos`) используют общий компонент навигации
2. Навигация содержит ссылки на основные разделы и работает корректно на мобильном (бургер-меню)
3. Footer единообразен на всех страницах
4. Для тёмных страниц (Плей Парк) допускается тёмный вариант навигации, но с тем же набором ссылок
5. Страница Личного кабинета (`/dashboard`) может иметь упрощённую навигацию (назад + логотип)

---

### US-6: Реальные контактные данные

> Как USER, желающий связаться с парком, я хочу видеть настоящие телефоны и ссылки на мессенджеры, чтобы позвонить или написать.

**Приоритет:** Must have

**Acceptance Criteria:**
1. Телефон, Telegram и WhatsApp в секции "Контакты" на лендинге содержат реальные данные (не плейсхолдеры)
2. Ссылки `tel:` и `https://wa.me/` ведут на правильные номера
3. JSON-LD на главной странице содержит реальный номер телефона
4. При нажатии на телефон на мобильном устройстве открывается приложение звонка с правильным номером
5. При нажатии на Telegram/WhatsApp открывается соответствующее приложение (или веб-версия)

---

### US-7: Производительность загрузки страниц

> Как USER, открывающий сайт парка с телефона в дороге (3G/4G), я хочу видеть содержимое страницы менее чем за 3 секунды, чтобы не уйти к конкурентам.

**Приоритет:** Must have

**Acceptance Criteria:**
1. Largest Contentful Paint (LCP) главной страницы < 2.5 сек при тестировании в Lighthouse на мобильном профиле
2. Cumulative Layout Shift (CLS) < 0.1 на всех публичных страницах
3. First Input Delay (FID) / Interaction to Next Paint (INP) < 200ms
4. Видео на главной и странице Барбекю Парк загружается лениво (не блокирует рендер)
5. Изображения (когда будут добавлены) используют формат WebP/AVIF и `next/image` с ленивой загрузкой

---

### US-8: Кэширование публичных страниц

> Как USER, возвращающийся на сайт повторно, я хочу видеть страницу мгновенно, чтобы не ждать загрузки каждый раз.

**Приоритет:** Should have

**Acceptance Criteria:**
1. Главная страница (`/`) рендерится статически (SSG) и ревалидируется не чаще чем раз в 60 секунд
2. Страница Парковки (`/parking`) рендерится статически (данные не зависят от БД в реальном времени)
3. Страницы Кафе, Беседок, Плей Парк используют ISR (Incremental Static Regeneration) с revalidation period 30-60 секунд вместо `force-dynamic`
4. Страница Аренды офисов использует ISR с revalidation period 300 секунд
5. Страница Личного кабинета остаётся `force-dynamic` (персональные данные)
6. Время ответа сервера (TTFB) для кэшированных страниц < 100ms

---

### US-9: Оптимизация видео на мобильном

> Как USER на мобильном устройстве с ограниченным трафиком, я не хочу автоматически загружать тяжёлые видео, чтобы сэкономить трафик и ускорить загрузку.

**Приоритет:** Should have

**Acceptance Criteria:**
1. На мобильных устройствах (< 768px) на главной странице вместо автопроигрывания видео показывается статичная картинка-постер (уже реализовано для hero, нужно проверить работоспособность)
2. На странице Барбекю Парк видео загружается лениво (атрибут `loading="lazy"` или Intersection Observer)
3. Видео на странице беседок не блокирует рендер основного контента
4. Указан атрибут `preload="none"` или `preload="metadata"` для видео, которые не сразу видны

---

### US-10: Сенсорные цели (Touch Targets) на мобильном

> Как USER с обычным смартфоном, я хочу легко попадать пальцем в кнопки и ссылки, чтобы не промахиваться и не раздражаться.

**Приоритет:** Should have

**Acceptance Criteria:**
1. Все интерактивные элементы (кнопки, ссылки, слоты времени) имеют минимальный размер 44x44px на мобильных устройствах
2. Слоты выбора времени на страницах бронирования (Барбекю Парк, Плей Парк) имеют достаточный отступ между собой (минимум 8px)
3. Кнопки корзины (+/-) в кафе достаточно крупные для нажатия пальцем
4. Категории меню (фильтр-пиллы) в кафе имеют достаточную область нажатия
5. Ни один интерактивный элемент на публичных страницах не имеет размер менее 44px по любой оси

---

### US-11: Мобильная оптимизация модального окна авторизации

> Как USER на смартфоне, я хочу удобно авторизоваться через модальное окно, чтобы быстро перейти к бронированию.

**Приоритет:** Should have

**Acceptance Criteria:**
1. Модальное окно авторизации занимает полную ширину экрана на мобильных устройствах (full-screen или bottom sheet)
2. Поля ввода (email, пароль, телефон) не перекрываются клавиатурой
3. Кнопки OAuth (Google, Яндекс, VK, Telegram) достаточно крупные для нажатия
4. Модальное окно корректно закрывается свайпом или кнопкой "Закрыть"
5. Фокус автоматически переходит на первое поле ввода при открытии

---

### US-12: Sticky-панель бронирования на мобильном

> Как USER, выбравший слоты на странице Плей Парк с телефона, я хочу видеть сводку и кнопку "Забронировать" без необходимости скроллить, чтобы быстро завершить бронирование.

**Приоритет:** Could have

**Acceptance Criteria:**
1. Sticky-панель бронирования на странице Плей Парк (уже реализована как `sticky bottom-4`) корректно отображается на мобильных экранах
2. Панель не перекрывает нижние слоты выбора времени
3. Информация в панели (название стола, количество часов, сумма) читаема на узком экране
4. Кнопка "Забронировать" легко нажимается пальцем

## Вне скоупа

1. **Редизайн визуального стиля** -- текущий Apple-inspired дизайн остаётся, только адаптируем под мобильные
2. **Новый функционал** -- никаких новых фич, только доводка существующих
3. **Админ-панель** -- мобильная адаптация админки в этом PRD не рассматривается (менеджеры работают с десктопа)
4. **Telegram-бот** -- бот уже работает, его оптимизация не входит в скоуп
5. **SEO-аудит** -- только исправление критичных проблем (JSON-LD). Полный SEO-аудит -- отдельный PRD
6. **Добавление реальных фото** -- на страницах остаются плейсхолдеры для фото. Загрузка реальных фото -- задача контент-менеджера, не разработчика
7. **Backend-оптимизация** -- запросы к БД, индексы, Redis-кэш. Только фронтенд-кэширование (ISR)
8. **Интеграционные тесты мобильных браузеров** -- ручная проверка через DevTools, автоматизация e2e-тестов на мобильных -- отдельная задача
9. **Dark mode** -- поддержка тёмной темы ОС не входит в скоуп
10. **i18n** -- многоязычность не входит в скоуп (только русский)

## Технические замечания (для архитектора)

> Эти заметки основаны на обзоре кода и предназначены для архитектора, принимающего технические решения. Product Owner фиксирует наблюдения, но не предписывает реализацию.

1. **Все публичные страницы с данными помечены `force-dynamic`.** Это означает, что каждый запрос идёт в БД, даже если данные меняются редко (меню кафе, список беседок). Архитектор должен определить стратегию кэширования для каждой страницы.

2. **Отсутствует `next/image`.** Когда реальные фото будут добавлены, важно использовать компонент `<Image>` из Next.js для автоматической оптимизации (WebP, responsive sizes, lazy loading).

3. **Два компонента загружают отзывы с API параллельно.** `HeroSectionWithVideo` и `AdvantagesSection` оба делают `fetch("/api/reviews")` независимо, создавая два запроса вместо одного. Потенциальная оптимизация -- поднять данные на уровень страницы.

4. **Видео на странице беседок загружается без `preload="metadata"`.** В отличие от hero на главной, где `preload="metadata"` указан, на странице gazebos видео начинает загрузку сразу (`autoPlay` без `preload`).

5. **Carousel отзывов использует `style jsx`.** Это единственное место в проекте, где используется CSS-in-JS. Для единообразия можно перевести на Tailwind-класс.

6. **Размер шрифта на кнопках корзины кафе.** Кнопки +/- имеют фиксированный размер 28x28px (`w-7 h-7`), что меньше рекомендуемого минимума 44x44px для мобильных устройств.
