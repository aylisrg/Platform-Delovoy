# PRD: Управленка — учёт расходов бизнес-парка

**Дата:** 2026-04-18
**Автор:** Product Owner
**RUN_ID:** `2026-04-18-cost-tracker-management-accounting`
**Статус:** Ready for Architect

---

## Проблема

Владелец парка "Деловой" начал тратить деньги на несколько IT-сервисов и рекламных каналов. Прямо сейчас нет единого места, где видно: сколько потрачено на хостинг, домен, рекламу, телефонию и прочие операционные нужды. Данные разбросаны по почте с чеками, личному кабинету Timeweb и кабинету Яндекс.Директа.

Это создаёт три практических проблемы:

1. **Потеря денег:** Подписки продлеваются автоматически, а владелец не замечает сервисы, которые перестали использоваться.
2. **Отсутствие картины:** Невозможно быстро ответить на вопрос "сколько мы тратим в месяц на поддержание платформы?" — нужно собирать данные из 4–5 источников.
3. **Масштабирование без контроля:** По мере роста (реклама, новые API-сервисы, дополнительная телефония) расходы будут только расти, и без трекера это превратится в "непонятные списания".

**Персона:** SUPERADMIN — владелец/управляющий парка, единственный человек с доступом к финансовым данным всей платформы.

---

## Решение

Раздел "Управленка" в административной панели — единая точка учёта расходов.

Механика работы в двух режимах:

1. **Recurring (автоматические):** Настраиваешь один раз — Timeweb 3 500 ₽/месяц, домен 1 400 ₽/год — и система сама создаёт запись расхода в нужную дату каждый месяц (или год). Не нужно помнить о продлениях: запись уже в системе, можно в любой момент проверить.

2. **Разовые (ручной ввод):** Потратил на что-то единоразово (настройка стороннего сервиса, дополнительные SMS для акции) — добавил вручную в 3 клика.

Итог: в начале каждого месяца открываешь "Управленку" и видишь: сколько потрачено в этом месяце, по каким категориям, какие recurring-расходы ожидаются до конца месяца.

---

## Что уже реализовано

- `FinancialTransaction` — иммутабельный финансовый лог для PS Park (SESSION_PAYMENT, REFUND, ADJUSTMENT). Используется только для доходов от бронирований. Расходов нет.
- `StockReceipt` + `StockReceiptItem` — закупки на склад кафе. Это расходы на товары, но они привязаны к инвентаризации, а не к управленческому учёту.
- `AuditLog` — логирование всех мутаций.
- `Module.config` (JSONB) — конфигурация модулей.

Единой модели для операционных расходов (IT, реклама, телефония) в схеме нет. Нужны новые модели.

## Целевая аудитория

| Персона | Роль в системе | Взаимодействие с фичей |
|---------|---------------|------------------------|
| Владелец / управляющий | SUPERADMIN | Единственный пользователь — настраивает recurring расходы, вводит разовые, смотрит дашборд |

MANAGER не имеет доступа к "Управленке" — это финансовые данные бизнеса, а не операционные данные модуля.

## User Stories

- **Как** владелец парка (SUPERADMIN)

**Acceptance Criteria:**

- [ ] AC-1.1: Страница `/admin/management` доступна только пользователям с ролью `SUPERADMIN`. Попытка доступа с другой ролью возвращает 403.

- [ ] AC-1.6: Список расходов за выбранный период отображается в виде таблицы: дата, название, категория, тип (разовый / повторяющийся), сумма. Таблица сортируется по дате (новые сначала).
- [ ] AC-1.7: Таблица поддерживает фильтрацию по категории.
- [ ] AC-1.8: Раздел "Управленка" добавлен в sidebar административной панели между "Аналитикой" и "Обратной связью".

### US-2: Настройка recurring-расходов (автоматические подписки)

- **Как** владелец парка (SUPERADMIN)

**Acceptance Criteria:**

- [ ] AC-2.1: В разделе "Управленка" есть вкладка "Регулярные расходы" со списком всех настроенных recurring-расходов.
- [ ] AC-2.2: Форма создания recurring-расхода содержит обязательные поля: название (текст, до 120 символов), категория (выбор из списка), сумма (положительное число, в рублях), периодичность (ежемесячно / ежеквартально / ежегодно), дата первого списания (date picker).
- [ ] AC-2.3: Необязательные поля формы: описание/комментарий (свободный текст), URL сервиса (например, ссылка на личный кабинет Timeweb).

- [ ] AC-2.6: Фоновый процесс ежедневно в 09:00 МСК проверяет все активные recurring-расходы и создаёт запись в журнале расходов, если наступила дата списания.
- [ ] AC-2.7: Созданная автоматически запись помечается флагом `isAutoGenerated = true` и содержит ссылку на recurring-расход (`recurringExpenseId`).
- [ ] AC-2.8: Recurring-расход можно редактировать (имя, сумма, категория, периодичность). Изменение вступает в силу для следующей записи — уже созданные записи не пересчитываются.
- [ ] AC-2.9: Удаление recurring-расхода мягкое (soft delete) — исторические записи расходов остаются в журнале.

## API-контракты

Все эндпоинты защищены проверкой `role === 'SUPERADMIN'`. Возвращают стандартный формат `apiResponse` / `apiError`.

### Recurring-расходы

```
GET    /api/management/recurring          — список recurring-расходов (активных и приостановленных)
POST   /api/management/recurring          — создать recurring-расход
PATCH  /api/management/recurring/:id      — обновить (название, сумма, категория, активность)
DELETE /api/management/recurring/:id      — soft delete (isDeleted = true)
```

Тело `POST /api/management/recurring`:
```json
{
  "name": "Timeweb VPS хостинг",
  "category": "it_infrastructure",
  "amount": 3500,
  "frequency": "monthly",
  "nextBillingDate": "2026-05-01",
  "description": "Основной VPS для delovoy-park.ru",
  "serviceUrl": "https://timeweb.cloud"
}
```

### Журнал расходов (отдельные записи)

```
GET    /api/management/expenses           — список расходов с фильтрами: ?from=&to=&category=&type=
POST   /api/management/expenses           — создать разовый расход вручную
PATCH  /api/management/expenses/:id       — редактировать (только ручные, isAutoGenerated = false)
DELETE /api/management/expenses/:id       — удалить (только ручные)
```

Тело `POST /api/management/expenses`:
```json
{
  "name": "Настройка рекламной кампании",
  "category": "advertising",
  "amount": 5000,
  "date": "2026-04-18",
  "description": "Работа агентства по настройке Яндекс.Директ"
}
```

## Схема данных (новые модели)

Архитектурные решения по моделям — в зоне ответственности Architect. Ниже — бизнес-требования к данным.

**RecurringExpense** — шаблон повторяющегося расхода:
- Имя, категория, сумма, периодичность (monthly / quarterly / yearly)
- Дата следующего списания, дата последнего созданного расхода
- Статус активности, дата создания, создан пользователем (userId)
- Необязательные: описание, URL сервиса

**Expense** — отдельная запись расхода (как ручной, так и auto-generated):
- Имя, категория, сумма, дата расхода
- Флаг `isAutoGenerated` — true для автосозданных
- Ссылка на `RecurringExpense` (nullable — для разовых расходов ссылки нет)
- Создан пользователем (userId), дата создания записи
- Поддержка soft delete для ручных записей

Автоматические записи (`isAutoGenerated = true`) нельзя редактировать и удалять через API — только через редактирование самого recurring-шаблона.

## Приоритет (MoSCoW)

**Категория: Should have**

**Обоснование:** Платформа запущена 17 апреля 2026. Расходы на IT-сервисы уже идут. Без трекера владелец не видит полной картины затрат. Это не блокирует работу бизнеса (клиенты бронируют, менеджеры работают), но напрямую влияет на управляемость: невидимые расходы растут незамеченными.

Статус "Should have" (не "Must have") обоснован: сейчас расходов относительно мало (3–5 сервисов), их можно отслеживать вручную. При росте числа рекламных каналов и сервисов (Phase 5.1–5.3) отсутствие трекера станет реальной болью. Поэтому лучше сделать сейчас, пока систем немного — проще заполнить начальные данные.

**Зависимости:**
- Phase 0: `AuditLog` — уже реализован, используется для логирования мутаций.
- Не зависит от Phase 5.1 (лояльность) и 5.2 (резиденты).
- Phase 5.3 (Дашборд владельца) будет использовать данные "Управленки" для P&L-строки — желательно иметь данные к тому моменту.

## Метрики успеха

| Метрика | Базовое значение | Целевое значение |
|---------|-----------------|-----------------|
| Доля известных IT-расходов, зафиксированных в системе | 0% (данных нет) | 100% известных подписок настроены как recurring через 1 неделю после деплоя |
| Время на получение ответа "сколько потрачено в этом месяце" | 15–20 минут (ручной сбор) | < 30 секунд (открыть дашборд) |
| Recurring-расходы с суммой 0 ₽ | N/A | 0 (все заполнены) через 2 недели после деплоя |
| Ежемесячная открываемость раздела суперадмином | 0 | Минимум 1 раз в месяц (владелец проверяет расходы) |

## Вне скоупа

- **Доходы в "Управленке"** — только расходы в этой итерации. P&L (прибыль/убыток) — Phase 5.3.
- **Автоматическое получение данных из Timeweb, Яндекса, GitHub через API** — нет интеграций с внешними сервисами. Суммы вносятся вручную или через recurring-шаблоны.
- **Интеграция с банковским счётом** — без банковского API, без импорта выписок.
- **Пользовательские категории расходов** — фиксированный список из 4 категорий. Расширение — в следующей итерации.
- **Доступ MANAGER к данным "Управленки"** — только SUPERADMIN. Менеджеры не видят финансовых данных бизнеса.
- **Утверждение расходов (approval workflow)** — нет согласования, владелец вводит сам.
- **Экспорт в Excel / CSV** — вне скоупа MVP. Данные доступны через API.
- **Расходы на закупки склада кафе** — уже трекаются через `StockReceipt` / `StockReceiptItem`. В "Управленку" не переносятся — это операционный учёт склада, а не управленческий.
- **Учёт налогов и НДС** — не предусмотрено.
- **Мультивалютность** — только рубли.
- **Уведомления** — алерты о превышении бюджета категории вне скоупа MVP.

## Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Владелец не заполнит суммы в seed-данных и будет видеть 0 ₽ везде | Высокая | Среднее | AC-4.2: визуальный маркер "Укажите сумму" на записях с нулём |
| Фоновый процесс пропускает дату создания recurring-записи (например, сервер был недоступен) | Средняя | Низкое | При следующем запуске процесс проверяет все пропущенные даты и создаёт записи ретроактивно |
| Автосозданную запись пытаются отредактировать / удалить через прямой API-вызов | Низкая | Среднее | `isAutoGenerated = true` блокирует PATCH и DELETE на уровне сервиса (возврат 422) |
| Phase 5.3 (Дашборд владельца) читает данные "Управленки" как часть P&L — схема может не совпасть | Средняя | Среднее | Зафиксировать в ADR: `Expense.amount` всегда в рублях, `Expense.date` — фактическая дата расхода (не createdAt) |
