## Контекст и проверка существующего

Перед написанием требований проверено текущее состояние кода:

- `src/lib/auth.ts` — YandexProvider уже реализован как custom OAuth, magic-link и credentials-провайдеры присутствуют. Google и VK тоже зарегистрированы.
- `src/components/ui/auth-modal.tsx` — AuthModal уже показывает кнопки Google, Яндекс, VK. Magic link email flow реализован на уровне UI.
- `src/modules/notifications/channels/email.ts` — emailAdapter через Resend существует, но шаблон примитивный: генерирует `<p>текст</p>` без HTML-дизайна.
- Страницы `/gazebos` и `/ps-park` уже содержат телефон из `telephony/service` — отображается в херо-секции.
- `landing-delovoy-park.ru/components/services-section.tsx` — фото беседок: используется `/media/IMG_3843-HDR_Custom.JPG.webp`.
- `landing-delovoy-park.ru/components/hero-section-with-video.tsx` — оверлей: `bg-white/70` (очень плотный белый).

**Вывод**: значительная часть auth-логики уже написана. Батч сосредоточен на: (1) устранении нерабочих провайдеров из UI, (2) полноценном email-сервисе с HTML-шаблонами, (3) трёх точечных UI-улучшениях лендинга, (4) виджете звонка на страницах бронирования.

## Проблемы и персоны

### Проблема A: Сломанные провайдеры в модале входа

В `AuthModal` отображаются кнопки Google и VK. Google OAuth требует настройки `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET`, которые не сконфигурированы в production. VK OAuth v2 имеет ограничения по scope и нестабильно работает. Пользователь нажимает кнопку — попадает на ошибку OAuth. Это ломает доверие.

**Персона**: USER — клиент парка, пытающийся войти.

### Проблема B: Email-уведомления без дизайна

Email-адаптер отправляет голый текст в `<p>` тегах. Клиент получает подтверждение бронирования или заказа в виде нечитаемого письма без брендинга, без структуры, без кнопки действия. Это выглядит как спам.

**Персона**: USER — клиент, получивший уведомление о бронировании беседки или заказе кафе.

### Проблема C: Нет заметного способа позвонить для бронирования

На страницах `/gazebos` и `/ps-park` телефон есть в херо, но только на десктопе — после скролла вниз к форме бронирования кнопка исчезает из видимости. Часть клиентов предпочитает звонить, но на страницах нет постоянно доступного напоминания об этом.

**Персона**: USER — посетитель старшего возраста или тот, кто не доверяет онлайн-форме.

### Проблема D: Видео на лендинге не видно

Hero-секция главной страницы показывает видео с парком, но поверх него стоит `bg-white/70` — 70% белый оверлей. Видео фактически не видно, остаётся только текст на белом фоне. Смысл видео-фона теряется.

**Персона**: потенциальный клиент или арендатор, впервые заходящий на сайт.

### Проблема E: Неактуальное фото беседок на главной

Секция "Всё для работы и отдыха" показывает `/media/IMG_3843-HDR_Custom.JPG.webp` для карточки Барбекю Парка. Это фото не передаёт атмосферу беседок так же хорошо, как фотографии, используемые на самой странице `/gazebos` (poster: `/media/IMG_3724_Custom.JPG.webp`).

**Персона**: посетитель главной страницы.

### Проблема F: Интерфейс статичный — нет жизни

Кнопки, карточки и секции не реагируют на взаимодействие пользователя. Нет плавных переходов при появлении контента. Сайт выглядит как статичная брошюра, а не живая платформа.

**Персона**: USER, MANAGER — все пользователи платформы.

## User Stories

---

### US-1: Вход через Яндекс без сюрпризов

**Модуль**: auth

- **Как** клиент парка
- **Я хочу** войти через Яндекс ID одним кликом
- **Чтобы** не создавать новый аккаунт и не запоминать пароль — у большинства пользователей Московской области уже есть аккаунт Яндекса

**Acceptance Criteria:**
- [ ] AC-1: На экране входа (AuthModal) видна кнопка "Войти через Яндекс" с иконкой Яндекса
- [ ] AC-2: Клик по кнопке открывает OAuth-страницу Яндекса без ошибок
- [ ] AC-3: После успешной авторизации пользователь возвращается на ту же страницу, откуда нажал "Войти"
- [ ] AC-4: Если пользователь уже существует в системе с тем же email — аккаунты связываются, не создаётся дубль
- [ ] AC-5: Кнопки Google и VK убраны из AuthModal (они нерабочие без настроенных OAuth credentials)
- [ ] AC-6: Вход через Telegram и Email Magic Link остаются доступны

**Приоритет**: Must have — Яндекс является основным OAuth-провайдером для пользователей МО. Текущие кнопки Google/VK ломают опыт входа.

### US-2: Вход через Email Magic Link без пароля

**Модуль**: auth

- **Как** клиент парка, у которого нет аккаунта в Telegram
- **Я хочу** получить одноразовую ссылку на email и войти по ней
- **Чтобы** не создавать пароль и не привязывать соцсеть

**Acceptance Criteria:**
- [ ] AC-1: На экране входа есть опция "Войти по email" — отдельная вкладка или ссылка
- [ ] AC-2: Пользователь вводит email → получает письмо с кнопкой "Войти в Деловой Парк" в течение 60 секунд
- [ ] AC-3: Ссылка в письме действительна 15 минут, одноразовая
- [ ] AC-4: После клика на ссылку пользователь авторизован и перенаправлен на `/dashboard`
- [ ] AC-5: Повторный клик на использованную ссылку показывает страницу "Ссылка уже использована или устарела"
- [ ] AC-6: Письмо отправляется в HTML-формате с брендингом (логотип, название, кнопка CTA)

**Приоритет**: Should have — flow уже частично реализован в UI и backend (`/api/auth/email/send`), нужно убедиться в корректной работе и добавить HTML-шаблон письма.

### US-3: HTML email-уведомление о бронировании беседки

**Модуль**: notifications + gazebos

- **Как** клиент, забронировавший беседку
- **Я хочу** получить красивое подтверждение на email
- **Чтобы** иметь визуально читаемое напоминание о дате, времени и деталях брони

**Acceptance Criteria:**
- [ ] AC-1: После создания брони беседки, если у пользователя есть email, уходит письмо
- [ ] AC-2: Письмо содержит: название беседки, дату, время начала и окончания, статус брони
- [ ] AC-3: Письмо содержит логотип или название "Деловой Парк" в шапке
- [ ] AC-4: В письме есть кнопка "Посмотреть бронирование" → ссылка на `/dashboard`
- [ ] AC-5: При отмене брони уходит отдельное письмо с уведомлением об отмене
- [ ] AC-6: При подтверждении брони менеджером уходит письмо "Ваша бронь подтверждена"
- [ ] AC-7: Письмо корректно отображается в Gmail и Яндекс.Почте (мобильная версия)

**Приоритет**: Must have — без брендированных email-уведомлений платформа выглядит незавершённой. Напрямую влияет на доверие клиентов.

## Технические ограничения (для Architect)

1. **Yandex OAuth** — `YandexProvider()` уже написан в `src/lib/auth.ts`. Нужны только `YANDEX_CLIENT_ID` и `YANDEX_CLIENT_SECRET` в `.env`. Architect должен подтвердить, что custom provider корректно работает с PrismaAdapter (account linking).

2. **Email (Resend)** — `emailAdapter` через Resend уже существует в `src/modules/notifications/channels/email.ts`. Нужен `RESEND_API_KEY` и `RESEND_FROM_EMAIL`. Задача Architect — спроектировать систему HTML-шаблонов (React Email или inline-HTML), интегрированную с существующим `notify()` flow.

3. **Удаление Google/VK из UI** — только изменение `auth-modal.tsx`. Провайдеры могут оставаться зарегистрированными в `auth.ts` (не ломают систему), но убираются из видимости пользователей.

4. **Booking Widget** — страницы `/gazebos` и `/ps-park` — Server Components. Виджет можно добавить как статичный React-компонент без клиентской логики.

5. **Видео оверлей** — изменение одной Tailwind-класса в `hero-section-with-video.tsx`. Нужна проверка контраста.

6. **Фото беседок** — изменение одного значения `coverPhoto` в `services-section.tsx`. Файл `/media/IMG_3724_Custom.JPG.webp` должен существовать в `public/`.

7. **Анимации** — предпочтительно через Tailwind CSS (`transition`, `animate-`) или Framer Motion (уже может быть в deps). Architect выбирает подход.

## Вне скоупа

Следующее **не входит** в этот батч и переносится в следующие итерации:

- **WhatsApp OTP**: провайдер уже зарегистрирован в системе, но интеграция с реальным WhatsApp Business API не настроена — тема отдельного PRD
- **VK OAuth**: убирается из UI, но не удаляется из кода — VK меняет условия API, решение принять позже
- **Google OAuth**: убирается из UI — требует верификации домена в Google Console, отдельная задача devops
- **SMS-уведомления**: не рассматриваются в этом батче
- **Push-уведомления браузера**: не рассматриваются
- **Восстановление пароля для аккаунтов с паролем**: отдельный flow, не входит в этот батч
- **A/B тестирование вариантов оверлея**: не нужно — есть конкретное техническое решение
- **Полный Code Review + Bugfixes как отдельная задача**: выносится в отдельный технический тикет для Developer без PRD (ревью — не продуктовая задача)
- **Программа лояльности, Резиденты, Дашборд владельца**: Phase 5.1-5.3 по дорожной карте, не входят в этот батч
