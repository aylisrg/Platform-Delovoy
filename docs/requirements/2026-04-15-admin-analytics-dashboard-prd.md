# PRD: Панель аналитики рекламы (`/admin/analytics`)

**Дата**: 15 апреля 2026
**Автор**: Product Owner
**Статус**: Draft

---

## Проблема

Владелец парка "Деловой" запустил три рекламные кампании в Яндекс.Директ (Беседки, PS Park, Офисы) и настроил Яндекс.Метрику с 6 целями конверсий. Чтобы оценить эффективность рекламы, ему приходится:

1. Заходить в интерфейс Яндекс.Директ — смотреть расход, клики, CTR по каждой кампании
2. Заходить в Яндекс.Метрику — смотреть трафик, источники, конверсии по целям
3. Заходить в админку платформы — смотреть фактические бронирования и заявки
4. Вручную сопоставлять данные: сколько потратил на рекламу -> сколько кликов -> сколько бронирований -> сколько денег заработал

Это 3 разных интерфейса, каждый со своей логикой, и ни один из них не показывает полную картину "реклама -> бронирования -> деньги" в контексте бизнеса парка.

**Последствия:**
- Владелец не понимает, какая кампания окупается, а какая сливает бюджет
- Решения о перераспределении бюджета принимаются на ощущениях, а не на данных
- Нет единой точки правды — данные разбросаны по трём системам
- Уходит 20-30 минут в день на ручной сбор цифр вместо принятия решений

---

## Решение

Единая панель аналитики рекламы в админке платформы (`/admin/analytics`), которая в одном экране показывает:
- **Трафик**: сколько людей пришло на сайт, откуда, что делали
- **Реклама**: сколько потратили, сколько кликов получили, какой CTR
- **Конверсии**: сколько из посетителей забронировали беседку, PS Park или отправили заявку на офис
- **Бизнес-результат**: связь между расходом на рекламу и реальными бронированиями/заявками

Данные обновляются автоматически. Владелец открывает одну страницу и за 30 секунд понимает: реклама работает или нет, куда двигать бюджет.

---

## Целевая аудитория

| Персона | Как использует панель |
|---------|----------------------|
| **SUPERADMIN** (владелец парка) | Ежедневно проверяет эффективность рекламы, принимает решения по бюджетам, отслеживает тренды конверсий |
| **MANAGER** (менеджер модуля) | Видит аналитику по своему направлению: менеджер беседок видит конверсии беседок, менеджер PS Park — свои (опционально) |

## User Stories

### US-1: Обзорная панель — вся картина на одном экране

**Как** владелец парка (SUPERADMIN),
**я хочу** видеть ключевые метрики рекламы и трафика на одном экране,
**чтобы** за 30 секунд понять текущее состояние дел без переключения между системами.

**Acceptance Criteria:**
- AC-1.1: Страница `/admin/analytics` доступна пользователям с ролью SUPERADMIN
- AC-1.2: На странице отображается блок "Трафик" с данными из Яндекс.Метрики: количество визитов, просмотры страниц, показатель отказов (bounce rate) за выбранный период
- AC-1.3: На странице отображается блок "Реклама (Директ)" с суммарными данными: показы, клики, CTR, общий расход в рублях, средняя цена клика — за выбранный период
- AC-1.4: На странице отображается блок "Конверсии" с количеством достигнутых целей по каждому направлению: беседки (submit/success), PS Park (submit/success), офисы (submit/success) — за выбранный период
- AC-1.5: Все данные загружаются при открытии страницы, время загрузки не более 5 секунд
- AC-1.6: Если API Яндекса недоступен или не настроен, вместо данных показывается понятное сообщение: "Интеграция не настроена" с указанием, какие переменные окружения нужно добавить

**Приоритет**: Must Have

---

### US-2: Выбор периода анализа

**Как** владелец парка,
**я хочу** переключать период анализа (сегодня, 7 дней, 30 дней, произвольный),
**чтобы** видеть как текущую ситуацию, так и тренды за более длительное время.

**Acceptance Criteria:**
- AC-2.1: На странице есть переключатель периода с вариантами: "Сегодня", "7 дней", "30 дней", "Произвольный период"
- AC-2.2: При выборе "Произвольный период" появляется date picker с возможностью выбрать дату начала и дату окончания
- AC-2.3: При смене периода все данные на странице обновляются без перезагрузки страницы
- AC-2.4: По умолчанию выбран период "7 дней"
- AC-2.5: Максимальный период — 90 дней (ограничение API Яндекса на детализацию)

**Приоритет**: Must Have

---

### US-3: Разбивка по рекламным кампаниям

**Как** владелец парка,
**я хочу** видеть статистику по каждой рекламной кампании отдельно (Беседки, PS Park, Офисы),
**чтобы** понимать, какое направление рекламы эффективнее и куда перераспределить бюджет.

**Acceptance Criteria:**
- AC-3.1: В блоке "Реклама" данные разбиты по кампаниям: "Беседки", "PS Park", "Офисы"
- AC-3.2: По каждой кампании отображаются: статус (активна/остановлена), показы, клики, CTR, расход в рублях, средняя цена клика
- AC-3.3: Кампании отсортированы по расходу (от большего к меньшему)
- AC-3.4: Визуально выделена кампания с лучшим CTR (зеленый индикатор) и с худшим CTR (оранжевый индикатор)

**Приоритет**: Must Have

---

### US-4: Источники трафика

**Как** владелец парка,
**я хочу** видеть, откуда приходят посетители на сайт,
**чтобы** понимать, какие каналы работают помимо платной рекламы.

**Acceptance Criteria:**
- AC-4.1: На странице есть блок "Источники трафика" с разбивкой по каналам: прямые заходы, поисковые системы, рекламные кампании (Директ), социальные сети, другие источники
- AC-4.2: По каждому источнику отображается: количество визитов и доля от общего трафика в процентах
- AC-4.3: Данные представлены в виде наглядного списка или таблицы, отсортированной по количеству визитов

**Приоритет**: Should Have

---

### US-5: Воронка конверсий по направлению

**Как** владелец парка,
**я хочу** видеть воронку конверсий по каждому направлению (беседки, PS Park, офисы),
**чтобы** понимать, на каком этапе теряются клиенты: пришли на сайт -> открыли форму -> отправили заявку -> бронь подтверждена.

**Acceptance Criteria:**
- AC-5.1: Для каждого направления (беседки, PS Park, офисы) отображается воронка из 3 шагов: визиты на страницу модуля -> отправка формы (submit) -> успешная конверсия (success)
- AC-5.2: Между шагами воронки показывается процент перехода (конверсия между этапами)
- AC-5.3: Если конверсия между этапами ниже 10%, шаг подсвечивается как проблемный (визуальное выделение)

**Приоритет**: Should Have

---

### US-6: Сводка "Расход -> Результат"

**Как** владелец парка,
**я хочу** видеть простую сводку: сколько потратил на рекламу и сколько это принесло бронирований/заявок,
**чтобы** оценивать окупаемость рекламы без калькулятора.

**Acceptance Criteria:**
- AC-6.1: На странице есть блок "Эффективность", который показывает: общий расход на Директ за период, общее количество конверсий (успешных бронирований + заявок) за тот же период, стоимость одной конверсии (расход / количество конверсий)
- AC-6.2: Стоимость конверсии рассчитывается отдельно по каждому направлению: стоимость бронирования беседки, стоимость бронирования PS Park, стоимость заявки на офис
- AC-6.3: Если конверсий за период нет, вместо стоимости показывается прочерк (не деление на ноль)
- AC-6.4: Рядом со стоимостью конверсии показывается стрелка тренда: выросла или снизилась по сравнению с предыдущим аналогичным периодом (например, текущая неделя vs. прошлая неделя)

**Приоритет**: Must Have

---

### US-7: Кэширование данных

**Как** владелец парка,
**я хочу** чтобы страница аналитики загружалась быстро,
**чтобы** не ждать ответа от API Яндекса при каждом открытии.

**Acceptance Criteria:**
- AC-7.1: Данные из API Яндекс.Метрики и Яндекс.Директ кэшируются на 15 минут
- AC-7.2: На странице отображается время последнего обновления данных: "Данные обновлены: 15.04.2026, 14:32"
- AC-7.3: Есть кнопка "Обновить данные", которая принудительно запрашивает свежие данные из API (сбрасывает кэш)
- AC-7.4: При ошибке запроса к API показываются последние кэшированные данные с пометкой "Данные могут быть неактуальны"

**Приоритет**: Must Have

---

### US-8: Доступ менеджеров к аналитике своего модуля

**Как** менеджер беседок,
**я хочу** видеть аналитику по конверсиям моего направления,
**чтобы** понимать, приводит ли реклама клиентов на мой модуль.

**Acceptance Criteria:**
- AC-8.1: Менеджер с привязкой к модулю "gazebos" видит блок конверсий только по целям gazebo_booking_submit и gazebo_booking_success
- AC-8.2: Менеджер не видит финансовые данные (расход на рекламу, стоимость конверсии) — только количественные метрики (визиты, конверсии)
- AC-8.3: Менеджер не видит данные других модулей

**Приоритет**: Could Have

## Приоритизация (MoSCoW)

### Must Have (без этого не запускаем)
- Обзорная панель с ключевыми метриками трафика, рекламы и конверсий (US-1)
- Выбор периода анализа: сегодня / 7 дней / 30 дней / произвольный (US-2)
- Разбивка по рекламным кампаниям Директа (US-3)
- Сводка "расход -> результат" со стоимостью конверсии (US-6)
- Кэширование данных API на 15 минут (US-7)

### Should Have (делаем сразу после Must Have)
- Источники трафика из Метрики (US-4)
- Воронка конверсий по направлению (US-5)

### Could Have (если останется время)
- Доступ менеджеров к аналитике своего модуля (US-8)
- Экспорт данных за период в CSV
- Еженедельный автоматический отчёт в Telegram (сводка по расходам и конверсиям)

### Won't Have (в этой версии не делаем)
- Управление кампаниями Директ из админки (запуск, остановка, изменение бюджетов) — только просмотр
- Графики и диаграммы (Recharts и т.д.) — в первой версии достаточно карточек с цифрами и таблиц
- A/B-тестирование объявлений через интерфейс
- Интеграция с Яндекс.Бизнес (карточка организации) — уже частично есть в маркетинге беседок, не дублируем
- Интеграция с Google Analytics / другими системами аналитики
- Прогнозирование бюджетов и конверсий (ML-модели)
- Сравнение с конкурентами (аукцион Директа)

## Метрики успеха

| Метрика | Цель | Как измеряем |
|---------|------|-------------|
| Частота использования | SUPERADMIN открывает панель >= 3 раз в неделю | AuditLog — заходы на `/admin/analytics` |
| Время на принятие решения | Владелец перестаёт заходить в интерфейсы Яндекса для оценки рекламы | Опрос через 2 недели после запуска |
| Обнаружение неэффективных кампаний | Перераспределение бюджета в течение первой недели использования | Факт изменения бюджетов кампаний после начала использования панели |
| Скорость загрузки | Страница загружается за <= 5 секунд (с кэшем) и <= 10 секунд (без кэша) | Замер времени рендеринга |
| Достоверность данных | Расхождение между панелью и интерфейсом Яндекса < 5% | Ручная сверка в первую неделю |

## Вне скоупа

1. **Управление рекламными кампаниями** — панель только для просмотра; запуск, остановка и настройка кампаний делаются через интерфейс Яндекс.Директ
2. **Графики и визуализации** — в первой версии карточки с цифрами и таблицы; графики (тренды по дням, heatmap загрузки) планируются в Phase 5.3 "Дашборд владельца"
3. **Привязка конверсий к конкретным объявлениям** — только агрегат по кампании, без детализации до уровня объявлений и ключевых слов
4. **Атрибуция выручки** — считаем количество конверсий, но не привязываем к ним сумму выручки (для этого нужна сквозная аналитика, которая выходит за рамки этой задачи)
5. **Уведомления об аномалиях рекламы** — автоматические алерты ("бюджет закончился", "CTR упал на 50%") не входят в первую версию
6. **Данные за период до запуска панели** — панель показывает данные начиная с момента настройки интеграций; исторические данные до этого не импортируются
7. **Роль MANAGER** — доступ менеджеров является Could Have и может не войти в первый релиз

## Зависимости

| Зависимость | Статус | Влияние |
|-------------|--------|---------|
| Яндекс.Метрика (счётчик 73068007, 6 целей) | Настроена | Источник данных о трафике и конверсиях |
| Яндекс.Директ (3 кампании) | Настроен | Источник данных о расходах и кликах |
| YANDEX_OAUTH_TOKEN | Есть | Единый токен авторизации для обоих API |
| Существующий клиент `src/lib/yandex.ts` | Есть | Базовый API-клиент, нужно расширить для новых метрик |
| Redis (кэш) | Развёрнут | Кэширование ответов API |
