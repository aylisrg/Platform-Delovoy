# PRD: Безопасный magic-link и публичный личный кабинет USER

**PRD-ID:** 2026-04-25-fix-login-public-profile  
**RUN_ID:** 2026-04-25-fix-login-public-profile  
**Дата:** 2026-04-25  
**Статус:** Передан Senior Dev (Architect упал по timeout, ADR пишет CTO)  
**Приоритет:** Must have

> **CTO-revision (2026-04-25):** при переходе к Stage 2 обнаружена критичная находка: страница `/dashboard` в `src/app/(public)/dashboard/page.tsx` уже существует и реализует **полноценный личный кабинет** — `<title>Личный кабинет</title>`, server-side `auth()`, bookings + orders + feedback + контакты через `<ContactsCard />`. **US-2 не требует создания новой страницы** — это был бы scope creep ровно того типа, который запрещает CLAUDE.md → "Scope guard". Реальный пробел — **навигация**: `redirectAfterLogin` после signin кидает USER на `/` (главная), не на `/dashboard`, и публичной ссылки на кабинет нет. Acceptance criteria US-2 переформулированы соответственно (см. ниже). US-1 (magic-link) не меняется.

## 1. Проблема и контекст

### 1.1 Magic-link: уязвимость авторизации

Платформа отправляет пользователю письмо со ссылкой вида `/api/auth/verify-email?token=X&email=Y`. Этот эндпоинт корректно проверяет токен через таблицу `VerificationToken` и удаляет его после использования (одноразовость реализована в `verifyMagicLink`). Проблема — в следующем шаге: после успешной проверки токена сервер редиректит клиента на `/auth/signin?magic=<userId>`, а страница входа вызывает `signIn("magic-link", { userId })`.

Credentials-провайдер `magic-link` в `src/lib/auth.ts:320–328` принимает `{ userId }` и сразу делает `prisma.user.findUnique({ id: userId })` без какой-либо дополнительной проверки. Это означает: кто угодно, кто знает cuid пользователя (из URL в логах браузера, из shared-device истории, из Referer-заголовка на внешнем ресурсе), может вызвать `signIn("magic-link", { userId: "<чужой-id>" })` и получить сессию этого пользователя. Токен при этом уже не нужен — он был инвалидирован при редиректе.

**Бизнес-риск:** вход в чужой аккаунт без пароля и без какого-либо доказательства владения email. Для платформы, хранящей бронирования, историю заказов и контактные данные — это критическая уязвимость.

### 1.2 Отсутствие публичного личного кабинета для USER

Пользователь, забронировавший беседку или сделавший заказ в кафе через веб-браузер, не имеет возможности посмотреть свои данные после авторизации. Существующие страницы:

- `/webapp/profile` — Telegram Mini App, работает только внутри Telegram, использует специфичный для tg-окружения стек.
- `/admin/clients` — доступна только менеджеру/суперадмину.

Готовый API для профиля уже есть: `GET /api/profile`, `PATCH /api/profile`, `GET /api/profile/contacts/*`. Данные о бронированиях доступны через `GET /api/gazebos/bookings?userId=` и `GET /api/ps-park/bookings?userId=`. Данные о заказах — через `GET /api/cafe/orders?userId=`. UI не хватает.

**Бизнес-риск:** пользователь не видит своих бронирований и не может управлять контактами — растут обращения в поддержку. Доверие к платформе снижается.

## 2. Целевая аудитория

- **US-1 (magic-link):** Гость / новый пользователь — любой человек, пытающийся войти через email. Роль до входа не определена.
- **US-2 (личный кабинет):** USER — физическое лицо, которое уже авторизовалось и пользуется B2C-сервисами (беседки, PS Park, кафе).

## 3. User Stories

### US-1: Безопасный вход по email magic-link

- **Как** пользователь, получивший письмо со ссылкой для входа
- **Я хочу** чтобы переход по ссылке из письма давал мне сессию надёжно и однократно
- **Чтобы** никто другой не смог войти в мой аккаунт, даже зная мой идентификатор

**Контекст:** текущий флоу после исправления должен выглядеть так:

```
1. Пользователь запрашивает magic-link → сервер создаёт VerificationToken в БД
2. Пользователь переходит по ссылке → /api/auth/verify-email проверяет token+email,
   инвалидирует VerificationToken, создаёт short-lived session token в Redis (TTL ≤ 5 мин)
3. Сервер редиректит на /auth/signin?magic=<session-token>  (не userId!)
4. Страница вызывает signIn("magic-link", { token: <session-token> })
5. Credentials-провайдер проверяет session-token в Redis, получает userId,
   удаляет запись из Redis (one-time use), возвращает пользователя
6. Сессия установлена
```

**Acceptance Criteria:**

- [ ] **AC-1.1 — Токен обязателен для входа.** Вызов `signIn("magic-link", { userId: "<любой-cuid>" })` без корректного одноразового токена возвращает ошибку авторизации. Сессия не создаётся.
- [ ] **AC-1.2 — Одноразовость.** Один и тот же токен, переданный в `signIn("magic-link", { token })` дважды подряд, работает только первый раз. Второй вызов возвращает ошибку (токен уже использован или не существует).
- [ ] **AC-1.3 — Срок действия.** Токен, не использованный в течение 5 минут после перехода по ссылке из письма (промежуточный redis-токен), перестаёт работать. Пользователь видит сообщение "Ссылка истекла" и предложение запросить новую.
- [ ] **AC-1.4 — VerificationToken не затронут.** Токен в письме (`/api/auth/verify-email?token=…`) по-прежнему инвалидируется немедленно при переходе — поведение этого шага не меняется.
- [ ] **AC-1.5 — Корректный флоу сохраняется.** Пользователь, перешедший по письму в течение 15 минут и завершивший вход в течение 5 минут после редиректа, успешно получает сессию.
- [ ] **AC-1.6 — Тесты зелёные.** `npm test` проходит полностью (базовый счётчик: 1611/1611 или более).
- [ ] **AC-1.7 — Существующие провайдеры не затронуты.** Вход через Telegram, Яндекс, Google, VK, пароль — работают как прежде.

### US-2: Публичный личный кабинет для USER

- **Как** авторизованный пользователь (роль USER)
- **Я хочу** видеть в веб-браузере свои данные: имя, контакты, бронирования и заказы
- **Чтобы** контролировать свою активность в парке без звонков в поддержку

**Acceptance Criteria (CTO-revised):**

- [ ] **AC-2.1 — USER попадает в кабинет после входа.** После любого успешного `signIn` (Telegram, Yandex, Google, VK, password, magic-link) — если роль USER, клиент перенаправляется на `/dashboard`. SUPERADMIN/ADMIN/MANAGER — на `/admin/dashboard` как и прежде.
- [ ] **AC-2.2 — Ссылка на кабинет доступна с публичных страниц.** На главной странице `/` (`src/app/(public)/page.tsx`) для авторизованного USER появляется кнопка/ссылка "Личный кабинет" → `/dashboard`. Для гостя — остаётся "Войти" → `/auth/signin`.
- [ ] **AC-2.3 — `/dashboard` остаётся защищённым.** Поведение не меняется: при отсутствии сессии — редирект (текущий код редиректит на `/`, оставляем как есть, не трогаем).
- [ ] **AC-2.4 — Существующая функциональность `/dashboard` сохранена.** Имя, контакты (Telegram, Yandex, email, phone), бронирования (gazebos + ps-park), заказы (cafe), feedback — продолжают отображаться. Новые поля не добавляются.
- [ ] **AC-2.5 — Кнопка выхода присутствует.** На `/dashboard` есть кнопка "Выйти" — вызывает `signOut({ callbackUrl: "/" })` из `next-auth/react`. Если уже есть в каком-то компоненте — не дублируем.
- [ ] **AC-2.6 — Magic-link редирект ведёт на кабинет.** После завершения magic-link флоу (US-1) USER оказывается на `/dashboard`, не на `/`.
- [ ] **AC-2.7 — Новый модуль и новые API-эндпоинты не создаются.** Никаких `src/modules/<новое>/` и никаких новых `/api/*` не добавляется. Изменения только в существующих файлах.
- [ ] **AC-2.8 — Тесты зелёные.** `npm test` проходит полностью. Логика `redirectAfterLogin` покрыта unit-тестом (для USER → `/dashboard`, для админа → `/admin/dashboard`).

## 6. Вне скоупа

Следующее явно **не входит** в данную итерацию:

- **Telegram WebApp (`/webapp/profile`)** — не трогаем. Отдельный клиент с отдельным стеком.
- **Программа лояльности** (Phase 5.1, issues #61–64) — баланс баллов, уровни, история начислений на странице профиля не появляются.
- **Витрина резидентов** (Phase 5.2, issues #65–68) — claim-механика, профиль резидента.
- **Дашборд владельца** (Phase 5.3) — не затрагивается.
- **Рефакторинг `auth.config.ts`** сверх необходимого — меняем только `Credentials("magic-link").authorize()` и логику формирования редиректа в `verify-email`.
- **Добавление новых OAuth-провайдеров** — не добавляем.
- **Подключение/рефакторинг middleware** — отсутствие `src/middleware.ts` — отдельная задача, не входит в этот pipeline (CTO уточнил: текущей защиты через `auth()` в layout/handler достаточно для MVP).
- **Платёжная информация на странице профиля** — не показываем.
- **Оплата бронирований** — отдельный pipeline.
- **Мобильная вёрстка `/profile`** — базовая адаптивность допустима, специальных mobile-first требований нет.
- **Push-уведомления с профиля** — не добавляем.
- **Сброс пароля через профиль** — не добавляем.
