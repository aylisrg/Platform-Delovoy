# PRD: Привязка офиса к обращению (feedback-office-linkage)

**Версия:** 1.0  
**Дата:** 2026-04-25  
**RUN_ID:** 2026-04-25-feedback-office-linkage  
**Статус:** Готов к передаче Architect

---

## Проблема

Когда арендатор или гость парка отправляет обращение (feedback) через платформу, администратор и менеджер не видят, из какого офиса пришёл вопрос или жалоба. Это замедляет работу: координатор вынужден вручную уточнять у пользователя его местонахождение, тратя дополнительное время на переписку прежде чем приступить к решению проблемы.

Отдельная проблема — качество данных. Если пользователь вводит номер офиса текстом, возникают опечатки: "А-12", "а12", "А 12" — три написания одного офиса. Данные невозможно достоверно сопоставить с реестром офисов.

Сейчас в модели `FeedbackItem` нет поля для офиса вообще. Нет ни связи с реестром, ни текстового поля. Фича отсутствует полностью.

---

## Решение

Добавить в форму обращения необязательное поле выбора офиса с умным поиском по реестру. Пользователь начинает вводить номер офиса — видит список совпадений с корпусом и этажом. Выбирает — обращение получает строгую ссылку на конкретный офис из реестра. Без выбора — форма работает как раньше, поле остаётся пустым.

Этот выбор отображается в личном кабинете пользователя и в администраторском дашборде рядом с обращением.

Архитектура FK на `Office.id` оставляет место для будущей автоподстановки офиса из активного договора аренды — без изменения схемы данных.

---

## Целевая аудитория

| Персона | Как использует фичу |
|---------|---------------------|
| **USER (арендатор)** | Указывает свой офис при создании обращения; видит его в своём дашборде как подтверждение |
| **USER (гость без офиса)** | Поле пропускает — форма работает как раньше |
| **MANAGER / ADMIN** | Видит офис автора в списке обращений и на детальной странице — быстрее идентифицирует источник проблемы |
| **SUPERADMIN (владелец)** | Видит полную картину в дашборде; понимает, из каких офисов приходит больше обращений |

## User Stories

### US-1: Выбор офиса при создании обращения

- **Как** зарегистрированный пользователь платформы (арендатор или гость)
- **Я хочу** указать свой офис при отправке обращения через удобный поиск по номеру
- **Чтобы** администратор сразу знал, откуда я, и мог быстрее помочь без лишних вопросов

**Acceptance Criteria:**

- [ ] AC-1.1: В форме обращения (`feedback-button.tsx`) появляется необязательное поле "Ваш офис" — combobox с поиском.
- [ ] AC-1.2: Combobox активируется при вводе минимум 1 символа и отображает до 10 совпадений по номеру офиса.
- [ ] AC-1.3: Поиск запускается с debounce 200 мс после ввода каждого символа.
- [ ] AC-1.4: Каждый вариант в списке отображает: номер офиса, корпус, этаж, статус (Свободен / Занят). Статус MAINTENANCE и неактивные записи в списке не отображаются.
- [ ] AC-1.5: После выбора офиса из списка combobox показывает выбранный вариант; данные формы содержат `officeId` (идентификатор из реестра), а не текст.
- [ ] AC-1.6: Если пользователь не выбрал офис — форма отправляется без `officeId`. Поле необязательно, отсутствие выбора не блокирует отправку.
- [ ] AC-1.7: При успешной отправке с выбранным офисом — `FeedbackItem.officeId` в базе данных равен `Office.id` выбранного офиса (FK, не строка).
- [ ] AC-1.8: При очистке поля поиска (крестик или ручное удаление текста) выбор сбрасывается, `officeId` не отправляется.
- [ ] AC-1.9: Combobox доступен только авторизованным пользователям (форма и так скрыта для гостей без сессии — требование не меняется).

**Приоритет:** Must have  
**Обоснование:** Без этой story вся фича не работает. Остальные stories зависят от того, что данные попадают в базу.

---

### US-2: Офис автора в личном кабинете ("Мои обращения")

- **Как** пользователь, который отправил обращение с привязкой офиса
- **Я хочу** видеть указанный офис рядом с моим обращением в личном кабинете
- **Чтобы** убедиться, что офис был зафиксирован корректно, и отслеживать статус своей заявки в контексте

**Acceptance Criteria:**

- [ ] AC-2.1: В секции "Мои обращения" (`/dashboard`) — если у обращения указан офис, рядом с типом и статусом отображается плашка с текстом формата "Корп. N, эт. K, оф. NNN".
- [ ] AC-2.2: Плашка стилистически отличается от плашек типа (Ошибка/Предложение) и статуса — нейтральный цвет (zinc), меньший шрифт, отдельный визуальный блок.
- [ ] AC-2.3: Если офис не указан — плашка не отображается совсем. Остальная раскладка карточки не сдвигается.
- [ ] AC-2.4: Данные для отображения берутся из связанной модели `Office` (корпус, этаж, номер) — без хранения дублирующего текста.
- [ ] AC-2.5: Существующие обращения без `officeId` (NULL) отображаются как раньше — без изменений.

**Приоритет:** Must have  
**Обоснование:** Без этой story пользователь не получает обратного подтверждения, что его офис зафиксирован. Доверие к форме снижается.

---

### US-3: Офис автора в административном дашборде

- **Как** менеджер или администратор, обрабатывающий обращения в `/admin/feedback`
- **Я хочу** видеть офис автора как в списке обращений, так и на детальной странице
- **Чтобы** мгновенно понимать из какого офиса пришла жалоба и не тратить время на уточнение

**Acceptance Criteria:**

- [ ] AC-3.1: На детальной странице обращения (`/admin/feedback/[id]`) — если офис указан, он отображается в блоке с информацией об авторе (рядом с именем и email пользователя). Формат: "Корп. N, эт. K, оф. NNN".
- [ ] AC-3.2: Данные для отображения берутся из связанной модели `Office` через join, не из текстового поля.
- [ ] AC-3.3: Если офис не указан — строка "Офис" не отображается. Блок информации об авторе не меняет структуру.
- [ ] AC-3.4: API `GET /api/feedback/[id]` возвращает объект `office` с полями `id`, `number`, `floor`, `building` если привязан, или `null`.
- [ ] AC-3.5: Существующие обращения без привязки (`officeId = NULL`) обрабатываются штатно.

**Приоритет:** Must have  
**Обоснование:** Это основной потребитель данных — менеджер и admin. Без этой story данные в базе есть, но пользы от них нет.

## Архитектурные ограничения (не реализуем, но должны учесть)

Автоподстановка офиса из активного `RentalContract` пользователя — не входит в данную итерацию. Однако:

- Поле `officeId` в `FeedbackItem` должно быть nullable FK на `Office.id` — чтобы в будущем сервисный код мог автоматически подставить офис из контракта без новой миграции схемы.
- API autocomplete офисов должен быть реализован как отдельный read-only endpoint (или использовать существующий `/api/offices`), чтобы в будущем его можно было использовать в других местах (например, в форме заявки на аренду).

## Приоритет (MoSCoW)

| Категория | Обоснование |
|-----------|-------------|
| **Must have** | Все три story — одна фича без возможности частичного запуска. Данные нужно записывать (US-1) и показывать (US-2, US-3) одновременно. |

**Зависимости:**
- Схема БД: миграция Prisma — добавление `officeId String?` + `office Office?` relation в `FeedbackItem`. Решение о конкретной форме миграции принимает Architect.
- API: нужен endpoint для поиска офисов по тексту (autocomplete). Architect решает, создавать новый или использовать существующий.
- Компоненты: `feedback-button.tsx` расширяется — добавляется combobox-состояние, autocomplete-запрос, передача `officeId` в FormData.
- Страницы: `dashboard/page.tsx` (секция feedbackItems) и `admin/feedback/[id]/page.tsx` — добавление рендеринга плашки офиса.

## Вне скоупа

Следующее явно **не входит** в данную итерацию:

1. **Автоподстановка офиса из `RentalContract`** — не реализуем, но архитектура FK это позволит без новой миграции.
2. **Изменения в `RentalContract`, `Tenant`, `RentalDeal`, `RentalInquiry`** — не трогаем.
3. **Карта офисов** — не делаем.
4. **Email-уведомления при создании обращения** — не трогаем существующую логику.
5. **Аналитика "сколько обращений из какого офиса"** — не делаем.
6. **Telegram WebApp форма обращения** — не создаём и не модифицируем, если существует.
7. **Изменения формата `apiResponse` / `apiError`** — не меняем.
8. **Новые модули в `src/modules/`** — не создаём. Используем существующий модуль `feedback`.
9. **Изменение flow модерации/статусов обращений** — не трогаем.
10. **Список обращений в `/admin/feedback/page.tsx`** — офис в колонку списка не добавляем в этой итерации (только детальная страница и дашборд пользователя). Если Architect сочтёт добавление в список тривиальным — это его решение, но не требование PO.
