# PRD: Управленческие панели Барбекю Парк и Плей Парк + Защита БД + Тесты админки

**Дата**: 14 апреля 2026
**Автор**: Product Owner
**Статус**: Draft

---

## Проблема

### 1. Барбекю Парк — устаревший UI бронирований

Админка Плей Парка имеет продвинутый интерфейс: timeline-сетка (по ресурсам и часам), быстрое бронирование кликом по слоту, карточка деталей бронирования, маркер текущего времени, активные сессии, история с чеками, навигация по датам. Админка Барбекю Парка осталась на уровне MVP: плоская таблица бронирований без визуального расписания, без timeline, без быстрого бронирования через клик по слоту. Менеджер беседок не видит загрузку на день "одним взглядом" и тратит время на ручной поиск свободных слотов.

**Разрыв в компонентах:**
- Плей Парк: 16 компонентов (timeline-grid, quick-booking-popover, booking-detail-card, active-sessions-panel, shift-panel, session-bill-modal, booking-history-table, date-navigator и др.)
- Барбекю Парк: 3 компонента (admin-booking-form, booking-actions, resource-editor)

### 2. Нет полноценных управленческих панелей

Текущие страницы `/admin/gazebos` и `/admin/ps-park` — это единственные страницы каждого модуля. Нет разделения на подразделы: бронирования, ресурсы, аналитика, настройки. Менеджер видит всё на одной перегруженной странице. Нет аналитики: сколько бронирований за неделю, средняя загрузка, выручка, пиковые часы.

### 3. Отсутствие тестов админки

Ни один компонент из `src/components/admin/` не покрыт тестами. Компоненты Плей Парка (16 шт.) и Барбекю Парка (3 шт.) могут сломаться при любом рефакторинге без обнаружения. Серверные страницы (`page.tsx`) с прямыми запросами к Prisma тоже не тестируются.

### 4. База данных не защищена от потери данных

Prisma используется с `db push` (нет папки `migrations/`). Это означает:
- При изменении схемы Prisma может удалить таблицы/столбцы с данными.
- Нет истории миграций — невозможно откатиться.
- Нет бэкапов на уровне инфраструктуры (или их наличие не гарантировано).
- `DELETE` и `DROP` операции ничем не ограничены.
- Seed-данные могут затереть production-данные.

## Решение

### Часть 1: Timeline-сетка для Барбекю Парка

Адаптировать визуальную систему Плей Парка для беседок:
- Timeline-grid с ресурсами по строкам и часами по столбцам (часы работы беседок: 08:00-23:00)
- Быстрое бронирование кликом по свободному слоту
- Карточка деталей бронирования при клике на занятый слот
- Маркер текущего времени (красная линия)
- Навигация по датам (вперед/назад/сегодня)
- Цветовая дифференциация статусов (PENDING — пунктир/amber, CONFIRMED — зеленый, CHECKED_IN — ярко-зеленый с пульсом)

**Ключевое решение**: максимально переиспользовать компоненты Плей Парка, параметризовав их moduleSlug. Где невозможно — создать gazebos-аналоги с минимальными изменениями.

### Часть 2: Управленческие панели

Создать полноценные разделы управления для обоих модулей с подстраницами:

**Структура `/admin/gazebos/`:**
- `/admin/gazebos` — главная: timeline-расписание + ожидающие подтверждения (как сейчас у PS Park)
- `/admin/gazebos/resources` — CRUD ресурсов (беседок): название, вместимость, цена, фото, вкл/выкл
- `/admin/gazebos/bookings` — полная история бронирований с фильтрами (дата, статус, ресурс, клиент), пагинация
- `/admin/gazebos/analytics` — метрики: загрузка по дням/неделям, выручка, популярные беседки, пиковые часы, средний чек
- `/admin/gazebos/settings` — настройки модуля: часы работы, минимальная/максимальная длительность бронирования, правила отмены

**Структура `/admin/ps-park/`:**
- `/admin/ps-park` — главная: timeline + смена + активные сессии (как сейчас, без изменений)
- `/admin/ps-park/resources` — CRUD столов: название, вместимость, цена, фото, вкл/выкл
- `/admin/ps-park/bookings` — полная история с фильтрами + просмотр чеков завершенных сессий
- `/admin/ps-park/analytics` — метрики: загрузка, выручка (наличные/карта), среднее время сессии, популярные столы
- `/admin/ps-park/settings` — настройки: часы работы, правила смен, цены по умолчанию

**Навигация**: внутри каждого раздела — табы или подменю (не в главном sidebar, а внутри страницы модуля).

### Часть 3: Тесты админки

Покрыть тестами ключевые компоненты и страницы админки:

**Приоритет 1 (Must Have):**
- Timeline-grid (рендеринг, клик по слоту, позиционирование бронирований, маркер текущего времени)
- Booking-actions (подтверждение, отмена, смена статуса)
- Admin-booking-form (валидация, отправка, обработка ошибок)
- Resource-editor / Table-editor (CRUD операции)
- Sidebar (рендеринг по ролям, навигация, badge-counts)

**Приоритет 2 (Should Have):**
- Quick-booking-popover (форма, валидация времени)
- Active-sessions-panel (рендеринг сессий, обновление)
- Shift-panel (открытие/закрытие смены)
- Booking-history-table (рендеринг, пагинация)
- Date-navigator (навигация по датам)

**Приоритет 3 (Could Have):**
- Session-bill-modal (расчет чека)
- Booking-detail-card (отображение деталей)
- Complete-session-button, Extend-session-button

**Подход к тестированию:**
- Vitest + React Testing Library
- Моки для fetch-запросов (no real API calls)
- Моки для `next/navigation` (useRouter, usePathname)
- Моки для Prisma (серверные компоненты тестируем через route handler тесты)
- Snapshot-тесты для стабильных UI-компонентов

### Часть 4: Защита базы данных

Многоуровневая защита production-данных:

**Уровень 1: Миграции вместо db push**
- Перейти с `prisma db push` на `prisma migrate` (создать первую baseline-миграцию)
- Все изменения схемы только через `prisma migrate dev` (dev) и `prisma migrate deploy` (prod)
- CI проверяет, что нет "грязных" изменений схемы без миграции

**Уровень 2: Soft Delete**
- Все модели с данными получают поле `deletedAt DateTime?`
- Вместо `DELETE` — `UPDATE SET deletedAt = now()`
- Prisma middleware для автоматической фильтрации `WHERE deletedAt IS NULL`
- Восстановление удалённых записей через админку (SUPERADMIN)

**Уровень 3: Автоматические бэкапы**
- Ежедневный pg_dump в отдельное хранилище (cron на VPS)
- Хранение последних 30 дневных + 12 месячных бэкапов
- Скрипт восстановления из бэкапа (задокументировать)
- Проверка бэкапа: еженедельный тест-восстановление на staging

**Уровень 4: Ограничения на уровне БД**
- PostgreSQL: `REVOKE DELETE ON ALL TABLES FROM app_user` — приложение физически не может удалять строки
- Отдельный DB-юзер для миграций (с полными правами) и для приложения (без DELETE/DROP)
- `CREATE RULE` или trigger, запрещающий DELETE на критичных таблицах (Booking, Order, RentalContract, Tenant)

**Уровень 5: Защита миграций**
- Линтер миграций: запрет `DROP TABLE`, `DROP COLUMN`, `TRUNCATE` в migration SQL
- Review обязателен для любого файла в `prisma/migrations/`
- Pre-deploy: автоматическая проверка, что миграция не содержит деструктивных операций

**Уровень 6: Мониторинг**
- Алерт в Telegram при `DELETE` / `TRUNCATE` через PostgreSQL audit log
- Алерт при падении размера БД более чем на 10% (признак массового удаления)
- Дашборд: количество записей по таблицам, тренд за неделю

## Целевая аудитория

| Персона | Как затрагивает |
|---------|----------------|
| **MANAGER (Барбекю Парк)** | Получает визуальное расписание, быстрое бронирование, аналитику — работа становится эффективнее |
| **MANAGER (Плей Парк)** | Получает структурированные подразделы, аналитику, настройки — меньше скролла, больше фокуса |
| **SUPERADMIN** | Уверенность в сохранности данных, возможность восстановления, аудит удалений |
| **Разработчик** | Тесты дают уверенность при рефакторинге, миграции дают контроль над схемой |

## User Stories (с Acceptance Criteria)

### US-1: Timeline-расписание для Барбекю Парка

**Как** менеджер Барбекю Парка,
**я хочу** видеть расписание беседок в виде timeline-сетки (ресурсы по строкам, часы по столбцам),
**чтобы** одним взглядом оценить загрузку на день и быстро находить свободные слоты.

**Acceptance Criteria:**
- [ ] Timeline-grid отображает все активные беседки по строкам, часы 08:00-22:00 по столбцам
- [ ] Бронирования отображаются цветными блоками: PENDING (amber/пунктир), CONFIRMED (зеленый), CHECKED_IN (ярко-зеленый с пульсом)
- [ ] Красная вертикальная линия показывает текущее время (обновляется каждую минуту)
- [ ] Навигация по датам: кнопки "вчера / сегодня / завтра" + выбор произвольной даты
- [ ] При клике на свободный слот открывается попап быстрого бронирования (имя, телефон, время начала/конца, кол-во гостей)
- [ ] При клике на бронирование открывается карточка деталей (клиент, телефон, статус, действия: подтвердить/отменить/check-in)
- [ ] Загрузка расписания по API: `GET /api/gazebos/timeline?date=YYYY-MM-DD`

### US-2: Раздел "Управление ресурсами" (Барбекю Парк)

**Как** менеджер Барбекю Парка,
**я хочу** управлять списком беседок на отдельной странице,
**чтобы** не мешать это с расписанием и иметь удобный CRUD.

**Acceptance Criteria:**
- [ ] Страница `/admin/gazebos/resources` отображает таблицу всех беседок
- [ ] Для каждой беседки: название, вместимость, цена/час, статус (активна/отключена)
- [ ] Кнопка "Редактировать" открывает модалку/форму с полями: название, описание, вместимость, цена, фото
- [ ] Кнопка "Добавить беседку" создает новый ресурс
- [ ] Переключатель "Активна/Отключена" — деактивированная беседка не показывается в публичном бронировании
- [ ] Невозможно удалить беседку с существующими бронированиями (soft delete)

### US-3: Раздел "Управление ресурсами" (Плей Парк)

**Как** менеджер Плей Парка,
**я хочу** управлять столами на отдельной странице,
**чтобы** CRUD столов не был спрятан в collapsible-секции на главной странице.

**Acceptance Criteria:**
- [ ] Страница `/admin/ps-park/resources` отображает таблицу всех столов
- [ ] Для каждого стола: название, вместимость (игроков), цена/час, статус
- [ ] Редактирование, добавление, деактивация аналогично Барбекю Парку
- [ ] Текущий `TableEditor` переиспользуется

### US-4: Полная история бронирований с фильтрами

**Как** менеджер модуля (Барбекю или Плей Парк),
**я хочу** видеть полную историю бронирований с возможностью фильтрации,
**чтобы** находить конкретные бронирования и анализировать данные.

**Acceptance Criteria:**
- [ ] Страницы `/admin/gazebos/bookings` и `/admin/ps-park/bookings`
- [ ] Таблица: дата, время, ресурс, клиент, телефон, статус, сумма
- [ ] Фильтры: по дате (от-до), по статусу (multi-select), по ресурсу, по клиенту (поиск)
- [ ] Пагинация: 20 записей на страницу
- [ ] Сортировка по дате (по умолчанию — новые сверху)
- [ ] Для завершенных бронирований PS Park — ссылка на чек
- [ ] API: `GET /api/{module}/bookings?page=1&status=COMPLETED&dateFrom=...&dateTo=...`

### US-5: Аналитика модуля

**Как** менеджер или SUPERADMIN,
**я хочу** видеть аналитику по модулю на отдельной странице,
**чтобы** понимать эффективность бизнеса и принимать решения.

**Acceptance Criteria:**
- [ ] Страницы `/admin/gazebos/analytics` и `/admin/ps-park/analytics`
- [ ] Карточки KPI: бронирований за сегодня / неделю / месяц, выручка за день / неделю / месяц, средний чек, средняя загрузка (%)
- [ ] График загрузки по дням (последние 30 дней) — bar chart
- [ ] Топ-3 популярных ресурса (по кол-ву бронирований)
- [ ] Пиковые часы (heatmap или bar chart по часам)
- [ ] Для PS Park дополнительно: разбивка выручки наличные/карта, среднее время сессии
- [ ] Данные загружаются через API: `GET /api/{module}/analytics?period=month`

### US-6: Настройки модуля

**Как** менеджер или SUPERADMIN,
**я хочу** настраивать параметры модуля без помощи разработчика,
**чтобы** оперативно менять бизнес-правила.

**Acceptance Criteria:**
- [ ] Страницы `/admin/gazebos/settings` и `/admin/ps-park/settings`
- [ ] Настройки: часы работы (открытие/закрытие), минимальная длительность бронирования, максимальная длительность, минимальное время до бронирования (часы)
- [ ] Изменения сохраняются в `Module.config` (JSONB)
- [ ] Валидация: часы работы корректны, min < max длительности
- [ ] Логирование изменений в AuditLog

### US-7: Внутримодульная навигация

**Как** менеджер,
**я хочу** переключаться между подразделами модуля через табы или подменю,
**чтобы** быстро находить нужный раздел.

**Acceptance Criteria:**
- [ ] На каждой странице модуля отображаются табы: "Расписание" | "Ресурсы" | "Бронирования" | "Аналитика" | "Настройки"
- [ ] Активный таб подсвечен
- [ ] Табы не дублируют главный sidebar — sidebar содержит только "Барбекю Парк" и "Плей Парк"
- [ ] Компонент табов — переиспользуемый между модулями

### US-8: Тесты компонентов админки

**Как** разработчик,
**я хочу** иметь тесты на ключевые компоненты админки,
**чтобы** рефакторинг не ломал существующий функционал.

**Acceptance Criteria:**
- [ ] Покрытие тестами: timeline-grid, booking-actions, admin-booking-form, resource-editor/table-editor, sidebar
- [ ] Каждый тест проверяет: рендеринг без ошибок, ключевые взаимодействия (клик, ввод), корректную передачу пропсов
- [ ] Моки: fetch, useRouter, usePathname, Prisma
- [ ] `npm test` проходит зеленым
- [ ] Тесты добавлены в CI pipeline

### US-9: Миграции вместо db push

**Как** SUPERADMIN / разработчик,
**я хочу** чтобы все изменения схемы БД проходили через миграции,
**чтобы** иметь историю изменений и возможность отката.

**Acceptance Criteria:**
- [ ] Создана baseline-миграция из текущей схемы
- [ ] `prisma migrate deploy` используется в production вместо `db push`
- [ ] CI проверяет: `prisma migrate diff` не показывает расхождений
- [ ] Документация: процесс создания миграции для разработчиков

### US-10: Soft Delete для критичных данных

**Как** SUPERADMIN,
**я хочу** чтобы удаленные данные можно было восстановить,
**чтобы** случайное удаление не привело к потере бизнес-данных.

**Acceptance Criteria:**
- [ ] Модели `Booking`, `Order`, `OrderItem`, `RentalContract`, `Tenant`, `Resource`, `MenuItem` получают поле `deletedAt`
- [ ] Все операции удаления заменены на `UPDATE SET deletedAt = now()`
- [ ] Prisma middleware автоматически фильтрует `deletedAt IS NULL` во всех запросах
- [ ] Админка SUPERADMIN: раздел "Корзина" — просмотр и восстановление удаленных записей
- [ ] API: `POST /api/admin/restore/{entity}/{id}` — восстановление записи (только SUPERADMIN)

### US-11: Автоматические бэкапы PostgreSQL

**Как** SUPERADMIN,
**я хочу** автоматические ежедневные бэкапы базы данных,
**чтобы** при катастрофическом сбое данные можно было восстановить.

**Acceptance Criteria:**
- [ ] Cron-задача: `pg_dump` ежедневно в 03:00 MSK
- [ ] Хранение: 30 дневных + 12 месячных бэкапов
- [ ] Бэкапы хранятся на отдельном диске/сервере (не на том же VPS)
- [ ] Скрипт `scripts/restore-backup.sh` для восстановления
- [ ] Telegram-алерт при неудачном бэкапе
- [ ] Еженедельный тест: восстановление последнего бэкапа на staging и проверка целостности

### US-12: Ограничение DELETE на уровне БД

**Как** SUPERADMIN,
**я хочу** чтобы приложение физически не могло удалять строки из критичных таблиц,
**чтобы** даже баг в коде или SQL-инъекция не привели к потере данных.

**Acceptance Criteria:**
- [ ] Создан отдельный PostgreSQL-юзер `delovoy_app` (для приложения) без прав DELETE на критичных таблицах
- [ ] Юзер `delovoy_migrate` (для миграций) сохраняет полные права
- [ ] Trigger/Rule на таблицах `Booking`, `Order`, `RentalContract`, `Tenant`: запрет DELETE, логирование попытки
- [ ] Документация: какие юзеры для чего, как переключать

| Метрика | Текущее значение | Целевое значение |
|---------|-----------------|-----------------|
| Время поиска свободного слота (Барбекю) | ~30 сек (скролл таблицы) | <3 сек (визуальный timeline) |
| Количество компонентов админки с тестами | 0 | 10+ (все Must Have) |
| Покрытие тестами админ-компонентов | 0% | >60% строк ключевых компонентов |
| Время восстановления БД из бэкапа | неопределено (нет бэкапов) | <30 мин |
| Потеря данных при ошибочном удалении | необратимо | 100% восстановимо (soft delete + бэкап) |
| Кликов для создания бронирования менеджером (Барбекю) | 8+ (форма) | 3-4 (клик по слоту + заполнение попапа) |
| Подстраницы в управленческих панелях | 1 на модуль | 5 на модуль |

## Вне скоупа

- Изменение публичных (B2C) страниц бронирования — только админка
- Модуль "Кафе" — его управленческая панель не входит в эту задачу
- Модуль "Аренда" — уже имеет свою структуру
- Telegram-бот — интеграция с новыми фичами будет отдельной задачей
- Платежная система — оплата онлайн не входит в скоуп
- Миграция исторических данных — работаем с текущими данными как baseline
- Шардинг/репликация PostgreSQL — избыточно на текущем масштабе
- Row Level Security (RLS) PostgreSQL — можно добавить позже поверх текущей RBAC

### Soft Delete — реализация через Prisma middleware

```typescript
// src/lib/soft-delete.ts
prisma.$use(async (params, next) => {
  if (SOFT_DELETE_MODELS.includes(params.model)) {
    if (params.action === 'delete') {
      params.action = 'update';
      params.args.data = { deletedAt: new Date() };
    }
    if (params.action === 'findMany' || params.action === 'findFirst') {
      params.args.where = { ...params.args.where, deletedAt: null };
    }
  }
  return next(params);
});
```
