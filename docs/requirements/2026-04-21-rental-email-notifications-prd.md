# PRD: Email-уведомления и рассылки в модуле Аренда

**Дата:** 2026-04-21
**RUN_ID:** `2026-04-21-rental-email-notifications`
**Модуль:** rental
**Статус:** Draft → Architect

---

## Проблема

Менеджер по аренде сегодня отслеживает платёжную дисциплину арендаторов вручную: проверяет, кто заплатил, звонит напоминающим звонком, фиксирует в Excel. При портфеле из 20–40 арендаторов это занимает несколько часов в месяц и порождает ошибки: кто-то получает звонок дважды, кто-то — вообще не получает, просрочки обнаруживаются поздно.

Параллельно отсутствует инструмент для разовых коммуникаций с арендаторами: сообщить о плановых работах, отправить документы, поздравить с продлением договора — всё это делается через личный телефон менеджера без истории и контроля.

Бизнес теряет деньги на поздно выявленных просрочках и тратит время менеджера на рутину, которую можно автоматизировать.

---

## Решение

Встроить в модуль Аренда систему email-коммуникаций с арендаторами, включающую:

1. **Ручную отправку** — менеджер пишет письмо одному арендатору или группе прямо из интерфейса, не выходя из платформы.
2. **Автоматические напоминания об оплате** — три этапа: за N дней до срока, в день платежа (если не оплачено), через 5 дней после срока (эскалация на менеджера).
3. **Редактор шаблонов** — менеджер правит текст письма в GUI без привлечения разработчиков.
4. **Настройки расписания** — пороги дней задаются в GUI, а не в коде.
5. **История отправок** — полный лог всех писем с датой, статусом и причиной ошибки.

Инфраструктура для отправки (SMTP через Yandex, `sendTransactionalEmail`) и системы уведомлений (scheduler, queue, NotificationLog) уже существуют — новый функционал встраивается поверх них.

---

## Целевая аудитория

| Персона | Роль | Получает ценность |
|---------|------|------------------|
| Менеджер аренды | MANAGER (rental) | Инструмент коммуникации с арендаторами, автоматизация напоминаний |
| Суперадмин / директор | SUPERADMIN | Контроль платёжной дисциплины, редактирование шаблонов |
| Арендатор | Tenant (получатель email) | Своевременные уведомления об оплате, официальная коммуникация |

---

## Что уже реализовано (не дублируем)

- `sendTransactionalEmail({ to, subject, html, text })` — готовый транспорт, nodemailer + Yandex SMTP.
- `emailAdapter` — канал в системе уведомлений.
- `scheduler.ts` — планировщик, вызывается каждые 5 минут, уже обрабатывает `processContractExpiryAlerts`.
- `NotificationLog` (Prisma) — лог уведомлений с полем `eventType` и `status`, используется для идемпотентности.
- `Tenant.email` + `Tenant.emailsExtra` (Json) — адреса для рассылки уже хранятся.
- `AuditLog` и `SystemEvent` — готовы к логированию.

Чего **нет** в текущей схеме и потребует добавления:
- Трекинг платежей (нет `paymentDate`, нет истории оплат) — см. раздел «Модель данных».
- Шаблоны писем, редактируемые в GUI.
- Настройки расписания рассылок.
- История email-отправок отдельно от общего `NotificationLog`.
- Механизм задач/эскалации для менеджера.

## User Stories

### US-1: Ручная отправка письма арендатору

- **Как** менеджер по аренде
- **Я хочу** написать письмо одному или нескольким арендаторам прямо из карточки арендатора или из общего списка
- **Чтобы** официально уведомить об изменениях, плановых работах, документах — и иметь историю этой коммуникации

**Acceptance Criteria:**

- [ ] AC-1.1: На странице арендатора (и на странице договора) есть кнопка «Отправить письмо», доступная SUPERADMIN и MANAGER (rental).
- [ ] AC-1.2: По клику открывается форма с полями: получатель (pre-filled из email арендатора), тема, тело письма (HTML-редактор или plain text с переключателем), возможность выбрать шаблон из списка.
- [ ] AC-1.3: Если у арендатора заполнено поле `emailsExtra`, все адреса доступны для выбора в форме (менеджер может добавить/убрать адреса из списка).
- [ ] AC-1.4: Если email у арендатора не указан, кнопка «Отправить письмо» неактивна и показывает подсказку «Добавьте email в карточку арендатора».
- [ ] AC-1.5: После отправки менеджер видит уведомление «Письмо отправлено» или сообщение об ошибке с причиной.
- [ ] AC-1.6: Каждая отправка сохраняется в истории писем (EmailLog) с: датой, темой, получателями, статусом (SENT / FAILED), ссылкой на арендатора/договор, именем отправителя.
- [ ] AC-1.7: Поле «От кого» в письме всегда равно `buh@delovoy-park.ru` — менеджер не может изменить отправителя.
- [ ] AC-1.8: Действие логируется в `AuditLog` (`action: "email.sent"`, `entity: "Tenant"`, `entityId: tenantId`).

**Приоритет:** Must have (P0) — базовая коммуникация с арендаторами.

### US-2: Массовая рассылка группе арендаторов

- **Как** менеджер по аренде
- **Я хочу** выбрать несколько арендаторов из списка и отправить им одно письмо
- **Чтобы** быстро разослать информацию всем арендаторам конкретного корпуса или этажа

**Acceptance Criteria:**

- [ ] AC-2.1: В списке арендаторов доступны чекбоксы для выбора нескольких записей.
- [ ] AC-2.2: При выборе 2+ арендаторов появляется панель с действием «Отправить письмо выбранным».
- [ ] AC-2.3: Письма отправляются каждому арендатору отдельным сообщением (не «всем в копии»), переменные шаблона подставляются индивидуально.
- [ ] AC-2.4: Арендаторы без email автоматически исключаются из рассылки; менеджер видит предупреждение «N арендаторов без email — будут пропущены».
- [ ] AC-2.5: В EmailLog создаётся запись для каждого получателя отдельно.
- [ ] AC-2.6: Массовая рассылка доступна только SUPERADMIN и MANAGER (rental). USER доступа к этому функционалу не имеет.

**Приоритет:** Should have (P1) — значительно ускоряет работу при корпусных рассылках, но базовая ручная отправка (US-1) покрывает 80% кейсов.

### US-3: Автоматическое напоминание об оплате — T-N дней (первое)

- **Как** менеджер по аренде
- **Я хочу** чтобы система автоматически напомнила арендатору об оплате за N дней до срока
- **Чтобы** у арендатора было время подготовить платёж, а менеджер не тратил время на звонки

**Acceptance Criteria:**

- [ ] AC-3.1: Система ежедневно (в 09:00 МСК) проверяет, у каких активных договоров (ACTIVE, EXPIRING) дата платежа текущего месяца наступит через N дней (по умолчанию N=5, настраивается).
- [ ] AC-3.2: Арендатор получает письмо на все email из карточки (`email` + `emailsExtra`) с темой и телом согласно шаблону `rental.payment_reminder_pre`.
- [ ] AC-3.3: Письмо содержит подстановки: имя арендатора, номер договора, сумму платежа, дату платежа, номер офиса, реквизиты для оплаты (из конфигурации модуля).
- [ ] AC-3.4: Система отправляет напоминание не более одного раза на один период оплаты (идемпотентность через `RentalPayment.firstReminderSentAt`).
- [ ] AC-3.5: Если SMTP недоступен, попытка логируется в `SystemEvent` с уровнем ERROR, повтор выполняется при следующем прогоне планировщика.
- [ ] AC-3.6: Письма не отправляются арендаторам с договорами в статусе DRAFT, EXPIRED или TERMINATED.

**Приоритет:** Must have (P0) — ключевой элемент автоматизации.

---

### US-4: Автоматическое напоминание в день платежа — T=0 (второе)

- **Как** менеджер по аренде
- **Я хочу** чтобы в день платежа система автоматически проверила, поступил ли платёж, и при необходимости отправила второе напоминание арендатору
- **Чтобы** снизить количество просрочек без дополнительных звонков

**Acceptance Criteria:**

- [ ] AC-4.1: В день наступления срока платежа (T=0) система проверяет статус записи `RentalPayment` текущего периода.
- [ ] AC-4.2: Если платёж не помечен как поступивший (`paidAt IS NULL`), арендатор получает второе письмо по шаблону `rental.payment_reminder_due`.
- [ ] AC-4.3: Если платёж уже помечен как поступивший, второе письмо не отправляется.
- [ ] AC-4.4: Одно письмо на один период оплаты (идемпотентность через `RentalPayment.dueDateReminderSentAt`).
- [ ] AC-4.5: В `EmailLog` фиксируется запись с типом `PAYMENT_DUE_REMINDER`.

**Приоритет:** Must have (P0).

---

### US-5: Эскалация на менеджера — T+5 дней (третий этап)

- **Как** менеджер по аренде
- **Я хочу** получить задачу «дойти к арендатору лично», если через 5 дней после срока платёж так и не поступил
- **Чтобы** не пропустить проблемного арендатора и вовремя принять меры

**Acceptance Criteria:**

- [ ] AC-5.1: На 5-й день после срока платежа, если `RentalPayment.paidAt IS NULL`, система создаёт запись `ManagerTask` с типом `OVERDUE_PAYMENT`, привязанную к договору и менеджеру модуля rental.
- [ ] AC-5.2: Одновременно система отправляет менеджеру alert в Telegram (через существующий `enqueueNotification`) и пишет запись в `SystemEvent` с уровнем WARNING.
- [ ] AC-5.3: Задача отображается в интерфейсе менеджера на странице «Мои задачи» в разделе Аренды: арендатор, сумма, просрочка в днях, кнопки «Оплата поступила» / «Отложить на 3 дня».
- [ ] AC-5.4: Менеджер может закрыть задачу, выбрав одну из причин: «Оплата поступила», «Арендатор предупредил — платит позже», «Договор расторгается».
- [ ] AC-5.5: Закрытие задачи через «Оплата поступила» автоматически проставляет `RentalPayment.paidAt = now()`.
- [ ] AC-5.6: Задача создаётся не более одного раза на один период (идемпотентность через проверку наличия открытой `ManagerTask` с тем же `contractId` и `periodMonth`).
- [ ] AC-5.7: После T+5 арендатору больше не приходят автоматические письма по этой просрочке — только личный визит менеджера.

**Приоритет:** Must have (P0) — без эскалации система теряет смысл на третьем шаге.

### US-6: Фиксация факта поступления платежа

- **Как** менеджер по аренде
- **Я хочу** отметить в системе, что платёж от арендатора поступил за конкретный месяц
- **Чтобы** автоматика не отправляла лишних напоминаний и статус договора отражал реальность

**Acceptance Criteria:**

- [ ] AC-6.1: На странице договора есть раздел «Платежи» с таблицей: период (месяц/год), сумма, дата платежа, дата поступления, статус.
- [ ] AC-6.2: Менеджер может отметить платёж как поступивший, нажав «Отметить оплаченным» и указав дату фактической оплаты.
- [ ] AC-6.3: После отметки система прекращает отправлять напоминания по этому периоду.
- [ ] AC-6.4: При создании нового договора система автоматически генерирует записи `RentalPayment` на весь срок договора (ежемесячно, от `startDate` до `endDate`).
- [ ] AC-6.5: При продлении договора (`renewContract`) записи `RentalPayment` дополняются новыми периодами.
- [ ] AC-6.6: Менеджер может скорректировать сумму конкретного периода (например, если была скидка), с обязательным указанием причины (записывается в `RentalChangeLog`).
- [ ] AC-6.7: Действие «отметить оплаченным» логируется в `AuditLog`.

**Приоритет:** Must have (P0) — без трекинга платежей невозможна идемпотентность напоминаний.

### US-7: Редактор шаблонов писем

- **Как** SUPERADMIN
- **Я хочу** изменить текст автоматического письма-напоминания об оплате прямо в интерфейсе
- **Чтобы** не ждать разработчика при изменении тона коммуникации или добавлении новых реквизитов

**Acceptance Criteria:**

- [ ] AC-7.1: В разделе «Аренда → Настройки» есть подраздел «Шаблоны писем» со списком всех шаблонов модуля.
- [ ] AC-7.2: Каждый шаблон содержит: ключ (системный, нередактируемый), название (отображаемое), тему письма, тело (HTML и/или plain text), статус (активен/неактивен).
- [ ] AC-7.3: Редактор шаблонов поддерживает переменные-плейсхолдеры в формате `{{переменная}}`. Список доступных переменных отображается рядом с редактором (см. раздел «Переменные шаблонов»).
- [ ] AC-7.4: При сохранении система валидирует, что все переменные в шаблоне входят в разрешённый список — нельзя сохранить шаблон с неизвестным плейсхолдером.
- [ ] AC-7.5: Доступен предпросмотр письма с тестовыми данными до сохранения.
- [ ] AC-7.6: Изменение шаблона логируется в `AuditLog` (`action: "email_template.updated"`).
- [ ] AC-7.7: Редактирование шаблонов доступно только SUPERADMIN. MANAGER (rental) может просматривать шаблоны, но не изменять.
- [ ] AC-7.8: При деактивации шаблона соответствующий тип автоматических писем не отправляется. Система пишет WARNING в `SystemEvent`.

**Приоритет:** Must have (P0) — без редактора шаблонов при любом изменении текста нужен разработчик.

- [ ] AC-8.5: Изменение настроек логируется в `AuditLog`.
- [ ] AC-8.6: Доступно только SUPERADMIN.

- **Я хочу** видеть, какие письма были отправлены конкретному арендатору и когда
- **Чтобы** не задавать дважды один вопрос, и убедиться, что напоминание действительно дошло

- [ ] AC-9.2: В общем разделе «Аренда → Письма» доступен сводный лог по всем арендаторам с фильтром по: арендатору, типу письма, статусу, дате.

## Модель данных

### Рекомендуемая модель трекинга платежей: `RentalPayment`

**Выбор:** полноценная модель `RentalPayment` против лёгкого поля `lastPaidUntil` на контракте.

Рекомендую `RentalPayment` — она необходима для:
- идемпотентности напоминаний (один раз за период);
- истории оплат на странице договора (AC-6.1);
- будущей интеграции с бухгалтерией (не «когда последний раз платили», а «все факты оплат»).

Поле `lastPaidUntil` — технический костыль, который придётся переписывать при первом же запросе бухгалтерии.

```
RentalPayment {
  id                      String   @id
  contractId              String
  contract                RentalContract
  periodMonth             Int              // 1..12
  periodYear              Int              // YYYY
  dueDate                 DateTime         // ожидаемая дата оплаты
  amount                  Decimal          // сумма за период (может отличаться от monthlyRate при скидке)
  currency                String           // RUB
  paidAt                  DateTime?        // факт оплаты, null = не оплачен
  firstReminderSentAt     DateTime?        // отправлено T-N напоминание
  dueDateReminderSentAt   DateTime?        // отправлено T=0 напоминание
  escalatedAt             DateTime?        // создана задача менеджера T+5
  notes                   String?          // причина корректировки суммы
  createdAt               DateTime         @default(now())
  updatedAt               DateTime         @updatedAt

  @@unique([contractId, periodYear, periodMonth])
  @@index([dueDate, paidAt])
}
```

### Хранение шаблонов писем: `EmailTemplate`

```
EmailTemplate {
  id          String   @id
  moduleSlug  String              // "rental"
  key         String   @unique    // "rental.payment_reminder_pre", "rental.payment_reminder_due", "rental.manual"
  name        String              // "Напоминание за N дней"
  subject     String              // тема письма (поддерживает {{переменные}})
  bodyHtml    String   @db.Text  // HTML-тело (поддерживает {{переменные}})
  bodyText    String?  @db.Text  // plain-text версия (опционально)
  variables   Json                // ["tenantName", "contractNumber", ...] — разрешённые плейсхолдеры
  isActive    Boolean  @default(true)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

### Настройки расписания рассылок: `RentalNotificationSettings`

```
RentalNotificationSettings {
  id                    String   @id
  preReminderDays       Int      @default(5)   // T-N
  escalationDaysAfter   Int      @default(5)   // T+M
  autoSendEnabled       Boolean  @default(true)
  updatedAt             DateTime @updatedAt
  updatedById           String?
}
```

Альтернатива — хранить в `Module.config` JSON для модуля "rental". Architect выбирает. Отдельная таблица предпочтительнее для type-safety и валидации.

### История отправок: `EmailLog`

```

### Задачи менеджера: `ManagerTask`

**Рекомендация по эскалации: вариант C (оба канала).**

- `ManagerTask` — экран в интерфейсе необходим: менеджер должен видеть открытые задачи, закрыть их с причиной, видеть историю.
- Telegram-алерт — дополнительный канал доставки. Без него менеджер может не зайти в интерфейс вовремя.
- `SystemEvent` (WARNING) — для мониторинга суперадмином.

Без экрана задач Telegram-алерт теряется в истории, менеджер не может отметить «проработано». Без Telegram менеджер может не проверить интерфейс в нужный день. Оба канала вместе закрывают проблему.

```
ManagerTask {

  resolution    String?         // причина закрытия
  deferUntil    DateTime?       // если «отложить»
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@unique([contractId, periodYear, periodMonth])  // одна задача на период
  @@index([status, assignedToId])

  OPEN

## Edge Cases

| Ситуация | Поведение |
|---|---|
| У арендатора нет email | Кнопка «Отправить письмо» неактивна, авторассылка пропускает арендатора, в `SystemEvent` (WARNING) пишется «Tenant без email: {tenantId}» |
| Невалидный email (не проходит формат) | Попытка отправки фиксируется в EmailLog со статусом FAILED и ошибкой SMTP |
| SMTP недоступен | EmailLog: FAILED + error. SystemEvent: ERROR. Планировщик повторит при следующем прогоне. Ручная отправка возвращает ошибку менеджеру |
| Договор в статусе DRAFT | Авторассылки не запускаются — только активные договоры (ACTIVE, EXPIRING) |
| Договор в статусе TERMINATED или EXPIRED | Авторассылки не запускаются. Ручная отправка доступна (менеджер может написать, например, о возврате депозита) |
| Несколько email у арендатора | Письмо отправляется на каждый адрес из `email` + `emailsExtra` в отдельных SMTP-сообщениях. В EmailLog — одна запись с `to: [addr1, addr2]` |
| Шаблон деактивирован | Соответствующий тип автоматических писем не отправляется. SystemEvent: WARNING. Менеджер видит предупреждение в интерфейсе |
| Платёж отмечен вручную до T=0 | Напоминание T=0 не отправляется (проверка `paidAt IS NOT NULL`) |
| Эскалация уже создана (повтор) | Дублирующая `ManagerTask` не создаётся — уникальный индекс `[contractId, periodYear, periodMonth]` |
| Дата платежа приходится на выходной/праздник | Система работает по фактической дате (`dueDate`), без сдвига. Бизнес-логика сдвига дат — вне скоупа |

## Приоритет (MoSCoW)

**Категория: Must have**

Обоснование: платёжная дисциплина арендаторов — прямой доход бизнеса. Одна просрочка на 50–100 тыс. руб./мес. перекрывает затраты на разработку всей фичи. Ручная рассылка устраняет слабое место «информация передаётся через личный телефон менеджера».

**Зависимости:**
- Существующий SMTP-транспорт (`channels/email.ts`) — готов.
- Планировщик (`scheduler.ts`) — готов, требует расширения.
- `NotificationLog` — готов, EmailLog является его специализацией для rental.
- `RentalContract` — готов, требует связи с `RentalPayment`.

**Деплой:** предлагается в Phase 5.0 параллельно с production-запуском, так как прямо влияет на операционный процесс менеджера.

## Вне скоупа

Следующее намеренно исключается из данной итерации:

- **SMS-уведомления арендаторам** — отдельный канал, требует интеграции с SMS-шлюзом.
- **Telegram-бот для арендаторов** — отправка алерта менеджеру через Telegram входит в скоуп, но создание диалога с арендатором — нет.
- **Онлайн-оплата аренды** — биллинговая интеграция (эквайринг, 1С, Сбербанк-бизнес) — отдельная фича.
- **Трекинг открытия письма** (open rate, click rate) — требует pixel-трекинга или ESP (SendGrid/Mailgun).
- **Отписка от рассылок (unsubscribe)** — нужна для B2C, в B2B-контексте аренды регулируется договором.
- **Автоматическая выписка счетов** — генерация PDF-счёта и его прикрепление к письму.
- **Интеграция с бухгалтерией** (1С, МойСклад) — синхронизация статусов оплат из внешней системы.
- **Изменение даты платежа** для конкретного арендатора — менеджер может скорректировать `dueDate` в `RentalPayment`, но UI для индивидуальных дат платежа не входит в MVP.
- **Push-уведомления** (браузер, мобильное приложение) для арендаторов.
