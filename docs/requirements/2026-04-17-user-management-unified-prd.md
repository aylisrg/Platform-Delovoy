# PRD: Unified User Management — Единое управление пользователями

**Дата**: 17 апреля 2026
**Автор**: Product Owner
**RUN_ID**: `2026-04-17-user-management-unified`
**Статус**: Draft

---

## Проблема

Платформа «Деловой» накопила несколько несвязанных точек управления пользователями, которые дублируют данные и создают путаницу как у администраторов, так и у клиентов.

### Текущее состояние и боли

**Боль 1 — Три отдельных раздела в админке вместо одного**

В сайдбаре есть три самостоятельных раздела: «Пользователи» (`/admin/users`), «Клиенты» (`/admin/clients`) и «Telegram» (`/admin/telegram`). Это три разных таблицы, каждая из которых показывает одних и тех же людей с разных углов. Администратор вынужден переключаться между вкладками, чтобы понять полную картину по конкретному человеку. Нет единого окна типа «Кто этот человек, как он к нам зашёл, чем пользуется, как с ним связаться».

- В «Пользователях» — управление ролями, сброс паролей, настройки notify-releases.
- В «Клиентах» — история бронирований, трат, активность по модулям. Здесь же спрятана функция слияния аккаунтов — суперадмин не найдёт её, не зная, где искать.
- В «Telegram» — список пользователей с привязанными Telegram-аккаунтами, настройки бота. Частичное дублирование с обоими разделами выше.

**Боль 2 — Страница входа показывает неработающие кнопки**

На экране авторизации (`/auth/signin`) отображаются кнопки «Войти через Google» и «Войти через VK». Ни один из этих провайдеров не настроен в production — клиент нажимает и получает ошибку. Это разрушает доверие. При этом WhatsApp (OTP через телефон) — рабочий способ входа, но он спрятан мелкой ссылкой внизу страницы, его почти никто не видит.

Telegram-виджет требует разрешения `data-request-access=write`, что у некоторых пользователей вызывает всплывающее окно Telegram с запросом номера телефона — это путает: «зачем Telegram просит мой телефон для входа на сайт?».

**Боль 3 — Пользователь не может управлять своими привязанными аккаунтами**

В личном кабинете (`/dashboard`) есть блок «Мои контакты» — там можно добавить email и телефон. Но:
- Нет информации о том, через какой провайдер создан аккаунт (Telegram, Яндекс, email).
- Нет кнопки «Отвязать» — если пользователь привязал email с ошибкой, он не может его убрать.
- Яндекс OAuth не отображается в списке привязанных аккаунтов, хотя технически уже поддерживается.
- VK-провайдер присутствует в данных (`vkId` в User, `vk` в ContactsCard), но VK убирается из продукта — эта строка в ЛК вводит в заблуждение.

**Боль 4 — Функция слияния аккаунтов скрыта и неочевидна**

Слияние (merge) двух пользователей в один — мощная операция, нужная суперадмину, когда один реальный человек создал несколько аккаунтов (например, зашёл через Telegram и отдельно через email). Механизм уже реализован в коде (`/api/admin/clients/merge`), но спрятан на детальной странице клиента в разделе «Клиенты». Нет никакого обнаружения: суперадмин не знает о его существовании, пока специально не зайдёт в карточку клиента.

### Персоны, которые страдают

- **SUPERADMIN** — тратит время на переключение между разделами, не может быстро найти полную карточку любого человека.
- **USER (клиент)** — видит неработающие кнопки входа, не может управлять привязанными каналами, не знает, какой аккаунт к нему «прикреплён».

---

## Решение

Четыре связанных улучшения, которые в сумме дают единую и понятную систему управления пользователями:

1. **Унификация в единый раздел «Пользователи»** в админке — все три раздела сливаются в один с вкладками.
2. **Чистая страница входа** — убираем неработающие провайдеры, выводим WhatsApp на видное место, убираем пугающий запрос прав у Telegram.
3. **Полный профиль в личном кабинете** — показываем все привязанные аккаунты, добавляем возможность отвязать каждый, фиксируем Яндекс, убираем VK.
4. **Явная точка слияния аккаунтов для суперадмина** — операция merge становится видимой из единого реестра, а не спрятанной на карточке клиента.

---

## Целевая аудитория

- **US-1, US-2**: SUPERADMIN — управляющий парка, архитектор системы.
- **US-3**: USER — клиент B2C-сервисов парка.
- **US-4**: USER — клиент, возвращающийся через другое устройство или другой канал.

## User Stories

### US-1: Единый реестр пользователей в админке

- **Как** суперадмин
- **Я хочу** видеть всех пользователей (и администраторов, и клиентов) в одном разделе с вкладками по типу
- **Чтобы** не переключаться между тремя разделами и быстро находить любого человека по одному поиску

**Acceptance Criteria:**

- [ ] AC-1.1: Раздел «Пользователи» в сайдбаре заменяет три отдельных раздела: «Пользователи», «Клиенты», «Telegram». Разделы «Клиенты» и «Telegram» удаляются из навигации.
- [ ] AC-1.2: Новый единый раздел содержит три вкладки: «Все» (весь реестр), «Команда» (SUPERADMIN + MANAGER), «Клиенты» (USER с хотя бы одним бронированием или заказом).
- [ ] AC-1.3: В карточке каждого пользователя в реестре отображаются: имя, контакты (email, телефон), все привязанные аккаунты (Telegram, Яндекс, email — значки с бейджами «подключён»/«не подключён»), роль, дата регистрации.
- [ ] AC-1.4: Поиск работает единым образом по всем трём вкладкам: по имени, email, телефону, Telegram username.
- [ ] AC-1.5: Для пользователей с ролью SUPERADMIN или MANAGER из вкладки «Команда» доступны: изменение роли, управление правами доступа к разделам, сброс пароля — те же действия, что были в старом разделе «Пользователи».
- [ ] AC-1.6: Для пользователей с ролью USER из вкладки «Клиенты» отображается сводка активности: количество бронирований, заказов, суммарные траты, последняя активность — те же данные, что были в старом разделе «Клиенты».
- [ ] AC-1.7: Функционал настроек Telegram-бота (chat ID, токен, карта уведомлений) переезжает во вкладку «Система» внутри раздела «Мониторинг». Раздел «Telegram» из навигации удаляется.
- [ ] AC-1.8: Все изменения ролей и прав логируются в AuditLog (действие, кто изменил, на что изменил).
- [ ] AC-1.9: Страница работает при 500+ пользователях без деградации: список пагинирован по 50 записей, поиск выдаёт результат менее чем за 1 секунду.

**Приоритет**: Must have — данные разрозненны сейчас, при росте числа пользователей администрирование становится невозможным.

### US-2: Слияние дублирующих аккаунтов — явная функция суперадмина

- **Как** суперадмин
- **Я хочу** иметь возможность объединить два аккаунта одного реального человека в один прямо из единого реестра
- **Чтобы** данные о клиенте были полными и не дублировались, а история бронирований и заказов была под одним аккаунтом

**Acceptance Criteria:**

- [ ] AC-2.1: В карточке любого пользователя в реестре есть кнопка «Объединить с другим аккаунтом», видимая только для SUPERADMIN.
- [ ] AC-2.2: При клике открывается диалог, в котором суперадмин ищет второй аккаунт по имени, email или телефону — поиск работает по тому же API, что и основной поиск в реестре.
- [ ] AC-2.3: Перед подтверждением система показывает preview: какой аккаунт остаётся (primary), какой удаляется (secondary), что именно переносится (бронирования: N, заказы: M), и явный список конфликтов (если у обоих есть email или Telegram — показывает, чей приоритет).
- [ ] AC-2.4: Подтверждение требует ввода текста «ОБЪЕДИНИТЬ» — защита от случайного клика.
- [ ] AC-2.5: После слияния: все бронирования, заказы и audit-логи secondary переносятся на primary. Secondary-аккаунт удаляется. Пользователь видит сообщение «Объединение выполнено».
- [ ] AC-2.6: Операция слияния логируется в AuditLog с полным описанием: кто выполнил, какие два аккаунта слиты, сколько записей перенесено.
- [ ] AC-2.7: Функция доступна только SUPERADMIN — MANAGER не видит кнопку «Объединить».
- [ ] AC-2.8: Нельзя объединить аккаунт сам с собой — система возвращает понятную ошибку.

**Приоритет**: Should have — логика уже реализована в `/api/admin/clients/merge`, нужно только переместить точку входа в видимое место. Объём изменений минимален.

### US-3: Чистая страница входа без неработающих провайдеров

- **Как** клиент парка, который впервые заходит на сайт
- **Я хочу** видеть только те способы входа, которые реально работают, и выбирать между ними без путаницы
- **Чтобы** войти с первой попытки и не думать, почему кнопка не работает

**Acceptance Criteria:**

- [ ] AC-3.1: Со страницы входа убраны кнопки «Войти через Google» и «Войти через VK».
- [ ] AC-3.2: На странице входа явно представлены четыре способа: Telegram (виджет, первостепенный), Яндекс (кнопка OAuth), WhatsApp (кнопка, открывает форму с номером телефона), Email / magic link (кнопка или ссылка).
- [ ] AC-3.3: WhatsApp находится на одном уровне видимости с Яндексом — не скрыт мелкой ссылкой внизу. Порядок: Telegram → Яндекс → WhatsApp → Email.
- [ ] AC-3.4: Telegram-виджет настроен без параметра `data-request-access=write` — виджет не запрашивает номер телефона пользователя при входе.
- [ ] AC-3.5: Если Telegram-виджет недоступен (NEXT_PUBLIC_TELEGRAM_BOT_NAME не задан), на его месте отображается заглушка «Telegram-вход временно недоступен» — без ошибок в консоли.
- [ ] AC-3.6: Все четыре способа входа работают в production на деле, а не только в коде — проверяется smoke-тестом в CI после деплоя.
- [ ] AC-3.7: Страница входа корректно отображается на мобильном экране шириной 375px — кнопки не обрезаются, порядок сохраняется.

**Приоритет**: Must have — неработающие кнопки разрушают первое впечатление при запуске в production (Phase 5.0 — 17 апреля 2026).

---

### US-4: Полный профиль с управлением привязанными аккаунтами

- **Как** пользователь, у которого уже есть аккаунт
- **Я хочу** видеть все свои привязанные аккаунты (Telegram, Яндекс, email, WhatsApp) и управлять ими — добавлять и отвязывать
- **Чтобы** контролировать, через какие каналы я получаю уведомления, и исправлять ошибки (например, привязал не тот email)

**Acceptance Criteria:**

- [ ] AC-4.1: В личном кабинете (`/dashboard`) в блоке «Мои контакты» отображаются все четыре канала: Telegram, Яндекс, email, WhatsApp (телефон). Для каждого — статус «Подключён» / «Не подключён».
- [ ] AC-4.2: Строка VK убрана из блока «Мои контакты» полностью — ни как «Подключён», ни как «Не подключён».
- [ ] AC-4.3: Для подключённых каналов есть кнопка «Отвязать». При нажатии — запрос подтверждения («Вы уверены? Вы не сможете войти через этот канал»). После подтверждения канал отвязывается, если у пользователя остаётся хотя бы один другой способ входа.
- [ ] AC-4.4: Система не позволяет отвязать последний привязанный канал — показывает сообщение «Это единственный способ входа. Привяжите другой канал перед отвязкой».
- [ ] AC-4.5: Для Telegram: если аккаунт не привязан, отображается кнопка «Привязать Telegram». По нажатию — открывается диалог с QR-кодом и deep-link `t.me/BotName?start=link_TOKEN`. Токен одноразовый, TTL 15 минут.
- [ ] AC-4.6: Для Яндекс: если аккаунт не привязан, кнопка «Привязать Яндекс» запускает OAuth-флоу — пользователь авторизуется в Яндексе и возвращается на страницу профиля с привязанным аккаунтом.
- [ ] AC-4.7: Для email: существующий флоу работает (ввод email → письмо со ссылкой → подтверждение). Поведение не меняется.
- [ ] AC-4.8: Для WhatsApp: существующий флоу работает (ввод номера → OTP в WhatsApp → подтверждение). Поведение не меняется.
- [ ] AC-4.9: В блоке «Настройки уведомлений» выбор «Предпочтительный канал» отображает только те каналы, которые реально привязаны. Нельзя выбрать Telegram, если Telegram не привязан.
- [ ] AC-4.10: Все изменения (привязка, отвязка) сохраняются немедленно и видны без перезагрузки страницы.

**Приоритет**: Should have — функция отвязки и отображения Яндекс-аккаунта добавляет важную ценность. Email и WhatsApp уже работают, задача — достроить до полноты.

## Метрики успеха

### US-3 (страница входа)
- Базовое: 100% пользователей видят неработающие кнопки Google/VK.
- Целевое: 0% пользователей видят нерабочие кнопки. Конверсия регистраций новых пользователей растёт на 10% за 2 недели после запуска (замеряется через AuditLog событий входа).

### US-1 (единый реестр)
- Базовое: администратор переключается между 3 разделами, чтобы найти полные данные по одному человеку.
- Целевое: все операции с пользователем выполняются из одного раздела. Количество жалоб (через /admin/feedback) на «не могу найти пользователя» = 0 в течение первого месяца.

### US-4 (профиль в ЛК)
- Базовое: 0% пользователей привязали более одного канала связи.
- Целевое: 15% активных пользователей (совершивших хотя бы одно бронирование) привязали 2+ канала через 4 недели после релиза.

### US-2 (merge)
- Базовое: суперадмин тратит 5+ минут, чтобы найти функцию слияния.
- Целевое: суперадмин находит кнопку «Объединить» менее чем за 30 секунд после открытия реестра.

## Вне скоупа

- Добавление новых OAuth-провайдеров (кроме Яндекс, которые уже есть).
- Самостоятельная регистрация аккаунта по паролю через публичный интерфейс — создание аккаунтов с паролем остаётся операцией суперадмина.
- Telegram Mini App (webapp) — управление профилем через бота остаётся в отдельном скоупе.
- Двухфакторная аутентификация (2FA) — не планируется в этой итерации.
- Массовое управление пользователями (bulk-операции, экспорт CSV) — следующая итерация.
- Удаление аккаунта пользователем самостоятельно — юридически сложный вопрос, откладывается.
- WhatsApp как OAuth-провайдер для входа — останется OTP через телефон, не OAuth.
- История входов (IP, устройства) — мониторинг безопасности, отдельная задача.

## Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| При удалении VK-провайдера из auth.ts потеряются существующие VK-аккаунты | Низкая (VK клиентов нет или единицы) | Средний | Перед удалением: запрос к БД `SELECT COUNT(*) FROM "Account" WHERE provider = 'vk'`. Если > 0 — сначала уведомить пользователей. |
