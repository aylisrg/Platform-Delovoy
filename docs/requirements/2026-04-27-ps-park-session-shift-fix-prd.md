# PRD: Bugfix — Завершение сессии и выручка смены PS Park

**RUN_ID**: `2026-04-27-ps-park-session-shift-fix`
**Тип**: Bugfix PRD (существующий модуль)
**Модуль**: `ps-park`
**Дата**: 2026-04-27
**Статус**: Готов к передаче Architect

---

## Проблема

Менеджеры PS Park не могут корректно завершить рабочую смену:

1. **Кнопки "Завершить" и "Отменить" сессию не реагируют** — при нажатии ничего не происходит, статус сессии остаётся без изменений, ошибка пользователю не показывается.

2. **Отчёт выручки при закрытии смены показывает 0 ₽** — даже если за смену были завершены несколько платных сессий.

Следствия: менеджер не может закрыть смену с корректными цифрами, выручка в `ShiftHandover` записывается нулевой, финансовая аналитика искажена.

---

## Корневые причины (диагностика PO на основе кода)

### Баг #1 — Завершение/отмена сессии

Обнаружено при изучении `src/modules/ps-park/service.ts` и `src/modules/booking/state-machine.ts`:

- State machine (`TRANSITIONS`) разрешает `CONFIRMED → COMPLETED` и `CONFIRMED → CANCELLED` для MANAGER — логика в сервисе корректна.
- Однако UI Admin-панели, по всей видимости, передаёт неверный `status` в теле запроса, либо запрос к `PATCH /api/ps-park/bookings/:id` не отправляется вовсе (обработчик клика не привязан, или `fetch` не вызывается из-за необработанной ошибки).
- `AuditLog` в route handler (`src/app/api/ps-park/bookings/[id]/route.ts`) логирует только `booking.status_change`, но не специализированные действия `session.complete` / `session.cancel` — это отдельный дефицит.
- При успешном завершении (`COMPLETED`) сервис создаёт `FinancialTransaction` и пишет `bill` в `metadata.bill`. При провале запроса транзакция не создаётся.

### Баг #2 — Выручка смены 0 ₽

- `getDayReport` агрегирует `FinancialTransaction WHERE type = SESSION_PAYMENT AND createdAt BETWEEN dayStart AND dayEnd`.
- Транзакция создаётся **только** при успешном переходе в `COMPLETED` через `updateBookingStatus`.
- Если баг #1 блокирует завершение сессий — транзакций нет — выручка 0 ₽. Баг #2 является **прямым следствием бага #1**.
- Дополнительный риск: `getDayReport` фильтрует по `createdAt` транзакции, а не по `date` сессии. Если сессия началась вчера и завершена сегодня, выручка будет в отчёте сегодняшнего дня — это корректное поведение, документировать явно.

### Отсутствующие функции (часть задачи)

- Авто-завершение по истечении `endTime` — нет роута `/api/ps-park/auto-complete`, нет cron/scheduled task.
- Пост-фактум позиции к `COMPLETED` сессии — `addItemsToBooking` блокирует любой статус кроме `PENDING`/`CONFIRMED`.

---

## Решение

Починить три связанных вещи:

1. **UI-слой** — убедиться, что кнопки завершения/отмены отправляют корректный `PATCH`-запрос с правильным `status`, обрабатывают ответ (в том числе 4xx ошибки) и отображают пользователю понятный фидбек.

2. **API/Сервис** — добавить специализированные `AuditLog`-записи (`session.complete`, `session.cancel`) для мутаций завершения сессии; убедиться в idempotency (повторный запрос на уже завершённую сессию возвращает 409 с понятной ошибкой).

3. **Авто-завершение и пост-фактум позиции** — добавить API endpoint для авто-завершения истёкших сессий и разрешить добавление позиций к `COMPLETED` сессии с пересчётом суммы.

**SUPERADMIN** — полный доступ ко всем операциям модуля.

## User Stories

### US-1: Завершение активной сессии

- **Как** менеджер PS Park
- **Я хочу** нажать кнопку "Завершить сессию" и получить подтверждение с суммой к оплате
- **Чтобы** клиент заплатил корректную сумму, а выручка попала в отчёт смены

**Acceptance Criteria:**

- [ ] AC-1.1: Менеджер нажимает "Завершить" на активной сессии (`CONFIRMED` или `CHECKED_IN`) — открывается диалог с расчётом суммы (`pricePerHour × billedHours + items`).
- [ ] AC-1.2: Менеджер подтверждает завершение — сессия переходит в статус `COMPLETED`, в ответе присутствует `totalAmount`.
- [ ] AC-1.3: После завершения создаётся запись `FinancialTransaction` с `type = SESSION_PAYMENT`, `totalAmount`, `cashAmount`, `cardAmount`.
- [ ] AC-1.4: В `AuditLog` создаётся запись с `action = "session.complete"`, `entity = "Booking"`, `entityId = <bookingId>`.
- [ ] AC-1.5: Если сессия завершена раньше запланированного `endTime` — к оплате выставляется только фактически отыгранное время (с округлением вверх до 15 минут).
- [ ] AC-1.6: Если сессия уже в статусе `COMPLETED` или `CANCELLED` — запрос на завершение возвращает HTTP 409 с кодом `INVALID_STATUS_TRANSITION` и читаемым сообщением пользователю в UI.
- [ ] AC-1.7: Ошибки API (4xx, 5xx) отображаются пользователю в UI — не теряются молча.

### US-2: Отмена сессии менеджером

- **Как** менеджер PS Park
- **Я хочу** отменить бронирование/сессию через кнопку в панели
- **Чтобы** освободить стол и зафиксировать отмену в системе

**Acceptance Criteria:**

- [ ] AC-2.1: Менеджер нажимает "Отменить" на сессии в статусе `PENDING`, `CONFIRMED` или `CHECKED_IN` — появляется диалог подтверждения с опциональным полем причины.
- [ ] AC-2.2: После подтверждения сессия переходит в `CANCELLED`, в ответе API присутствует `status: "CANCELLED"`.
- [ ] AC-2.3: Отменённая сессия не попадает в выручку смены (не создаётся `FinancialTransaction`).
- [ ] AC-2.4: В `AuditLog` создаётся запись `action = "session.cancel"` с `entityId` и причиной отмены (если указана).
- [ ] AC-2.5: Если сессия уже `CANCELLED` или `COMPLETED` — возвращается HTTP 409, UI показывает ошибку.
- [ ] AC-2.6: Если у отменённой подтверждённой сессии были товарные позиции — остатки возвращаются на склад (уже реализовано в сервисе, нужно проверить что UI-путь вызывает нужную ветку).

### US-3: Корректная выручка при закрытии смены

- **Как** менеджер PS Park
- **Я хочу** видеть реальную сумму выручки при закрытии смены
- **Чтобы** сдать кассу с корректными данными и не пересчитывать вручную

**Acceptance Criteria:**

- [ ] AC-3.1: После завершения 3 сессий на разных столах (каждая через US-1) — при закрытии смены `ShiftHandover.cashTotal + ShiftHandover.cardTotal` равно сумме всех `FinancialTransaction.totalAmount` за эту дату.
- [ ] AC-3.2: Отменённые сессии не учитываются в выручке смены.
- [ ] AC-3.3: Закрытие смены без единой завершённой сессии не вызывает ошибку — `cashTotal = 0`, `cardTotal = 0`.
- [ ] AC-3.4: `getDayReport` возвращает список транзакций с разбивкой по наличным и безналичным.
- [ ] AC-3.5: В UI при закрытии смены отображается: суммарная выручка, количество сессий, разбивка наличные/безналичные.

### US-4: Авто-завершение истёкших сессий

- **Как** система (cron/scheduled job)
- **Я хочу** автоматически завершать сессии, у которых истёк `endTime`
- **Чтобы** менеджер не упустил сессию и система зафиксировала оплату даже без ручного действия

**Acceptance Criteria:**

- [ ] AC-4.1: При вызове `POST /api/ps-park/auto-complete` (или внутренней cron-функции) — все `CONFIRMED` и `CHECKED_IN` сессии с `endTime < now` переводятся в `COMPLETED`.
- [ ] AC-4.2: Для каждой авто-завершённой сессии создаётся `FinancialTransaction` с суммой `pricePerHour × billedHours + itemsTotal`.
- [ ] AC-4.3: В `AuditLog` создаётся запись `action = "session.auto_complete"` с указанием `actor = "CRON"` в метаданных.
- [ ] AC-4.4: Повторный вызов авто-завершения на уже завершённых сессиях не создаёт дублирующих транзакций (idempotency).
- [ ] AC-4.5: Endpoint `/api/ps-park/auto-complete` защищён — вызывается только с cron-токеном (`CRON_SECRET` из env) или внутри серверного cron-хука, не публичен без авторизации.

### US-5: Пост-фактум добавление позиций к завершённой сессии

- **Как** менеджер PS Park
- **Я хочу** добавить забытые позиции (напиток, доп.час, штраф) к уже завершённой сессии
- **Чтобы** клиент заплатил полную сумму и выручка смены была точной

**Acceptance Criteria:**

- [ ] AC-5.1: Менеджер может добавить позиции к сессии в статусе `COMPLETED` через `POST /api/ps-park/bookings/:id/items` или существующий `PATCH`.
- [ ] AC-5.2: После добавления позиций `metadata.itemsTotal` пересчитывается, создаётся дополнительная `FinancialTransaction` с `type = ADJUSTMENT` на сумму добавленных позиций.
- [ ] AC-5.3: `getDayReport` включает `ADJUSTMENT` транзакции в суммарную выручку смены.
- [ ] AC-5.4: В `AuditLog` создаётся запись `action = "session.items_added_post_complete"` с перечнем добавленных позиций.
- [ ] AC-5.5: Добавление позиций к сессии старше 24 часов требует явного подтверждения (предупреждение в UI), но технически не блокируется.

## Технические ограничения (для Architect)

Список наблюдений из кода — не инструкция по реализации, а контекст для оценки:

- **State machine** в `src/modules/booking/state-machine.ts` разрешает `CONFIRMED → COMPLETED` для MANAGER. Переход `CHECKED_IN → COMPLETED` тоже разрешён. Сервисная логика в `updateBookingStatus` корректна — создаёт `FinancialTransaction` и `bill` в metadata.
- **`addItemsToBooking`** в `service.ts` блокирует статусы кроме `PENDING`/`CONFIRMED` — для US-5 нужно расширение этой функции или отдельный путь для `COMPLETED`.
- **`getDayReport`** уже агрегирует по `FinancialTransaction` — для US-5 достаточно добавить `ADJUSTMENT` в тип транзакции (уже есть в enum `FinancialTxType`).
- **Авто-завершение** требует нового endpoint и/или серверного cron. Endpoint должен быть защищён `CRON_SECRET`.
- **AuditLog** в route handler `/api/ps-park/bookings/[id]/route.ts` использует общий `booking.status_change` — нужно добавить специализированные `session.complete` / `session.cancel`.

## Вне скоупа

- Интеграция с платёжными системами (онлайн-оплата).
- Изменение state machine (добавление новых статусов).
- Изменение логики billing (округление до 15 минут) — уже реализовано и работает корректно.
- Рефакторинг модуля notifications для PS Park.
- Новые UI-компоненты за пределами форм завершения/отмены сессии.
- Экспорт отчётов (PDF/Excel) для смены.
- Логика штрафов при поздней отмене клиентом (`cancelBooking`) — отдельная функция, работает независимо.
- WhatsApp/SMS уведомления при завершении сессии.

## Edge Cases

| Сценарий | Ожидаемое поведение |
|----------|---------------------|
| Завершить уже завершённую сессию | HTTP 409, `INVALID_STATUS_TRANSITION`, сообщение в UI |
| Отменить уже отменённую сессию | HTTP 409, `INVALID_STATUS_TRANSITION`, сообщение в UI |
| Завершить сессию, которая ещё не начала | State machine: `CONFIRMED → COMPLETED` разрешено. Billing: `billedHours = 0` если `startTime > now`. Ожидаемо — менеджер должен видеть предупреждение в UI о нулевой сумме. |
| Закрыть смену без открытых сессий | `cashTotal = 0`, `cardTotal = 0`, HTTP 200 |
| Авто-завершение: сессия уже завершена вручную | Пропустить без ошибки (idempotency) |
| Авто-завершение: `pricePerHour = null` | `totalAmount = itemsTotal` (стол без цены), транзакция всё равно создаётся |
| Пост-фактум позиция к сессии > 24ч назад | Разрешить с предупреждением в UI |
