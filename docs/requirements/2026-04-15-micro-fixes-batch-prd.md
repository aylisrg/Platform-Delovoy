## 1. Контекст и проблема

Платформа "Деловой Парк" была запущена в продакшен 14 апреля 2026. После первого дня эксплуатации владелец парка (SUPERADMIN) собрал список замечаний и улучшений на основе реального использования. Это первый пакет пост-запусковых правок.

Характер правок:
- Фактические ошибки в контенте (расстояние до Москвы)
- Проблемы с кэшем у части пользователей
- Доработка UX публичной части (офисы, навигация)
- Улучшения админ-панели (клиенты, обратная связь, хелпер)
- Инфраструктурные задачи (бекап БД)
- Подготовка реальных данных аренды

Целевая аудитория правок: **все три персоны** (USER, MANAGER, SUPERADMIN).

### STORY-2: Публичная страница офисов — свободные офисы + кнопка заявки

**Приоритет:** Should Have
**Персона:** USER (потенциальный арендатор)
**Оценка:** M (4-8 часов)

**Как** потенциальный арендатор,
**я хочу** видеть свободные офисы с подробной информацией и отправить заявку на конкретные офисы,
**чтобы** быстро выбрать подходящее помещение и начать процесс аренды.

#### Текущее поведение
Публичная страница `/rental` (`src/app/(public)/rental/page.tsx`) уже показывает все офисы из БД с разбивкой по этажам. Есть форма заявки (`InquiryForm`) с выпадающим списком офисов. Однако:
- Нет кнопки "Отправить запрос" на каждой карточке офиса
- Нет возможности выбрать несколько офисов
- Нет предзаполнения формы при клике на карточку
- Успешное сообщение говорит "Заявка отправлена! Мы свяжемся с вами в ближайшее время" — владелец хочет уточнить "в рабочее время"

#### Acceptance Criteria
- [ ] На каждой карточке свободного офиса (status=AVAILABLE) есть кнопка "Отправить запрос"
- [ ] При клике на кнопку — страница скроллится к форме заявки внизу, и выбранный офис предзаполняется
- [ ] Пользователь может выбрать несколько офисов (мульти-селект вместо одиночного dropdown)
- [ ] Кнопка "Отправить запрос" на карточке добавляет офис к уже выбранным (а не заменяет)
- [ ] Сообщение об успехе: "Мы получили вашу заявку, свяжемся с вами в рабочее время"
- [ ] На занятых офисах (OCCUPIED, MAINTENANCE, RESERVED) кнопка не отображается
- [ ] Форма заявки корректно отправляет список выбранных офисов на бекенд

#### Технические заметки
- Файлы: `src/app/(public)/rental/page.tsx`, `src/components/public/rental/inquiry-form.tsx`
- `InquiryForm` сейчас принимает `officeId: string` — нужно поменять на массив `officeIds: string[]`
- API `POST /api/rental/inquiries` нужно проверить — поддерживает ли несколько офисов
- Scroll-to-form: `document.getElementById('inquiry-form')?.scrollIntoView({ behavior: 'smooth' })`

### STORY-3: Принудительный сброс кэша

**Приоритет:** Must Have
**Персона:** USER (посетитель со старым кэшем)
**Оценка:** S (1-2 часа)

**Как** владелец платформы,
**я хочу** чтобы все пользователи видели актуальную версию сайта,
**чтобы** не получать жалобы на устаревший контент.

#### Текущее поведение
Next.js 15 с `output: "standalone"` (файл `next.config.ts`). Нет явных настроек cache-control для статических ресурсов. Часть пользователей видит закэшированную версию (CDN, service worker, browser cache).

#### Acceptance Criteria
- [ ] Добавлены HTTP-заголовки для предотвращения кэширования HTML-страниц (`Cache-Control: no-cache, no-store, must-revalidate` для HTML; длительный кэш со `s-maxage` + immutable для статики с хэшем в имени)
- [ ] Если есть CDN/прокси (Nginx) — добавлена конфигурация cache-busting или purge
- [ ] Версионирование статики (Next.js buildId) работает корректно: при новом деплое пользователи получают свежие бандлы
- [ ] Проверить наличие service worker — если есть, удалить или обновить стратегию
- [ ] После деплоя этого пакета правок — старый контент не показывается

#### Технические заметки
- Добавить `headers()` в `next.config.ts` с правильной стратегией кэширования
- Проверить Nginx/reverse proxy конфигурацию на VPS (Timeweb)
- Next.js по умолчанию добавляет hash к бандлам `_next/static/` — это уже работает
- Основная проблема скорее всего в кэшировании HTML-страниц на уровне браузера или CDN

### STORY-8: Мерж клиентов

**Приоритет:** Could Have
**Персона:** SUPERADMIN
**Оценка:** L (8-16 часов)

**Как** суперадмин,
**я хочу** иметь возможность объединить два аккаунта одного клиента,
**чтобы** корректно агрегировать историю покупок, если человек зашёл через Telegram и через email как два разных пользователя.

#### Текущее поведение
Система не поддерживает объединение аккаунтов. Если один человек зарегистрировался через Telegram, а потом через email — это два разных User в БД.

#### Acceptance Criteria
- [ ] В профиле клиента (`/admin/clients/:id`) есть кнопка "Объединить с другим клиентом"
- [ ] При нажатии — поиск второго аккаунта по имени, email, телефону, Telegram ID
- [ ] Показ preview: что будет объединено (бронирования, заказы, контактные данные)
- [ ] Выбор "основного" аккаунта (в который сольются данные)
- [ ] После мержа:
  - Все bookings и orders переносятся на основной аккаунт
  - Все accounts (провайдеры) переносятся на основной аккаунт
  - Контактные данные (email, phone, telegramId, vkId) переносятся (если не конфликтуют)
  - "Поглощённый" аккаунт деактивируется (soft delete) или удаляется
- [ ] Операция логируется в AuditLog
- [ ] Подтверждение перед мержем ("Вы уверены? Это действие необратимо")

#### Технические заметки
- Нужен API `POST /api/admin/clients/merge` с body `{ primaryId, secondaryId }`
- Транзакция Prisma: перенос bookings, orders, accounts + удаление secondary
- Unique constraints: email, phone, telegramId, vkId — нужно обработать конфликты
- Начать с backend + простого UI; полный UX можно доработать позже

### STORY-11: Автоматический бекап базы данных

**Приоритет:** Must Have
**Персона:** SUPERADMIN
**Оценка:** M (4-8 часов)

**Как** владелец платформы,
**я хочу** чтобы база данных автоматически бекапилась раз в сутки,
**чтобы** не потерять данные при сбое.

#### Текущее поведение
Скрипт `scripts/backup-db.sh` уже существует и реализует:
- `pg_dump` с gzip-сжатием
- Ротацию (30 дней daily, 12 месяцев monthly)
- Telegram-алерт при ошибке

Однако:
- Скрипт НЕ подключён к cron на VPS
- Docker-compose НЕ содержит сервис для бекапов
- Volume `/backups/postgres` НЕ настроен

#### Acceptance Criteria
- [ ] Бекап БД выполняется автоматически каждый день в 03:00 MSK
- [ ] Бекапы хранятся минимум 30 дней (daily) + 12 месяцев (monthly)
- [ ] При ошибке бекапа — Telegram-алерт в чат администратора
- [ ] Бекап не влияет на производительность приложения (off-peak часы)
- [ ] Существует инструкция по восстановлению из бекапа (уже есть: `scripts/restore-backup.sh`)
- [ ] Бекап-решение не требует ручного вмешательства владельца

#### Технические заметки
**Вариант A (cron на VPS):**
```
0 3 * * * /root/platform-delovoy/scripts/backup-db.sh >> /var/log/backup.log 2>&1
```
**Вариант B (Docker-сервис):**
Добавить в `docker-compose.yml` сервис `backup`:
```yaml
backup:
  image: postgres:16-alpine
  volumes:
    - ./scripts/backup-db.sh:/backup.sh
    - backup_data:/backups/postgres
  entrypoint: crond -f
  depends_on:
    - postgres
```
Предпочтительный вариант: **A** (проще, backup-db.sh уже готов).
Нужно только: SSH на VPS, настроить crontab, создать директорию `/backups/postgres`.

### STORY-12: Хелпер-помощник в админке

**Приоритет:** Could Have
**Персона:** MANAGER / SUPERADMIN
**Оценка:** L (8-16 часов)

**Как** менеджер, впервые работающий с админ-панелью,
**я хочу** видеть контекстные подсказки в каждом разделе,
**чтобы** понимать, зачем нужен каждый раздел и как с ним работать, без обращения к владельцу.

#### Текущее поведение
В админке нет контекстных подсказок, онбординга или справки. Менеджеры вынуждены спрашивать владельца, что значат технические термины (например, "Округление слота минуты" в настройках Плей Парка).

#### Acceptance Criteria
- [ ] В каждом разделе админки (Dashboard, Барбекю Парк, Плей Парк, Кафе, Аренда, Склад, и т.д.) есть персонаж-помощник
- [ ] Помощник — плавающая иконка "?" (или аватар персонажа) в правом нижнем углу области контента
- [ ] При наведении/клике — всплывающее окно с текстом:
  - Зачем нужен этот раздел (1-2 предложения)
  - Что здесь можно делать
  - Объяснение неочевидных терминов (контекстно для раздела)
- [ ] Подсказки можно скрыть ("Больше не показывать"), настройка сохраняется в localStorage
- [ ] Контент подсказок хранится в конфигурации (не хардкод в каждом компоненте) — словарь подсказок по slug раздела
- [ ] Примеры подсказок:
  - **Плей Парк > Настройки > "Округление слота минуты"**: "Минимальный шаг бронирования в минутах. Например, 30 = можно бронировать только 9:00, 9:30, 10:00..."
  - **Склад**: "Учёт товаров для всех модулей парка. Приход товара, списание, просроченные позиции."
  - **Клиенты**: "Все люди, которые зарегистрировались на сайте. Даже если они ещё ничего не купили."

#### Технические заметки
- Создать компонент `AdminHelper` в `src/components/admin/admin-helper.tsx`
- Словарь подсказок: `src/lib/admin-hints.ts` — объект `Record<string, { title: string; description: string; tips?: string[] }>`
- Компонент получает `sectionSlug` через prop или через `usePathname()`

Владелец спрашивает: как указать, какие фото поставить на Барбекю Парк, а какие на Кафе.

#### Текущая реализация фотографий

Фотографии в проекте хранятся как **статические файлы** в директории `public/media/`. Привязка к разделам — через код:

1. **Лендинг (главная страница):**

   - Секция "Офисы" (`landing-delovoy-park.ru/components/offices-section.tsx`) — массив `offices` с полем `photo`

2. **Публичные страницы модулей:**
   - Фото беседок / столов PS — через поле `metadata` (JSON) в таблице `Resource` в БД. Можно задать через админку в разделе "Ресурсы"

#### Рекомендуемый порядок действий для владельца

1. **Подготовить фотографии:**
   - Сделать / выбрать фотографии для Барбекю Парка (беседки, мангал, территория) и для Кафе (интерьер, блюда, барная стойка)

   - Назвать файлы понятно: `bbq-besedka-1.webp`, `cafe-interior-1.webp`

2. **Передать разработчику** набор фотографий с пометкой:
   - "Эти фото — на Барбекю Парк (лендинг + страница бронирования)"
   - "Эти фото — на Кафе (лендинг + страница меню)"

3. **Разработчик:**
   - Загрузит фото в `public/media/` с оптимизацией (WebP, сжатие)
   - Обновит пути в `services-section.tsx`
   - Для ресурсов (беседок) — обновит через API или админку

4. **В будущем** (Phase 5.2+):
