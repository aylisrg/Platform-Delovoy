# PRD: Приход товара с кастомным названием + очистка тестовых данных

**Дата:** 2026-04-12
**Статус:** Draft
**Исходный запрос:** "Приход товара: кастомное название, количество, примечание (опционально), дата записи прихода. Убрать тестовые товары из базы данных."

---

## 1. Проблема

### 1.1 Приход товара
Текущий функционал (`POST /api/inventory/receive`) технически уже поддерживает приём товара по свободному названию (функция `receiveStockByName`): если SKU с таким именем уже есть — делает RECEIPT, если нет — создаёт новый SKU автоматически. Однако:

- **Нет UI.** Страница `/admin/inventory` не существует. Менеджер не может записать приход товара без прямого вызова API.
- **Дата прихода не задаётся вручную.** `InventoryTransaction.createdAt` проставляется автоматически в момент HTTP-запроса. Если менеджер записывает вчерашнюю поставку — дата будет неверной. В базе данных нет поля `receivedAt` (фактическая дата прихода, отдельная от технического `createdAt`).
- **Форма не валидируется на клиенте** — поле "название" ничем не ограничено, ошибки видны только после отправки.

### 1.2 Тестовые данные
База данных содержит тестовые SKU и транзакции, накопленные в ходе разработки (seed-скрипт, ручные тесты). Эти данные засоряют отчёты, аналитику остатков и список низких запасов.

---

## 2. Решение

### 2.1 Форма прихода товара (новый Admin UI)
Создать страницу `/admin/inventory` с формой "Записать приход". Поля:

| Поле | Обязательность | Описание |
|------|----------------|----------|
| Название товара | Обязательное | Свободный текст. Если товар с таким названием уже есть — приход добавляется к нему. Если нет — создаётся автоматически. |
| Количество | Обязательное | Целое число ≥ 1. |
| Примечание | Необязательное | До 500 символов. Например: "Поставщик Иванов, накладная №117". |
| Дата прихода | Обязательное | Дата (без времени). По умолчанию — сегодня. Можно изменить на прошедшую дату. |

После отправки: показать подтверждение с названием товара, количеством и новым остатком.

### 2.2 Хранение фактической даты прихода
Добавить поле `receivedAt DateTime` в модель `InventoryTransaction` (опциональное, для обратной совместимости с уже существующими записями). При новом приходе через форму — проставлять значение из формы. При приходе через API без этого поля — использовать `createdAt`.

### 2.3 Отображение транзакций прихода
На той же странице `/admin/inventory` — таблица последних приходов (тип RECEIPT и INITIAL) с колонками: дата прихода, название товара, количество, примечание, кто записал.

### 2.4 Очистка тестовых данных
Написать скрипт `scripts/clear-test-inventory.ts`, который удаляет из базы данных тестовые SKU и все связанные с ними транзакции. Скрипт запускается однократно вручную (`npm run clear-test-inventory`). Критерий отбора тестовых данных — задаётся списком в самом скрипте (по именам SKU или по дате создания до определённого порога).

---

## 3. Целевая аудитория

**Основная:** Менеджер склада / администратор, который записывает еженедельные поставки товаров (напитки, еда, расходники для PS Park и беседок).

**Вторичная:** SUPERADMIN — видит историю всех приходов, проверяет остатки.

---

## 4. User Stories и Acceptance Criteria

### US-1: Запись прихода товара через форму

**Как** менеджер,
**я хочу** открыть страницу и за 10 секунд записать приход,
**чтобы** остаток в системе совпадал с реальным количеством на складе.

**Acceptance Criteria:**

- [ ] AC-1.1: По адресу `/admin/inventory` отображается форма с полями: название, количество, примечание (необязательное), дата прихода.
- [ ] AC-1.2: Поле "дата прихода" по умолчанию содержит сегодняшнюю дату. Можно выбрать любую прошедшую дату (не будущую).
- [ ] AC-1.3: При пустом поле "название" или "количество" форма не отправляется, показывается inline-ошибка под полем.
- [ ] AC-1.4: После успешной отправки в верхней части страницы появляется зелёный баннер: "Приход записан: [Название товара], +[Количество] шт. Текущий остаток: [N] шт."
- [ ] AC-1.5: Форма очищается после успешной отправки и готова к следующему вводу.
- [ ] AC-1.6: Страница доступна только авторизованным пользователям с ролью MANAGER или SUPERADMIN. Гость получает редирект на `/auth/signin`.

---

### US-2: Кастомное название товара (новый vs существующий)

**Как** менеджер,
**я хочу** вводить название товара свободным текстом,
**чтобы** не выбирать из справочника и быстро записать любой товар, включая новый.

**Acceptance Criteria:**

- [ ] AC-2.1: Если товар с введённым названием (без учёта регистра) уже есть в базе — создаётся транзакция типа RECEIPT, остаток товара увеличивается на указанное количество. Новый SKU не создаётся.
- [ ] AC-2.2: Если товара с таким названием нет — автоматически создаётся новый SKU с категорией "Товары" и этот же приход регистрируется как начальный остаток (тип INITIAL).
- [ ] AC-2.3: Название товара ограничено 200 символами. При превышении — ошибка валидации на клиенте до отправки.

---

### US-3: Фактическая дата прихода

**Как** менеджер,
**я хочу** указать дату прихода (например, вчерашнее число),
**чтобы** история поставок отражала реальные события, а не момент ввода данных.

**Acceptance Criteria:**

- [ ] AC-3.1: В транзакции сохраняется поле `receivedAt` с датой, выбранной в форме.
- [ ] AC-3.2: В таблице истории приходов колонка "Дата прихода" отображает значение `receivedAt`, а не `createdAt`.
- [ ] AC-3.3: Нельзя выбрать дату в будущем (поле "дата прихода" ограничено today включительно).
- [ ] AC-3.4: Существующие транзакции без `receivedAt` не ломаются — в таблице в этой колонке показывается прочерк или fallback на `createdAt`.

---

### US-4: История приходов

**Как** менеджер,
**я хочу** видеть таблицу последних приходов,
**чтобы** проверить, что предыдущие поставки записаны корректно.

**Acceptance Criteria:**

- [ ] AC-4.1: На странице `/admin/inventory` ниже формы отображается таблица с колонками: Дата прихода, Название товара, Количество, Примечание, Кто записал.
- [ ] AC-4.2: Таблица показывает последние 50 записей типа RECEIPT и INITIAL, отсортированных по дате прихода (новые вверху).
- [ ] AC-4.3: Таблица загружается независимо от формы — ошибка загрузки таблицы не блокирует форму.

---

### US-5: Очистка тестовых данных

**Как** суперадмин,
**я хочу** удалить тестовые SKU и транзакции из базы данных,
**чтобы** аналитика остатков и история приходов отражали только реальные данные.

**Acceptance Criteria:**

- [ ] AC-5.1: Существует скрипт `scripts/clear-test-inventory.ts`, который можно запустить командой `npm run clear-test-inventory`.
- [ ] AC-5.2: Скрипт выводит в консоль список SKU и транзакций, которые будут удалены, и запрашивает подтверждение (ввод "yes") перед удалением.
- [ ] AC-5.3: После выполнения скрипт выводит: сколько SKU удалено, сколько транзакций удалено.
- [ ] AC-5.4: Скрипт удаляет только записи, соответствующие критериям тестовых данных (конкретные имена или дата создания до порогового значения). Производственные данные не затрагиваются.
- [ ] AC-5.5: Если тестовых данных нет — скрипт завершается с сообщением "Тестовых данных не найдено. Ничего не удалено."

## 6. Метрики успеха

| Метрика | Цель | Способ проверки |
|---------|------|-----------------|
| Время записи одного прихода | ≤ 30 секунд с момента открытия страницы | Ручной замер |
| Доля ошибок при записи прихода | 0 ошибок при корректных данных | QA-тест на happy path |
| Соответствие `receivedAt` введённой дате | 100% | Проверка в таблице транзакций |
| Тестовые данные в базе после запуска скрипта | 0 записей из тестового набора | SELECT из БД после скрипта |

---

## 7. Вне скоупа

- Управление справочником SKU (создание, редактирование, архивация) — отдельная задача.
- Ценообразование и стоимость прихода — не запрашивалось.
- Загрузка приходов из файла (Excel, CSV) — не запрашивалось.
- Уведомления при низком остатке — уже есть в системе, не требует изменений.
- Мобильная версия формы прихода — не запрашивалось.
- История операций типа SALE, ADJUSTMENT, RETURN — не в скоупе этой задачи.
- Удаление данных через UI (только скрипт, запускаемый вручную) — UI-инструмент очистки вне скоупа.

---

## 8. Технические ограничения и зависимости

- Схема базы данных требует миграции: добавить поле `receivedAt DateTime?` в модель `InventoryTransaction`.
- Страница `/admin/inventory` должна быть защищена существующим middleware (роли MANAGER, SUPERADMIN).
- API `POST /api/inventory/receive` расширяется полем `receivedAt` (опциональное, ISO date string `YYYY-MM-DD`). Обратная совместимость сохраняется — существующие вызовы без этого поля продолжают работать.
- Скрипт очистки работает только в среде с переменной `DATABASE_URL` и требует подтверждения перед деструктивной операцией.
