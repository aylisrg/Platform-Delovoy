# PRD: Система скидок при чекауте (Беседки + PlayStation Park)

**Дата:** 2026-04-17
**Автор:** Product Owner
**RUN_ID:** `2026-04-17-checkout-discount-system`
**Статус:** Ready for Architect

---

## Проблема

Сейчас менеджер не имеет никакого способа изменить итоговую сумму бронирования при оформлении. Любая корректировка (постоянный клиент, акция, извинение за неудобство, групповой визит) решается словесно или занесением "чужой" суммы в журнал вручную — без фиксации причины и без привязки к конкретному сотруднику.

Это создаёт три проблемы:
1. **Контроль**: владелец не знает, сколько и кому дали скидок и почему.
2. **Злоупотребление**: ничто не мешает менеджеру систематически занижать сумму без следа.
3. **Аналитика**: фактическая выручка в отчётах не совпадает с выручкой после скидок.

**Персоны:** MANAGER (применяет скидку при чекауте), SUPERADMIN (видит все скидки, настраивает лимиты).

---

## Решение

Добавить в форму чекаута (завершение бронирования) поле для ручной скидки с тремя обязательными параметрами:

- **Процент скидки** (от 1 до максимально разрешённого, задаётся в настройках модуля)
- **Причина** — обязательный выбор из справочника + возможность ввести свободный текст
- **Итоговая сумма** — пересчитывается в реальном времени и показывается менеджеру до подтверждения

Каждая применённая скидка пишется в `AuditLog` с полным снимком данных: кто дал, на какое бронирование, какой процент, причина, исходная и итоговая суммы.

---

## Что уже реализовано

- `Booking.metadata` (JSONB) содержит `basePrice`, `pricePerHour`, `totalPrice` — снимок цены на момент создания брони.
- PS Park: при переводе в `COMPLETED` рассчитывается `bill` в `metadata` и создаётся запись `FinancialTransaction.SESSION_PAYMENT` с `totalAmount`, `cashAmount`, `cardAmount`.
- Gazebos: при переводе в `COMPLETED` выручка берётся из `metadata.totalPrice`.
- Оба модуля: `updateBookingStatus` принимает `cashAmount` / `cardAmount` (PS Park), но скидка нигде не предусмотрена.
- `AuditLog` уже используется для мутаций.

| Персона | Роль в системе | Взаимодействие с фичей |

| Менеджер беседок / PS Park | MANAGER | Вводит скидку при чекауте |
| Владелец / управляющий | SUPERADMIN | Видит все скидки в отчёте, настраивает лимиты |

## User Stories

### US-1: Менеджер применяет скидку при завершении бронирования (COMPLETED)

- **Как** менеджер беседок или PlayStation Park
- **Я хочу** указать скидку (процент и причину) при переводе бронирования в статус COMPLETED
- **Чтобы** клиент заплатил скорректированную сумму, а я не вводил данные в обход системы

**Acceptance Criteria:**

- [ ] AC-1.1: При переводе бронирования в `COMPLETED` форма чекаута содержит необязательное поле "Скидка (%)" со значением по умолчанию 0.
- [ ] AC-1.2: Поле скидки принимает только целые числа от 1 до `maxDiscountPercent` (настраивается в конфиге модуля, по умолчанию 30).
- [ ] AC-1.3: Если введена скидка > 0, поле "Причина скидки" становится обязательным.
- [ ] AC-1.4: Список причин включает: "Постоянный клиент", "Корпоративная скидка", "Акция / промо", "Компенсация за неудобство", "Другое (уточните)". При выборе "Другое" показывается текстовое поле (обязательно заполнить, не менее 5 символов).
- [ ] AC-1.5: При вводе скидки итоговая сумма пересчитывается в реальном времени и показывается в форме до подтверждения. Формула: `discountedAmount = originalAmount × (1 − discountPercent / 100)`, округление до рубля.
- [ ] AC-1.6: Если менеджер попытается ввести скидку больше разрешённого максимума, API возвращает ошибку `DISCOUNT_EXCEEDS_LIMIT` с кодом 422.
- [ ] AC-1.7: Скидка 0% или отсутствие поля обрабатывается как "без скидки" — поведение не меняется от текущего.
- [ ] AC-1.8: После успешного чекаута в ответе API возвращается `discountPercent`, `discountAmount`, `discountReason`, `finalAmount`.

---

### US-2: Скидка фиксируется в аудит-логе

- **Как** суперадмин
- **Я хочу** видеть в логах каждую применённую скидку — кто, когда, сколько, причина
- **Чтобы** контролировать злоупотребления и понимать фактические потери на скидках

**Acceptance Criteria:**

- [ ] AC-2.1: При применении скидки > 0 создаётся запись `AuditLog` с `action = "booking.discount_applied"`, `entity = "Booking"`, `entityId = <bookingId>`.
- [ ] AC-2.2: `AuditLog.metadata` содержит: `{ managerId, managerName, originalAmount, discountPercent, discountAmount, finalAmount, discountReason, discountNote (если "Другое"), bookingId, moduleSlug, resourceName, clientName, appliedAt }`.
- [ ] AC-2.3: Снимок скидки сохраняется в `Booking.metadata` под ключом `discount: { percent, amount, reason, note, appliedBy, appliedAt }`.
- [ ] AC-2.4: `FinancialTransaction` для PS Park создаётся с `totalAmount = finalAmount` (после скидки), поле `metadata` содержит `originalAmount` и `discountPercent`.
- [ ] AC-2.5: Для beседок аналогично: `metadata.totalPrice` в Booking обновляется до `finalAmount`.

---

### US-3: Суперадмин видит сводный отчёт по скидкам

- **Как** суперадмин
- **Я хочу** видеть общую сумму и количество скидок по модулям за период
- **Чтобы** понимать, сколько выручки было отдано и кем

**Acceptance Criteria:**

- [ ] AC-3.1: В панели администратора, в разделе аудита, существует фильтр по `action = "booking.discount_applied"`.
- [ ] AC-3.2: Каждая запись отображает: имя менеджера, название ресурса, имя клиента, исходная сумма, процент скидки, итоговая сумма, дата.
- [ ] AC-3.3: Итого по выбранному периоду показывает: общее количество скидок, суммарный размер скидок в рублях.

---

### US-4: Суперадмин настраивает максимальный процент скидки

- **Как** суперадмин
- **Я хочу** задать максимально допустимый процент скидки отдельно для каждого модуля
- **Чтобы** менеджер не мог превысить разрешённый потолок

**Acceptance Criteria:**

- [ ] AC-4.1: В настройках модуля (Config GUI, `Module.config`) есть поле `maxDiscountPercent` (целое число 1–100, по умолчанию 30).
- [ ] AC-4.2: Изменение `maxDiscountPercent` вступает в силу немедленно для всех последующих чекаутов этого модуля.
- [ ] AC-4.3: Если `maxDiscountPercent` не задан в конфиге, система использует значение по умолчанию 30%.
- [ ] AC-4.4: Изменение настройки логируется в `AuditLog` с `action = "module.config_updated"`.

## Ограничения и бизнес-правила

| Правило | Значение |
|---------|----------|
| Кто может давать скидку | Только пользователи с ролью `MANAGER` или `SUPERADMIN`, имеющие доступ к разделу (`gazebos` / `ps-park`) |
| Максимальная скидка | Настраивается в `Module.config.maxDiscountPercent`, по умолчанию 30% |
| Причина обязательна | При скидке > 0 причина всегда обязательна; без причины API отклоняет запрос с 422 |
| Скидка только при чекауте | Скидку можно применить только при переводе бронирования из `CONFIRMED` / `CHECKED_IN` → `COMPLETED`. Ретроактивное изменение завершённого бронирования запрещено. |
| Минимальная итоговая сумма | Итоговая сумма не может быть ниже 0. Скидка 100% недоступна через UI (можно настроить `maxDiscountPercent = 99`). |
| Скидка атомарна с чекаутом | Запись скидки в `Booking.metadata`, `AuditLog` и `FinancialTransaction` создаются в одной транзакции с изменением статуса на `COMPLETED`. |
| Бронирования без цены | Если `pricePerHour` ресурса не задан и `metadata.totalPrice = 0` — скидка применяется, но `discountAmount = 0` (записывается корректно). |

## Влияние на существующий код

### Изменения в схеме данных

Новых таблиц не требуется. Все данные хранятся в существующих моделях:

- `Booking.metadata` (JSONB) — добавляется ключ `discount` (необязательный).
- `FinancialTransaction.metadata` (JSONB) — для PS Park добавляется `originalAmount` и `discountPercent`.
- `Module.config` (JSONB) — добавляется поле `maxDiscountPercent`.

Миграция БД не нужна — всё в JSONB.

### Затрагиваемые модули

| Файл | Изменение |
|------|-----------|
| `src/modules/gazebos/service.ts` | `updateBookingStatus` принимает `discountPercent?` и `discountReason?`; при `COMPLETED` рассчитывает `finalAmount`, сохраняет в `metadata.discount` |
| `src/modules/ps-park/service.ts` | Аналогично; `FinancialTransaction.totalAmount` = `finalAmount`, в metadata добавляется `originalAmount` |
| `src/modules/gazebos/validation.ts` | Новая Zod-схема `CheckoutDiscountSchema` |
| `src/modules/ps-park/validation.ts` | Новая Zod-схема `CheckoutDiscountSchema` |
| `src/modules/booking/types.ts` | Добавляется тип `BookingDiscount` в `BookingMetadata` |
| `src/app/api/gazebos/bookings/[id]/route.ts` | PATCH принимает `discountPercent`, `discountReason`, `discountNote` из body |
| `src/app/api/ps-park/bookings/[id]/route.ts` | Аналогично |
| `src/lib/logger.ts` | Уже содержит `logAudit` — без изменений; вызов с новым `action` |

### Новые эндпоинты

Отдельных эндпоинтов не нужно. Скидка передаётся в тело существующего `PATCH /api/{module}/bookings/:id`:

```
PATCH /api/gazebos/bookings/:id
Body: {
  "status": "COMPLETED",
  "cashAmount": 1500,
  "cardAmount": 0,
  "discountPercent": 10,
  "discountReason": "permanent_client",
  "discountNote": null
}
```

Ответ:
```json
{
  "success": true,
  "data": {
    "id": "...",
    "status": "COMPLETED",
    "originalAmount": 1667,
    "discountPercent": 10,
    "discountAmount": 167,
    "finalAmount": 1500,
    "discountReason": "permanent_client"
  }
}
```

## Приоритет (MoSCoW)

**Категория: Must have**

**Обоснование:** Без этой функции менеджеры применяют скидки в обход системы — занижают `cashAmount` без записи причины. Это прямая потеря контроля над выручкой с первого дня работы платформы. Платформа запускается 17 апреля 2026 — скидки начнут применяться сразу.

**Зависимости:**
- Phase 0: `AuditLog`, `Module.config`, `Booking.metadata` — уже реализованы.
- Phase 2: `updateBookingStatus` в обоих модулях — уже реализован.
- Фича не блокирует и не блокируется Phase 5.1 (программа лояльности), но должна быть реализована раньше — программа лояльности будет читать `finalAmount` из транзакций.

## Метрики успеха

| Метрика | Базовое значение | Целевое значение |
|---------|-----------------|-----------------|
| Доля чекаутов со скидкой, зафиксированных в системе | 0% (все мимо) | 100% скидок через систему |
| Количество записей `booking.discount_applied` в AuditLog | 0 | Совпадает с количеством скидок, о которых сообщают менеджеры |
| Время применения скидки при чекауте | N/A | Не более 15 секунд дополнительных действий к стандартному чекауту |
| Отказов API из-за превышения лимита (`DISCOUNT_EXCEEDS_LIMIT`) | N/A | < 1% от попыток применить скидку (менеджеры знают лимит) |

## Вне скоупа

- **Автоматические скидки** — по дате, времени суток, занятости (принадлежит Phase 5.1 и выше).
- **Промокоды** — отдельная система с кодами и квотами.
- **Программа лояльности** — баллы и уровни (Phase 5.1, отдельный PRD).
- **Скидки на бронирования в статусе PENDING или CONFIRMED** — скидка применяется только в момент чекаута (COMPLETED). Изменение суммы подтверждённого, но не завершённого бронирования вне скоупа.
- **Ретроактивная корректировка** — изменение итоговой суммы уже завершённых бронирований.
- **Разные типы скидок** — фиксированная сумма (не процент). Только процент.
- **Согласование скидки с суперадмином** — не требуется подтверждение перед применением, только постфактум аудит.
- **Скидки в модуле Кафе** — другая модель (заказы, не бронирования). Отдельная задача.
- **UI отчёта по скидкам** — AC-3.x использует уже существующий UI журнала аудита с фильтром. Новый дашборд скидок не создаётся.

## Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Менеджер указывает причину формально ("другое" без описания) | Средняя | Низкое | Минимум 5 символов для free-text обязателен по AC-1.4 |
| Скидка применяется, но `FinancialTransaction` создаётся с неверной суммой | Низкая | Высокое | Всё в одной транзакции (AC-2.4); тест на happy path обязателен |
| Менеджер двух модулей (gazebos + ps-park) пытается применить скидку выше лимита одного из них | Средняя | Низкое | `maxDiscountPercent` проверяется на стороне API по `moduleSlug` из бронирования |
| Ретроактивная правка через API напрямую (bypass UI) | Низкая | Высокое | Проверка статуса на уровне сервиса: `COMPLETED` → отклонить; скидка только при переходе → `COMPLETED` |
| Phase 5.1 (лояльность) читает `totalPrice` из metadata вместо `finalAmount` | Средняя | Средняя | Зафиксировать в ADR: `metadata.discount.finalAmount` — каноническое поле выручки после скидки |
