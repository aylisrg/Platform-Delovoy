# PRD: Интеграция телефонии Novofon в Platform Delovoy

**Версия**: 1.0  
**Дата**: 2026-04-12  
**Статус**: Draft  

---

## Проблема

Сейчас клиенты, которые хотят уточнить детали бронирования беседки или стола в PS Park, не могут связаться с менеджером прямо с сайта. Единственный канал — Telegram-бот, который требует наличия аккаунта. Часть аудитории (особенно старшего возраста или корпоративные клиенты) предпочитает телефонный звонок.

Со стороны менеджера нет инструмента для исходящего обзвона клиентов из интерфейса системы: чтобы подтвердить бронирование по телефону, нужно вручную набрать номер в отдельном телефоне. История звонков нигде не фиксируется. Запись звонков для контроля качества отсутствует.

**Персоны, для которых это боль:**
- **USER (клиент B2C)** — хочет позвонить и уточнить детали (доступность, условия, добраться), не переключаясь с сайта.
- **MANAGER (беседки / PS Park)** — хочет позвонить клиенту в один клик прямо из карточки бронирования, не набирая вручную.
- **SUPERADMIN** — хочет видеть статистику звонков и иметь доступ к записям.

---

## Решение

Интегрировать виртуальную АТС Novofon в два B2C модуля (Беседки и PS Park) и в административные панели этих модулей:

1. **Кнопка "Позвонить" на публичных страницах** — клиент нажимает, браузер инициирует звонок через WebRTC (если поддерживается платформой) или открывает `tel:` ссылку. Оба модуля используют один виртуальный номер Novofon.
2. **Click-to-call из карточки бронирования для менеджера** — кнопка "Позвонить клиенту" в панели менеджера вызывает Novofon Call API (`start.employee_call`), соединяя менеджера с клиентом через виртуальную АТС.
3. **Запись звонков и история** — все звонки, инициированные через платформу, логируются в `SystemEvent`, записи доступны менеджеру в карточке бронирования.

---

## Целевая аудитория

| Роль | Сценарий |
|------|----------|
| USER | Позвонить в беседки или PS Park со страницы сервиса |
| MANAGER (gazebos / ps-park) | Позвонить клиенту из карточки бронирования, просмотреть историю и запись звонка |
| SUPERADMIN | Просмотреть сводную статистику звонков, настроить интеграцию |

## User Stories

### US-1: Кнопка звонка на публичной странице

- **Как** клиент, находящийся на странице `/gazebos` или `/ps-park`
- **Я хочу** нажать кнопку "Позвонить" и немедленно связаться с менеджером
- **Чтобы** уточнить детали до бронирования (наличие, условия, как добраться)

**Acceptance Criteria:**
- [ ] AC-1: На странице беседок и PS Park отображается заметная кнопка "Позвонить" с виртуальным номером Novofon.
- [ ] AC-2: На десктопе кнопка открывает `tel:` ссылку — браузер передаёт звонок в телефонное приложение или SIP-клиент.
- [ ] AC-3: На мобильном устройстве `tel:` ссылка инициирует прямой звонок.
- [ ] AC-4: Оба модуля (беседки и PS Park) используют один и тот же виртуальный номер Novofon.
- [ ] AC-5: Номер телефона отображается в виде читаемого текста рядом с кнопкой (для ручного набора).
- [ ] AC-6: Номер телефона хранится в конфигурации модуля (`Module.config`) и не зашит в код.

---

### US-2: Click-to-call из карточки бронирования

- **Как** менеджер беседок или PS Park
- **Я хочу** нажать кнопку "Позвонить клиенту" в карточке бронирования
- **Чтобы** связаться с клиентом без ручного набора номера

**Acceptance Criteria:**
- [ ] AC-1: В карточке бронирования (где есть `clientPhone`) отображается кнопка "Позвонить".
- [ ] AC-2: При нажатии платформа вызывает Novofon Call API метод `start.employee_call`, соединяя SIP-линию менеджера с телефоном клиента.
- [ ] AC-3: В случае ошибки (клиент недоступен, API недоступен) менеджер получает понятное сообщение об ошибке прямо в интерфейсе.
- [ ] AC-4: После инициации звонка в карточке бронирования отображается статус "Звонок инициирован" с временной меткой.
- [ ] AC-5: Если у бронирования нет номера телефона клиента, кнопка скрыта или недоступна с подсказкой.

---

### US-3: Журнал звонков по бронированию

- **Как** менеджер
- **Я хочу** видеть историю звонков по конкретному бронированию
- **Чтобы** знать, сколько раз звонили клиенту, кто звонил и когда

**Acceptance Criteria:**
- [ ] AC-1: В карточке бронирования есть секция "История звонков" (отображается, только если были звонки).
- [ ] AC-2: Каждая запись в истории содержит: дата/время, направление (исходящий/входящий), продолжительность, статус (отвечен / нет), кто инициировал.
- [ ] AC-3: История персистентна — сохраняется в `SystemEvent` с `source: "telephony"` и привязкой к `entityId` (ID бронирования).
- [ ] AC-4: История отображается без перезагрузки страницы (lazy-load или inline).

---

### US-4: Доступ к записям звонков

- **Как** менеджер или SUPERADMIN
- **Я хочу** прослушать запись конкретного звонка
- **Чтобы** разобрать спорную ситуацию или проверить качество обслуживания

**Acceptance Criteria:**
- [ ] AC-1: Для звонков, по которым Novofon вернул ссылку на запись, в журнале звонков отображается кнопка "Прослушать".
- [ ] AC-2: Запись открывается в браузере (HTML5 audio player или новая вкладка) — без скачивания.
- [ ] AC-3: Доступ к записям ограничен: USER не может прослушивать записи звонков.
- [ ] AC-4: Если запись ещё не готова (задержка обработки Novofon), показывается статус "Запись обрабатывается".

---

### US-5: Входящий звонок — вебхук и атрибуция

- **Как** менеджер
- **Я хочу** чтобы входящий звонок на виртуальный номер автоматически логировался в системе
- **Чтобы** видеть в журнале звонков не только исходящие, но и входящие

**Acceptance Criteria:**
- [ ] AC-1: Платформа принимает вебхук от Novofon при входящем звонке на виртуальный номер.
- [ ] AC-2: По номеру звонящего система пытается найти совпадение с `clientPhone` в активных бронированиях (текущий день ±3 дня).
- [ ] AC-3: Если совпадение найдено — звонок атрибутируется к бронированию и попадает в его журнал.
- [ ] AC-4: Если совпадение не найдено — звонок логируется в `SystemEvent` как неатрибутированный входящий.
- [ ] AC-5: Вебхук-эндпоинт защищён (проверка подписи Novofon или IP-whitelist).

---

### US-6: Настройка интеграции суперадмином

- **Как** SUPERADMIN
- **Я хочу** настроить параметры интеграции Novofon через Config GUI
- **Чтобы** не требовать деплоя при изменении номера, SIP-аккаунтов или API-ключей

**Acceptance Criteria:**
- [ ] AC-1: В разделе "Модули" / настройках есть секция "Телефония Novofon" с полями: API-ключ, виртуальный номер, ID SIP-линии менеджера (по модулю).
- [ ] AC-2: Изменения конфигурации применяются без перезапуска приложения.
- [ ] AC-3: Конфигурация хранится в `Module.config` (JSONB) — отдельно для `gazebos` и `ps-park`, но с возможностью указать общий номер.
- [ ] AC-4: Суперадмин может включить/выключить телефонный функционал для каждого модуля отдельно.
- [ ] AC-5: Тест-кнопка "Проверить соединение" вызывает Novofon Data API и показывает статус аккаунта.

## Приоритет (MoSCoW)

### Must Have
- US-1: Кнопка "Позвонить" (`tel:` ссылка) на публичных страницах беседок и PS Park
- US-2: Click-to-call через Novofon Call API из карточки бронирования для менеджера
- US-6: Хранение конфигурации в `Module.config`, включение/выключение через Config GUI

### Should Have
- US-3: Журнал звонков по бронированию
- US-4: Доступ к записям звонков

### Could Have
- US-5: Вебхук для входящих звонков и автоатрибуция к бронированию
- WebRTC-звонок прямо в браузере (без переключения в телефонное приложение) — при наличии WebRTC SDK от Novofon

### Won't Have (этот спринт)
- Интеграция телефонии в модули Кафе, Аренда офисов — отдельный спринт
- IVR (голосовое меню) — не нужно для точечных сервисов с одним направлением
- SMS-уведомления через Novofon — уже реализованы через Telegram

## Технические ограничения и решения

### Novofon API

**Data API**: `https://dataapi-jsonrpc.novofon.ru/2.0` — JSON-RPC 2.0, используется для управления данными аккаунта, получения статистики звонков, доступа к записям.

**Call API**: используется для инициации звонков (`start.employee_call`, `start.simple_call`), управления активными звонками.

**Аутентификация**: API-ключ (постоянный) или login/password (сессия 1 час). Рекомендуется API-ключ, хранящийся в `.env` как `NOVOFON_API_KEY`.

**IP Whitelist**: Novofon требует whitelist IP-адресов для API-запросов. Необходимо добавить IP VPS в настройках безопасности Novofon.

**WebRTC**: Официальной WebRTC SDK в документации Novofon не обнаружено. Для браузерных звонков используется `tel:` ссылка (US-1). Возможность WebRTC уточнить у поддержки Novofon отдельно.

### Новый модуль: `src/modules/telephony/`

Телефония реализуется как **shared-сервис** (не доменный модуль), аналогично `notifications`. Структура:

```
src/modules/telephony/
├── service.ts        # Бизнес-логика: инициация звонка, получение записей, логирование
├── types.ts          # Типы: CallRecord, CallDirection, CallStatus
├── validation.ts     # Zod-схемы для вебхука Novofon
└── novofon-client.ts # HTTP-клиент для Novofon API (Data API + Call API)
```

### Новый API эндпоинт

```
POST /api/telephony/call          — инициировать исходящий звонок (менеджер → клиент)
POST /api/telephony/webhook       — вебхук от Novofon (входящие звонки, события)
GET  /api/telephony/calls         — список звонков (с фильтрацией по entityId, moduleSlug)
GET  /api/telephony/calls/:id/recording — получить ссылку на запись
GET  /api/telephony/health        — health check интеграции
```

### Хранение данных

Звонки логируются в `SystemEvent`:

```json
{
  "level": "INFO",
  "source": "telephony",
  "message": "outbound_call_initiated",
  "metadata": {
    "moduleSlug": "gazebos",
    "entityId": "<bookingId>",
    "direction": "outbound",
    "callId": "<novofon_call_id>",
    "clientPhone": "+7...",
    "initiatedBy": "<managerId>",
    "recordingUrl": null,
    "duration": null,
    "status": "initiated"
  }
}
```

Запись и длительность дополняются после завершения звонка через вебхук или polling.

### Конфигурация модулей

В `Module.config` для `gazebos` и `ps-park` добавляются поля:

```json
{
  "telephony": {
    "enabled": true,
    "virtualNumber": "+74951234567",
    "displayNumber": "+7 (495) 123-45-67",
    "sipLineId": "sip_line_id_from_novofon",
    "recordCalls": true
  }
}
```

## Метрики успеха

| Метрика | Цель (3 месяца после запуска) |
|---------|-------------------------------|
| Конверсия страницы беседок (клик на "Позвонить" / уникальные посетители) | >= 5% |
| Время на обработку бронирования менеджером (от создания до подтверждения) | Снижение на 20% |
| % бронирований, где менеджер звонил клиенту через платформу | >= 60% от бронирований с телефоном |
| Количество потерянных входящих (нет ответа) — видно из журнала | Baseline за первый месяц |
| Uptime интеграции | >= 99% (нет ошибок Novofon API в SystemEvent CRITICAL) |

## Вне скоупа

- WebRTC-звонок прямо в браузере без переключения в отдельное приложение (требует подтверждения WebRTC поддержки от Novofon).
- Интеграция телефонии в модули Кафе и Аренда офисов.
- IVR и голосовые сценарии.
- Автоматические напоминания о бронировании через звонок (robocall).
- SMS через Novofon.
- Биллинг и управление балансом Novofon через интерфейс платформы.
- Мобильное приложение менеджера с VoIP-функцией.

## Открытые вопросы

1. **WebRTC**: Поддерживает ли Novofon браузерные WebRTC-звонки? Нужно запросить у поддержки или изучить документацию SIP-настроек.
2. **SIP-линии менеджеров**: Есть ли у менеджеров беседок и PS Park настроенные SIP-линии в Novofon? Если нет — click-to-call работает через мобильный/стационарный телефон менеджера (Novofon сначала звонит менеджеру, потом соединяет с клиентом).
3. **IP Whitelist**: IP адрес VPS должен быть заранее добавлен в Novofon — блокер для деплоя.
4. **Записи звонков**: Опция записи включена на стороне Novofon? Требует проверки настроек аккаунта.
5. **Один номер для двух модулей**: Как Novofon маршрутизирует входящие? Нужен один сценарий на АТС, который перенаправляет на дежурного менеджера.
