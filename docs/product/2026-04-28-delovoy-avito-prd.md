# PRD: Деловой Авито — интеграция Avito Professionals API

**Раздел админки:** `/admin/avito`
**Дата:** 2026-04-28
**Статус:** Готов к передаче Архитектору
**Автор:** Product Owner

---

## Проблема

Бизнес-парк "Деловой" активно использует Авито как канал привлечения клиентов для Барбекю Парка и PlayStation Park. На данный момент:

1. Статистика по объявлениям смотрится в ручном режиме на сайте Авито — менеджер должен заходить в личный кабинет отдельно, без связи с операционной системой парка.
2. Лиды из Авито Мессенджер теряются: клиент пишет на Авито, менеджер видит это на телефоне через приложение Авито, ответ задерживается на часы. Потеря лида происходит молча.
3. PS Park к Авито не подключён вообще — нет даже базовой статистики.
4. Отзывы на Авито не мониторятся системой — негативный отзыв можно пропустить на несколько дней.
5. Пропущенные звонки с Авито не попадают ни в какую систему и фактически теряются.

Результат: канал Авито работает вхолостую. Деньги тратятся на размещение объявлений, но конверсия из лида в бронирование не отслеживается, а скорость реакции — главный фактор конверсии в нишах «аренда беседок» и «PS-зал» — не контролируется.

---

## Решение

Единый раздел администратора «Деловой Авито» (`/admin/avito`), который:

- Показывает статистику по всем объявлениям парка в одном месте — с привязкой к модулям (Барбекю Парк, PS Park).
- Превращает входящие сообщения из Авито Мессенджер в задачи в канбане (Phase 5.4), позволяя менеджеру отвечать клиенту прямо из платформы.
- Автоматически отвечает клиенту в первые 60 секунд и отправляет ссылку на бронирование.
- Алертирует об отзывах ≤3 звёзд и пропущенных звонках в Telegram.

Лиды и события Авито обогащают существующие модули (задачи, беседки, PS Park), а не создают параллельную структуру.

---

## Целевая аудитория

| Роль | Кто в реальности | Что получает |
|------|-----------------|--------------|
| MANAGER (gazebos) | Администратор Барбекю Парка | Видит лиды и статистику только своего объявления |
| MANAGER (ps-park) | Менеджер PS Park | Видит лиды и статистику PS Park — впервые подключается |
| SUPERADMIN | Управляющий / владелец парка | Видит все объявления, баланс кошелька, сводный рейтинг |
| Маркетолог парка | Сотрудник, ведущий рекламу | Сравнение метрик по периодам, динамика |

---

## Что уже реализовано (не делаем повторно)

- `src/lib/avito.ts` — OAuth2 клиент с кешированием токена в Redis на 23.5 ч. Функция `getAvitoItemStats()` запрашивает метрики одного объявления.
- `/admin/gazebos/marketing` — виджет статистики для одного объявления беседок (views, contacts, calls, favorites, uniqViews, missedCalls).
- Phase 5.4 Tasks — канбан с `Task`, `TaskCategory`, `TaskComment`, `TaskEvent`, поддержкой `source=API`, channel-agnostic `NotificationDispatcher` с интерфейсом `INotificationChannel`.
- ENV `AVITO_CLIENT_ID`, `AVITO_CLIENT_SECRET` — один аккаунт на весь парк.

## User Stories и Acceptance Criteria

### Этап 1 — Дашборд и мульти-объявление (P0)

---

#### US-1.1: Список всех объявлений парка

- **Как** SUPERADMIN или MANAGER
- **Я хочу** открыть раздел «Деловой Авито» и увидеть список всех активных объявлений аккаунта с их метриками
- **Чтобы** не переключаться между платформой и личным кабинетом Авито для базового контроля

**Acceptance Criteria:**
- [ ] AC-1: Страница `/admin/avito` доступна только авторизованным пользователям с ролью MANAGER или SUPERADMIN.
- [ ] AC-2: На странице отображается список объявлений, полученных из Avito Core API для аккаунта парка.
- [ ] AC-3: Каждое объявление показывает: название, статус (активное/снятое), метрики за выбранный период — views, uniqViews, contacts, favorites, calls, missedCalls.
- [ ] AC-4: Менеджер видит только объявления, привязанные к его модулю. SUPERADMIN видит все.
- [ ] AC-5: При отсутствии объявлений, привязанных к модулю менеджера, отображается информационное сообщение «Нет подключённых объявлений».
- [ ] AC-6: Если API Авито недоступен, отображается последнее кешированное значение с меткой времени обновления или сообщение «Данные недоступны».

---

#### US-1.2: Привязка объявления к модулю парка

- **Как** SUPERADMIN
- **Я хочу** в разделе «Деловой Авито» назначить каждому объявлению аккаунта принадлежность к модулю (Барбекю Парк / PS Park)
- **Чтобы** менеджеры видели только свои объявления, а лиды и звонки автоматически маршрутизировались к правильному менеджеру

**Acceptance Criteria:**
- [ ] AC-1: SUPERADMIN может назначить объявление одному из модулей: `gazebos`, `ps-park` (или оставить без привязки).
- [ ] AC-2: Привязка сохраняется в системе и немедленно вступает в силу — менеджер соответствующего модуля начинает видеть объявление в своём виде.
- [ ] AC-3: Одно объявление может быть привязано только к одному модулю.
- [ ] AC-4: При удалении привязки объявление перестаёт отображаться у менеджера, но его метрики сохраняются в истории.
- [ ] AC-5: SUPERADMIN видит непривязанные объявления отдельным блоком «Без модуля».

---

#### US-1.3: Выбор периода метрик и сравнение

- **Как** MANAGER или SUPERADMIN
- **Я хочу** переключать период статистики между «7 дней» и «30 дней» и видеть изменение показателей
- **Чтобы** понимать тренд — растёт ли интерес к услуге или падает

**Acceptance Criteria:**
- [ ] AC-1: На странице объявления доступно переключение периода: 7 дней / 30 дней.
- [ ] AC-2: При смене периода все метрики (views, uniqViews, contacts, favorites, calls, missedCalls) пересчитываются.
- [ ] AC-3: Рядом с каждой метрикой за текущий период показывается дельта к предыдущему аналогичному периоду (например, «+12% к прошлым 7 дням»). Дельта выделяется цветом: зелёный — рост, красный — падение.
- [ ] AC-4: Данные кешируются: повторный запрос того же периода в течение 30 минут не инициирует новый запрос к Avito API.

---

#### US-1.4: Баланс кошелька и тариф Авито

- **Как** SUPERADMIN
- **Я хочу** видеть на дашборде Авито текущий баланс кошелька и активный тариф объявления
- **Чтобы** не пропустить момент, когда нужно пополнить баланс или продлить размещение

**Acceptance Criteria:**
- [ ] AC-1: На дашборде `/admin/avito` отображается текущий баланс кошелька Авито в рублях.
- [ ] AC-2: При балансе ниже 500 ₽ отображается предупреждение (жёлтый индикатор).
- [ ] AC-3: Данные о балансе обновляются не реже одного раза в час.
- [ ] AC-4: MANAGER не видит раздел баланса и тарифов — только SUPERADMIN.

---

### Этап 2 — Мессенджер Авито как канал лидов (P0, самая ценная часть)

---

#### US-2.1: Входящее сообщение из Авито становится задачей

- **Как** менеджер Барбекю Парка или PS Park
- **Я хочу** видеть в канбане платформы новую задачу каждый раз, когда клиент написал на Авито
- **Чтобы** не пропустить ни одного входящего лида и обрабатывать всё в одном месте, а не переключаться в приложение Авито

**Acceptance Criteria:**
- [ ] AC-1: При получении нового входящего сообщения от клиента через Авито Мессенджер в системе автоматически создаётся задача типа «Лид: Авито/{модуль}» (например, «Лид: Авито/Барбекю Парк»).
- [ ] AC-2: Задача создаётся в категории, соответствующей `moduleSlug` объявления из привязки (US-1.2). Если объявление не привязано — задача создаётся в категории «Авито / Без модуля» и назначается SUPERADMIN.
- [ ] AC-3: В теле задачи отображается: имя клиента на Авито, текст первого сообщения, ссылка на объявление, ссылка на диалог в Авито Мессенджере.
- [ ] AC-4: Если от одного клиента по одному объявлению уже есть открытая задача (статус не «Закрыто»/«Выполнено») — новое сообщение добавляется комментарием к существующей задаче, а не создаёт дубликат.
- [ ] AC-5: Если тот же webhook-запрос приходит повторно (retry от Авито) — задача не создаётся повторно (idempotency по `message_id`).
- [ ] AC-6: При сбое обработки webhook-а запрос логируется в `SystemEvent` с уровнем WARNING, и Авито получает ответ 200 OK (чтобы не вызвать бесконечные retry). Необработанное сообщение помещается в очередь на повторную попытку.
- [ ] AC-7: Задача автоматически назначается ответственному менеджеру модуля (если назначен в `ModuleAssignment`).
- [ ] AC-8: Менеджер получает уведомление о новом лиде через настроенный канал (Telegram — приоритетный) в течение 2 минут после получения сообщения.

---

#### US-2.2: Ответ клиенту прямо из задачи

- **Как** менеджер
- **Я хочу** ответить клиенту в Авито Мессенджер прямо из карточки задачи, не открывая приложение Авито
- **Чтобы** скорость ответа не зависела от того, есть ли у менеджера под рукой телефон с Авито

**Acceptance Criteria:**
- [ ] AC-1: В карточке задачи с источником «Авито» отображается раздел «Ответить в Авито» с полем для ввода текста и кнопкой «Отправить».
- [ ] AC-2: После нажатия «Отправить» сообщение уходит клиенту в Авито Мессенджер через API Авито.
- [ ] AC-3: Успешно отправленный ответ отображается в хронологии событий задачи (TaskEvent) с меткой «Отправлено в Авито» и временем отправки.
- [ ] AC-4: Если отправка не удалась (ошибка API Авито), менеджер видит сообщение об ошибке прямо в UI и возможность повторить отправку. Ошибка логируется.
- [ ] AC-5: Входящие ответы клиента (новые сообщения в том же диалоге) также добавляются как комментарии к задаче через webhook (AC-4 из US-2.1).
- [ ] AC-6: Ответ фиксируется в `AuditLog` — кто, когда, что отправил.

---

#### US-2.3: Автоответ клиенту в первые 60 секунд

- **Как** клиент, написавший на Авито
- **Я хочу** в течение минуты получить ответ о том, что моё обращение принято, и ссылку для самостоятельного бронирования
- **Чтобы** не ждать живого менеджера и иметь возможность сразу забронировать

**Acceptance Criteria:**
- [ ] AC-1: После создания задачи из входящего сообщения система автоматически отправляет клиенту в Авито Мессенджер приветственный ответ в течение 60 секунд.
- [ ] AC-2: Текст автоответа берётся из настроек модуля (`Module.config`) — каждый модуль может иметь свой текст. По умолчанию используется общий шаблон парка.
- [ ] AC-3: Автоответ содержит ссылку на страницу бронирования соответствующего модуля (например, `delovoy-park.ru/gazebos` или `delovoy-park.ru/ps-park`).
- [ ] AC-4: Автоответ отправляется только на первое сообщение в диалоге — не на каждое входящее сообщение.
- [ ] AC-5: SUPERADMIN может включить или отключить автоответ для каждого модуля отдельно в настройках раздела «Деловой Авито».
- [ ] AC-6: Факт отправки автоответа фиксируется в хронологии задачи (TaskEvent).

---

#### US-2.4: Метрика скорости реакции на лиды

- **Как** SUPERADMIN
- **Я хочу** видеть среднее время первого ответа менеджера на лиды из Авито за последние 7 и 30 дней
- **Чтобы** контролировать ключевой показатель конверсии и понимать, соблюдается ли стандарт «ответ за 15 минут»

**Acceptance Criteria:**
- [ ] AC-1: На дашборде «Деловой Авито» отображается блок «Скорость реакции»: среднее время от получения первого сообщения клиента до первого ответа менеджера (исключая автоответ), за периоды 7 дней и 30 дней.
- [ ] AC-2: Метрика рассчитывается отдельно для каждого модуля (Барбекю Парк, PS Park).
- [ ] AC-3: Если среднее время реакции превышает 30 минут — показатель выделяется красным.
- [ ] AC-4: Учитываются только рабочие часы (09:00–21:00 МСК) — ответы вне этого окна исключаются из расчёта.

---

#### US-2.5: Процент отвеченных лидов

- **Как** SUPERADMIN
- **Я хочу** видеть, какой процент лидов из Авито получил ответ от менеджера в течение 1 часа
- **Чтобы** понимать эффективность обработки Авито-лидов и принимать решения о найме или изменении процессов

**Acceptance Criteria:**
- [ ] AC-1: На дашборде отображается показатель «% лидов, отвеченных за 1 ч» за последние 30 дней.
- [ ] AC-2: Считаются лиды, по которым менеджер отправил хотя бы один ответ (не автоответ) в течение 60 минут с момента получения первого сообщения.
- [ ] AC-3: Если показатель ниже 70% — индикатор красный. 70–90% — жёлтый. Выше 90% — зелёный.

---

### Этап 3 — Мониторинг отзывов (P1)

---

#### US-3.1: Алерт при негативном отзыве

- **Как** SUPERADMIN
- **Я хочу** немедленно получать уведомление в Telegram, когда на объявление приходит отзыв с оценкой 3 звезды и ниже
- **Чтобы** успеть быстро отреагировать на публичный негатив — ответить, предложить решение

**Acceptance Criteria:**
- [ ] AC-1: Система проверяет новые отзывы по всем объявлениям парка с периодичностью раз в час.
- [ ] AC-2: При обнаружении нового отзыва с рейтингом ≤3 звёзд SUPERADMIN и назначенный менеджер модуля получают уведомление в Telegram в течение 5 минут после следующего планового опроса.
- [ ] AC-3: Уведомление содержит: имя автора отзыва, оценку, текст отзыва (первые 300 символов), ссылку на объявление.
- [ ] AC-4: Один и тот же отзыв не порождает повторное уведомление.

---

#### US-3.2: Средний рейтинг в дашборде

- **Как** SUPERADMIN или MANAGER
- **Я хочу** видеть текущий средний рейтинг каждого объявления на странице «Деловой Авито»
- **Чтобы** иметь актуальную картину репутации на Авито без ручного обхода объявлений

**Acceptance Criteria:**
- [ ] AC-1: На карточке каждого объявления отображается средний рейтинг (например, «4.7 / 5») и общее количество отзывов.
- [ ] AC-2: Данные о рейтинге обновляются при каждом плановом опросе (раз в час).
- [ ] AC-3: Средний рейтинг по модулю (агрегация по всем его объявлениям) будет передан в дашборд владельца Phase 5.3 — как именованная метрика `avito_avg_rating:{moduleSlug}`.

---

### Этап 4 — Пропущенные звонки как лиды (P1)

---

#### US-4.1: Пропущенный звонок становится задачей

- **Как** менеджер
- **Я хочу** видеть в канбане задачу каждый раз, когда с Авито поступил пропущенный звонок
- **Чтобы** перезвонить клиенту и не потерять тёплый лид — человек, который звонит, почти готов к покупке

**Acceptance Criteria:**
- [ ] AC-1: При поступлении webhook-события о пропущенном звонке с Авито в системе создаётся задача с категорией «Пропущенный звонок: Авито/{модуль}» и высоким приоритетом.
- [ ] AC-2: Маршрутизация задачи по `moduleSlug` — аналогично US-2.1 AC-2.
- [ ] AC-3: В задаче указаны: номер телефона звонившего (если Авито передаёт), время звонка, название объявления.
- [ ] AC-4: Если номер телефона совпадает с телефоном существующего `User` или `Tenant` в системе — в metadata задачи добавляется ссылка на этот профиль.
- [ ] AC-5: Idempotency: повторный webhook по тому же звонку не создаёт дублирующую задачу.
- [ ] AC-6: Менеджер получает уведомление о пропущенном звонке через настроенный канал (Telegram) в течение 5 минут.

## Приоритет (MoSCoW)

| Story | Приоритет | Обоснование |
|-------|-----------|-------------|
| US-1.1 Список объявлений | Must have | Без этого нет единого места управления — весь раздел теряет смысл |
| US-1.2 Привязка к модулю | Must have | Фундамент маршрутизации для всех последующих этапов |
| US-1.3 Периоды и сравнение | Must have | Без сравнения метрики — просто числа без контекста |
| US-1.4 Баланс кошелька | Should have | Важно, но некритично для первого запуска |
| US-2.1 Лид в канбан | Must have | Главная бизнес-ценность интеграции — ни один лид не теряется |
| US-2.2 Ответ из задачи | Must have | Без исходящей связи интеграция односторонняя и неполная |
| US-2.3 Автоответ | Must have | Прямо влияет на конверсию: клиент в первые 60с не брошен |
| US-2.4 Скорость реакции | Should have | Нужна для управления, но не блокирует MVP |
| US-2.5 % отвеченных лидов | Should have | Нужна для управления, но не блокирует MVP |
| US-3.1 Алерт на отзыв ≤3 | Should have | Репутационный риск реален, но не блокирует запуск |
| US-3.2 Рейтинг в дашборде | Could have | Полезно, но Phase 5.3 ещё не готова |
| US-4.1 Звонки в задачи | Should have | Ценно, но реже случается, чем мессенджер |

## Метрики успеха

| Метрика | Сейчас (базовое) | Цель (через 4 недели после запуска) |
|---------|-----------------|-------------------------------------|
| Среднее время первого ответа на лид | Не измеряется (оценочно: 2–6 ч) | < 15 мин в рабочие часы |
| % лидов с ответом в течение 1 ч | Не измеряется | > 80% |
| Пропущенные лиды (без ответа вообще) | Не измеряется (оценочно: 30–50%) | < 10% |
| Время обнаружения отзыва ≤3 | До нескольких дней | < 1 ч (следующий плановый опрос) |
| Пропущенные звонки с перезвоном | Не измеряется | > 90% перезвонов в тот же день |
| Переключений между платформой и Авито в рабочую смену | Не измеряется | Стремится к 0 (все действия из одного интерфейса) |

## Зависимости

1. **Phase 5.4 Tasks** — должна быть слита в `main` и доступна в production. Подтверждено: слита (коммит `0662392`).
2. **Публичный HTTPS endpoint** — для получения webhook от Авито (`/api/avito/webhook/messenger`, `/api/avito/webhook/calls`) нужен публичный домен с валидным SSL. Production-домен `delovoy-park.ru` уже развёрнут (с 2026-04-14).
3. **Авито Professionals API ключи** — реальные `AVITO_CLIENT_ID` и `AVITO_CLIENT_SECRET` уже есть (используются для статистики беседок). Для Messenger API может потребоваться явное включение scope `messenger` в личном кабинете Avito Pro.
4. **Avito User ID** — для отправки сообщений через Messenger API нужен `user_id` аккаунта парка в Авито (получается через `GET /core/v1/accounts/self`). Сейчас не сохраняется — нужно добавить.
5. **Настройка webhook URL в Avito** — администратор Авито-аккаунта парка должен зарегистрировать URL webhook'а в личном кабинете Avito Pro (одноразовое действие).
6. **Новая таблица `AvitoItem`** — привязка Avito `item_id` к `moduleSlug`. Создание таблицы — задача Архитектора. Минимальные поля: `id`, `avitoItemId` (string), `moduleSlug`, `title`, `isActive`, `createdAt`. Одноразовая миграция ENV `AVITO_ITEM_ID` → первая запись в таблице при инициализации.

## Вне скоупа (текущая итерация)

Следующие функции **осознанно не входят** в данную итерацию. Могут быть рассмотрены в Phase 2 интеграции Авито после получения feedback от менеджеров:

- Модуль `src/modules/avito/` — не создаётся. Это интеграция, а не бизнес-домен. Вся логика расширяет `src/lib/avito.ts` и использует существующие модули.
- Загрузка и управление XML-фидами объявлений (autoload). Авито поддерживает XML-импорт, но это требует отдельного продуктового решения.
- Покупка VAS-услуг (продвижение, выделение, XL-объявления) из админки. Управление рекламным бюджетом — отдельная функция с финансовыми рисками.
- Редактирование контента объявлений (текст, фото, цены, условия) из платформы. Требует отдельного UI и согласования с командой.
- Автоматические ответы на основе NLP или ИИ. В первой итерации — только шаблонный автоответ с ссылкой на бронирование.
- Интеграция статистики Авито в публичный личный кабинет клиента (`/profile`). Нет продуктового обоснования.
- Интеграция с другими площадками (Яндекс Недвижимость, ЦИАН, Domclick). Авито — первый приоритет.
- Модерация и ответы на отзывы из платформы (публичные ответы от имени аккаунта). Требует дополнительного скоупа этапа 3.

## Открытые вопросы к Архитектору (для ADR)

1. **Webhook security**: Поддерживает ли Avito Professionals API HMAC-подпись для верификации webhook-запросов? Если нет — какой механизм защиты (secret-токен в URL, IP whitelist)? Какой срок действия токена и как он ротируется?

2. **Avito User ID**: `GET /core/v1/accounts/self` возвращает `user_id` аккаунта. Нужно ли его хранить в ENV или в таблице `AvitoItem`/отдельной таблице настроек интеграции?

3. **Миграция `AVITO_ITEM_ID`**: Текущий ENV-переменная должна стать первой записью `AvitoItem`. Нужен ли скрипт миграции данных или достаточно seed-скрипта? Нужно ли поддерживать обратную совместимость `getAvitoItemStats()` — или рефакторим с новым API?

4. **Polling vs Webhook для мессенджера**: Avito поддерживает webhook для мессенджера. Архитектору нужно подтвердить scope доступных событий (новое сообщение, прочтение, пропущенный звонок) и формат payload. Если webhook недоступен для конкретного плана Авито Pro — нужен polling fallback. Какова стратегия?

5. **Dedup ключ для лидов**: Уникальность задачи определяется по паре (`chat_id`, `item_id`) или по (`user_id`, `item_id`)? Один клиент может написать с разных аккаунтов.

6. **Rate limits Avito API**: Каковы актуальные лимиты Avito API на: (а) запросы статистики (`/stats/v1/...`), (б) отправку сообщений Messenger (`/messenger/v1/...`), (в) запросы отзывов? Нужна ли очередь или достаточно retry с backoff?

7. **Отправка сообщений в Messenger**: Авито Messenger API может требовать, чтобы первое сообщение в чате было инициировано пользователем — то есть автоответ может быть технически невозможен, если API не допускает отправку в «холодный» чат. Нужно уточнить документацию и, возможно, ограничить автоответ только существующими диалогами.

8. **AvitoChannel как INotificationChannel**: Имеет ли смысл Avito Messenger регистрировать как `NotificationChannelKind` в системе Phase 5.4? Или это исходящий канал только для ответов в задачах, без подписки пользователя? Архитектор определяет, нужен ли новый enum-value.

9. **Хранение истории сообщений**: Сообщения диалога на Авито хранятся в `TaskComment`. Нужно ли дополнительно хранить входящие сообщения в отдельной таблице (как в email-модулях) — или `TaskComment` достаточно?

## Риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Rate limit Avito API на статистику — запросы всех менеджеров в пике | Средняя | Средний | Единый кеш в Redis (30 мин) для статистики; все менеджеры читают из кеша, не из API напрямую |
| Webhook Авито упал или недоступен — retry шторм | Низкая | Высокий | Ответ 200 OK при любом входящем webhook, ставим в очередь; idempotency по `message_id` предотвращает дубли |
| `getAvitoItemStats()` сломается после рефакторинга | Средняя | Средний | Обратная совместимость: сохраняем сигнатуру функции, внутренне переключаем на новый `AvitoItem.avitoItemId` из таблицы вместо ENV |
| API Авито Messenger закрыт или недоступен на текущем тарифе | Средняя | Высокий | Уточнить scope доступа до начала разработки этапа 2. Если недоступен — этап 2 блокирован, приоритизируем этапы 1, 3, 4 |
| Автоответ воспринимается клиентами как спам и снижает конверсию | Низкая | Средний | A/B: включаем автоответ только для одного модуля на первые 2 недели, меряем конверсию лидов в бронирования |
| Разрыв соответствия `chat_id` ↔ `Task` при параллельных сообщениях | Средняя | Средний | Запись dedup-индекса в Redis с TTL 24ч при создании задачи; при race condition — один поток выигрывает, остальные добавляют комментарии |
| Авито меняет формат webhook payload без предупреждения | Низкая | Средний | Логировать raw payload каждого webhook в `SystemEvent` на 30 дней; при парсинг-ошибке — алерт SUPERADMIN |
| Номер телефона с пропущенного звонка не передаётся Авито | Средняя | Низкий | Задача создаётся без телефона, с текстом «Номер недоступен». Линковка к User/Tenant — опциональный энричмент, не блокирует создание задачи |
